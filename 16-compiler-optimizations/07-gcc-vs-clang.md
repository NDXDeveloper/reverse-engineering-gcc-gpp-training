# 16.7 — GCC vs Clang comparison: assembly pattern differences

> **Associated source files**: `binaries/ch16-optimisations/opt_levels_demo.c`, `gcc_idioms.c`  
> **Compilation**: `make s16_7` (produces `_clang_O0` and `_clang_O2` variants, requires `clang`)

---

## Introduction

Up to this point, this entire chapter has focused on GCC. But in the real world, the binary you're analyzing may just as well have been compiled with **Clang/LLVM** — the other major open-source C/C++ compiler. On macOS, Clang is the default compiler. Many Linux projects use it too (the Linux kernel can be compiled with Clang since version 4.15, and Android uses exclusively Clang since 2019). Finally, commercial compilers like Intel's (ICX) are now LLVM-based.

For the reverse engineer, identifying a binary's original compiler is a valuable piece of tactical information. It allows choosing the right pattern recognition heuristics, anticipating applied optimizations, and adjusting analysis tool parameters (Ghidra, IDA). And the best way to identify the compiler is to recognize its "tics" — the characteristic patterns that differ between GCC and Clang.

GCC and Clang share the same goals (translating C/C++ into performant machine code) and often produce very similar assembly. But their internal optimization paths differ, creating subtle but recognizable divergences. This section catalogs the most important ones.

> 💡 To reproduce comparisons, Compiler Explorer ([godbolt.org](https://godbolt.org)) is the ideal tool: enter the same C code, select GCC and Clang side by side, and observe the differences.

---

## How to identify a binary's compiler

Before detailing pattern differences, here are the "high-level" clues that often allow a quick determination.

### Clue 1 — Version strings in the binary

If the binary isn't stripped, the `.comment` section often contains the compiler version:

```bash
$ readelf -p .comment build/opt_levels_demo_O2
  [     0]  GCC: (Ubuntu 13.2.0-23ubuntu4) 13.2.0

$ readelf -p .comment build/opt_levels_demo_clang_O2
  [     0]  clang version 18.1.3 (...)
```

This is the most direct method, but it only works on non-stripped binaries. A `strip` or `-s` removes `.comment`.

### Clue 2 — Initialization functions

GCC and Clang emit slightly different initialization functions in the `.init` and `.fini` sections. GCC uses `__libc_csu_init` and `__libc_csu_fini` (on older systems with glibc < 2.34), while Clang emits different patterns for global constructor initialization.

### Clue 3 — Internal naming conventions

Internal symbols generated by the compiler have different names:

- GCC: `.LC0`, `.LC1` for literals in `.rodata`, `.L1`, `.L2` for labels.  
- Clang: `.Lcmp_true`, `.Lpcrel_hi0`, or LLVM-generated names like `__unnamed_1`.

In a stripped binary, these internal labels are removed. But in a binary with debug symbols, they're visible.

### Clue 4 — Assembly patterns

This is the core of this section. Some patterns are quasi-exclusive to one compiler or the other. We detail them below.

---

## Difference 1 — Function prologues

A function's prologue and epilogue are the first thing the disassembler shows. GCC and Clang have different preferences.

### GCC in `-O0`

GCC systematically uses the classic `rbp` frame pointer:

```asm
function:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 0x20                     ; space for local variables
    ; ...
    leave                                ; equivalent to mov rsp, rbp ; pop rbp
    ret
```

The `leave` in the epilogue is characteristic of GCC. It's rarely emitted by Clang.

### Clang in `-O0`

Clang also uses `rbp` in `-O0`, but its prologue is sometimes slightly different:

```asm
function:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 0x20
    ; ...
    add    rsp, 0x20                     ; explicit rsp restoration
    pop    rbp                           ; no leave
    ret
```

Clang prefers `add rsp, N` + `pop rbp` instead of `leave`. Both are functionally equivalent, but `leave` is a quasi-exclusive GCC marker.

### In `-O2` — frame pointer omission

Both compilers omit the frame pointer in `-O2` by default (`-fomit-frame-pointer`), but their conventions differ for leaf functions (that don't call any other function):

```asm
; GCC — leaf function in O2: no prologue at all
leaf_function:
    imul   eax, edi, 42
    ret

; Clang — same thing, identical
leaf_function:
    imul   eax, edi, 42
    ret
```

For leaf functions, both compilers produce identical code. Differences appear in more complex functions.

## Difference 2 — Choosing between `cmov` and branching

Both compilers use `cmov` to eliminate branches, but their decision heuristics differ.

### Simple ternary pattern

```c
int max(int a, int b) {
    return (a > b) ? a : b;
}
```

GCC and Clang produce the same code — a `cmp` + `cmov`. No difference here.

### Pattern with computation in branches

```c
int f(int x) {
    if (x > 0)
        return x * 3 + 1;
    else
        return x * 2 - 5;
}
```

#### GCC in `-O2`

GCC tends to **keep the branch** when both branches contain computation:

```asm
    test   edi, edi
    jle    .L_negative
    lea    eax, [rdi+rdi*2]             ; x * 3
    add    eax, 1                        ; + 1
    ret
.L_negative:
    lea    eax, [rdi+rdi]               ; x * 2
    sub    eax, 5                        ; - 5
    ret
```

Two separate paths, each ending with `ret`.

#### Clang in `-O2`

Clang is often **more aggressive** in using `cmov`. It computes both branches then selects the result:

```asm
    lea    eax, [rdi+rdi*2]             ; eax = x * 3
    add    eax, 1                        ; eax = x * 3 + 1  ("then" branch)
    lea    ecx, [rdi+rdi]               ; ecx = x * 2
    sub    ecx, 5                        ; ecx = x * 2 - 5  ("else" branch)
    test   edi, edi
    cmovle eax, ecx                     ; if x <= 0: eax = ecx
    ret
```

Clang computes **both** results, then chooses with a `cmov`. It's more predictable for the pipeline (no branch), but wastes work if only one branch is taken.

### What RE should remember

If you see both branches of an if/else computed then selected by `cmov` — even when the branches contain multiple instructions — it's a Clang indicator. GCC is more conservative and prefers a branch when branches are non-trivial.

## Difference 3 — Division by constant: same result, different path

Both GCC and Clang use the magic number for divisions by constant, but the chosen constants and adjustment sequences can differ.

### Example: `x / 7` (signed)

#### GCC

```asm
    mov    eax, edi
    mov    edx, 0x92492493
    imul   edx
    add    edx, edi                      ; additive correction
    sar    edx, 2
    mov    eax, edx
    shr    eax, 31
    add    eax, edx
```

#### Clang

```asm
    mov    eax, edi
    mov    ecx, 0x92492493
    imul   ecx                           ; same magic number
    add    edx, edi                      ; same correction
    sar    edx, 2
    shr    edi, 31                       ; sign extraction on original register
    add    eax, edx, edi                 ; or lea eax, [rdx+rdi]
```

For `/7`, both compilers choose the same magic number. But the registers used and the order of correction operations differ. For other divisors, the magic numbers can be **mathematically different** (there are often multiple correct M/S pairs for the same divisor), making comparison more delicate.

### What RE should remember

The magic number itself is not a reliable compiler indicator — both may choose the same one. It's the surrounding instructions (register choices, correction order, use of `lea` vs `add`) that carry the signature.

## Difference 4 — Switch handling

### Jump table: data position

Both compilers generate jump tables for dense switches, but the way they compute the target address differs.

#### GCC — offsets relative to the table

```asm
    lea    rax, [rip+.L_table]           ; base = table address
    movsxd rdx, DWORD PTR [rax+rdi*4]   ; offset = table[index]
    add    rax, rdx                      ; target = base + offset
    jmp    rax
```

GCC stores **offsets relative to the table itself** (signed 32-bit offsets). The final address is `table_address + offset`.

#### Clang — offsets relative to the `lea` instruction

```asm
    lea    rcx, [rip+.LJTI0_0]          ; base = table address
    movsxd rax, DWORD PTR [rcx+4*rdi]   ; offset = table[index]
    add    rax, rcx                      ; target = base + offset
    jmp    rax
```

The pattern is very similar, but Clang often uses a different naming convention for the table (`.LJTI0_0` instead of `.L_table`) and may choose different registers.

### Sparse switch: search strategy

For sparse switches, both compilers use comparison trees, but Clang tends to produce **more balanced** trees (true binary search), while GCC may produce linear cascades when the number of cases is small (< 6).

## Difference 5 — Loop unrolling and vectorization

This is one of the areas where differences between GCC and Clang are most pronounced.

### Unrolling factor

For the same loop in `-O2`, GCC and Clang often choose different unrolling factors:

```c
void add_arrays(int *dst, const int *src, int n) {
    for (int i = 0; i < n; i++)
        dst[i] += src[i];
}
```

- **GCC `-O2`**: unrolling by 2 or 4 (conservative).  
- **Clang `-O2`**: unrolling by 4 or 8 (more aggressive by default).

In `-O3`, both become aggressive, but Clang tends to unroll more.

### Vectorized loop style

#### GCC — classic prologue/body/epilogue

GCC follows the three-part structure described in Section 16.3:

```asm
    ; Bounds test
    cmp    ecx, 3
    jle    .L_scalar
    ; Vectorized loop (SSE)
.L_vec:
    movdqu xmm0, [rsi+rax]
    paddd  xmm0, [rdi+rax]
    movdqu [rdi+rax], xmm0
    add    rax, 16
    cmp    eax, edx
    jl     .L_vec
    ; Scalar epilogue
.L_scalar:
    ; ...
```

#### Clang — vectorized loop with pointer accumulation

Clang often uses a different style, with incremented pointers instead of an index:

```asm
    ; Clang vectorizes with pointers
    mov    rax, rdi                      ; ptr_dst = dst
    mov    rcx, rsi                      ; ptr_src = src
    mov    edx, esi                      ; counter

.L_vec:
    movdqu xmm0, [rcx]                  ; load from ptr_src
    movdqu xmm1, [rax]                  ; load from ptr_dst
    paddd  xmm0, xmm1
    movdqu [rax], xmm0                  ; store to ptr_dst
    add    rax, 16                       ; ptr_dst += 16
    add    rcx, 16                       ; ptr_src += 16
    add    edx, -4                       ; counter -= 4
    jne    .L_vec
```

The difference is subtle but characteristic: GCC uses an **index** (`[rsi+rax]`) while Clang increments **pointers** directly (`add rax, 16`). The functional result is identical, but the addressing pattern differs.

### Reduction vectorization

For an accumulation loop (`sum += a[i]`), Clang often uses **multiple vector accumulators** (2 or 4 `xmm` registers in parallel) to mask latency, while GCC uses a single accumulator:

```asm
; Clang — reduction with 4 accumulators
    pxor   xmm0, xmm0                   ; acc0
    pxor   xmm1, xmm1                   ; acc1
    pxor   xmm2, xmm2                   ; acc2
    pxor   xmm3, xmm3                   ; acc3
.L_vec:
    paddd  xmm0, [rdi]
    paddd  xmm1, [rdi+16]
    paddd  xmm2, [rdi+32]
    paddd  xmm3, [rdi+48]
    add    rdi, 64
    ; ...
    ; Final reduction: xmm0 += xmm1 += xmm2 += xmm3
    paddd  xmm0, xmm1
    paddd  xmm2, xmm3
    paddd  xmm0, xmm2
    ; Then horizontal reduction
```

If you see 2 or 4 `pxor xmm, xmm` initializing accumulators at the start of a loop, followed by `paddd` to each of them, it's a Clang indicator.

## Difference 6 — Loop alignment and `nop` padding

Both compilers align branch targets (loop start, jump labels) for instruction cache performance reasons, but their padding strategy differs.

### GCC — classic `nop` or `nop DWORD PTR [rax]`

GCC uses `nop` instructions of various sizes to align to 16 bytes:

```asm
    nop                                  ; 1 byte
    nop    DWORD PTR [rax+rax*1+0x0]     ; 8 bytes (long nop)
    nop    WORD PTR cs:[rax+rax*1+0x0]   ; 10 bytes
    ; Alignment reached
.L_loop:
    ; ...
```

These "long nops" are instructions that do nothing but occupy a variable number of bytes. They appear as multi-byte `nop`s in `objdump`.

### Clang — `nop` + `.p2align` alignment

Clang uses a `.p2align` directive that produces similar nops, but tends to align more aggressively (often to 32 bytes instead of 16) and sometimes uses different nop patterns:

```asm
    .p2align 5, 0x90                     ; align to 32 bytes with 0x90 (nop)
.LBB0_1:
    ; loop
```

### What RE should remember

A sequence of multi-byte nops before a loop label is alignment padding — ignore it during analysis. The alignment size (16 vs 32 bytes) can be a compiler clue.

## Difference 7 — Strings and constants in `.rodata`

### `.rodata` organization

GCC and Clang organize literal constants differently in `.rodata`:

- **GCC**: format strings are grouped at the beginning of `.rodata`, in order of first use. Floating-point constants are separate.  
- **Clang**: strings and constants are sometimes interleaved, and Clang can merge strings that share a common suffix (*string merging* more aggressive).

### String fusion

```c
printf("Hello, world!\n");  
printf("world!\n");  
```

GCC stores both strings separately in `.rodata`. Clang can detect that `"world!\n"` is a suffix of `"Hello, world!\n"` and store only the long string, with an offset pointer for the short one:

```
; Clang .rodata
.LC0:  "Hello, world!\n\0"
; "world!\n" points to .LC0 + 7 (no duplication)
```

This is a size optimization that doesn't exist in GCC (except with `-fmerge-constants`, which only handles exact suffixes).

## Difference 8 — Local variable initialization

```c
int data[8] = {0};
```

### GCC

GCC often uses a sequence of `mov QWORD PTR [rsp+offset], 0` or a `rep stosq`:

```asm
    ; GCC — zeroing by qword pairs
    mov    QWORD PTR [rsp],    0
    mov    QWORD PTR [rsp+8],  0
    mov    QWORD PTR [rsp+16], 0
    mov    QWORD PTR [rsp+24], 0
```

### Clang

Clang often prefers SIMD instructions for zeroing, even for non-SIMD data:

```asm
    ; Clang — zeroing via xmm
    xorps  xmm0, xmm0                   ; xmm0 = {0, 0, 0, 0}
    movaps XMMWORD PTR [rsp],    xmm0   ; 16 bytes to zero
    movaps XMMWORD PTR [rsp+16], xmm0   ; 16 bytes to zero
```

`xorps xmm0, xmm0` is the canonical way to zero an SSE register — it's the SIMD equivalent of `xor eax, eax`. The `movaps` (Move Aligned Packed Single) stores 16 bytes at once.

### What RE should remember

The use of `xorps xmm0, xmm0` + `movaps` to initialize scalar data (not floats) is a Clang marker. GCC reserves SIMD registers for SIMD data and prefers `mov QWORD PTR, 0` for scalar initializations.

## Difference 9 — Handling `abs()` and `min`/`max`

### `abs(x)`

#### GCC (recent)

```asm
    mov    eax, edi
    neg    eax
    cmovs  eax, edi                     ; neg + cmovs
```

#### Clang

```asm
    mov    eax, edi
    sar    eax, 31                       ; mask = sign
    xor    edi, eax                      ; flip if negative
    sub    edi, eax                      ; +1 if negative (two's complement)
    mov    eax, edi
```

Clang systematically prefers the arithmetic pattern `sar` + `xor` + `sub`, while recent GCC uses `neg` + `cmovs`. Both are correct; the choice is a compiler marker.

### `min(a, b)` / `max(a, b)`

Both compilers use `cmp` + `cmov`, but the operand order may differ:

```asm
; GCC — min(a, b)
    cmp    edi, esi
    mov    eax, esi                      ; eax = b (default)
    cmovle eax, edi                     ; if a <= b: eax = a

; Clang — min(a, b)
    cmp    edi, esi
    cmovle esi, edi                     ; if a <= b: esi = a (overwrites b)
    mov    eax, esi                      ; return result
```

GCC initializes the result with a default value then conditionally overwrites. Clang does the `cmov` directly on an operand then copies to `eax`. Functionally identical, stylistically different.

## Difference 10 — Function calls and tail calls

### Tail calls

Both compilers perform tail call optimization, but Clang is **more aggressive**: it applies it in situations where GCC keeps a `call` + `ret` out of caution. In particular, Clang more frequently performs tail calls when there are complex parameter rearrangements.

### libc calls

For a simple `printf("Hello\n")`, GCC sometimes replaces the call with `puts("Hello")` (removing the `\n` and changing the function), while Clang often keeps `printf`. This `printf` → `puts` replacement is a GCC marker.

```asm
; GCC
    lea    rdi, [rip+.LC0]              ; "Hello" (without \n)
    call   puts@plt                     ; puts instead of printf!

; Clang
    lea    rdi, [rip+.LC0]              ; "Hello\n"
    xor    eax, eax                     ; 0 float args (variadic)
    call   printf@plt                   ; printf preserved
```

The `xor eax, eax` before a `call printf` is a System V AMD64 variadic calling convention marker: `al` must contain the number of SSE registers used for float arguments. GCC sometimes omits it when it knows there are no float arguments; Clang includes it systematically.

## Difference 11 — Bit rotation

```c
unsigned int rotl(unsigned int x, int n) {
    return (x << n) | (x >> (32 - n));
}
```

Both compilers recognize the pattern and emit `rol`. No significant difference here — it's one of the rare completely identical patterns.

However, for constant rotations in a larger context (hash, crypto), Clang tends to reschedule rotations more aggressively, which can change the order of `rol`/`ror` relative to the source.

---

## Difference 12 — Startup functions (CRT)

The startup code (*C Runtime*, CRT) visible before `main()` differs depending on the compiler and libc version.

### GCC + glibc

The entry point `_start` calls `__libc_start_main` with a well-known signature. The functions `__libc_csu_init` and `__libc_csu_fini` (on glibc < 2.34) are a strong GCC marker.

### Clang + glibc

Clang uses the same CRT as GCC (provided by glibc), but may emit different global constructors/destructors. The presence of `__libc_csu_init` is therefore not a reliable GCC marker if the system CRT is shared.

### Clang + musl or another libc

If the binary uses musl (common in Alpine Linux containers), the CRT is different from glibc. The absence of `__libc_csu_init` and the presence of `__init_libc` or `__libc_start_main_stage2` are musl indicators, regardless of compiler.

## Summary table of markers

| Pattern | GCC | Clang |  
|---|---|---|  
| Function epilogue | `leave` | `add rsp, N` + `pop rbp` |  
| `printf("Hello\n")` → | `call puts` (optimized) | `call printf` (preserved) |  
| `xor eax, eax` before variadic call | Sometimes omitted | Systematic |  
| `abs(x)` | `neg` + `cmovs` | `sar 31` + `xor` + `sub` |  
| if/else with computation in branches | Classic branching | Compute both + `cmov` |  
| Local array zeroing | `mov QWORD [rsp+off], 0` | `xorps xmm0` + `movaps` |  
| Addressing in vectorized loop | Index `[base+rax*4]` | Incremented pointers |  
| SIMD reduction accumulators | 1 `xmm` register | 2–4 `xmm` registers |  
| Unrolling in `-O2` | Conservative (×2–4) | Aggressive (×4–8) |  
| Loop alignment | 16 bytes | 16 or 32 bytes |  
| String suffix merging | Rare | Common |

## Ambiguous cases: when distinction is impossible

It's important to be honest about the limits: for many patterns, GCC and Clang produce **identical** or near-identical assembly. This is the case for:

- Division-by-constant magic numbers (often the same M/S pairs).  
- Dense switch jump tables (same structure).  
- `cmov` for simple ternaries.  
- Basic SIMD instructions (`movdqu`, `paddd`, etc.).  
- Standard function calls (System V convention respected by both).  
- `test` + `setcc` + `movzx` patterns for booleans.

In these cases, compiler identification must rely on "high-level" clues (`.comment` section, CRT, etc.) rather than instruction patterns.

Moreover, successive versions of each compiler progressively modify their patterns. A GCC 8 emits a different `abs()` than GCC 13. A Clang 14 vectorizes differently than Clang 18. The markers described here are general tendencies, not absolute rules.

## Practical RE strategy

Here's a four-step approach when analyzing a binary of unknown origin:

**1. Check `.comment` first.** This is the quickest test. If the section exists, it gives the compiler and its version.

```bash
readelf -p .comment target_binary 2>/dev/null  
strings target_binary | grep -iE 'gcc|clang|llvm'  
```

**2. Look at the CRT.** Initialization functions and the `_start` structure are strong indicators.

**3. Observe characteristic patterns.** Look for the markers in the summary table: `leave` vs `add rsp` + `pop rbp`, `puts` vs `printf`, scalar vs SIMD zeroing, addressing style in loops.

**4. Don't over-invest in identification.** Knowing whether it's GCC or Clang is useful for calibrating your expectations, but it's not a prerequisite for analysis. The fundamental idioms (magic numbers, cmov, jump tables, vectorization) are the same — only implementation details vary. Focus your energy on understanding the program's logic rather than exact compiler identification.

## Tools to go further

**Compiler Explorer** ([godbolt.org](https://godbolt.org)) — Compare the same C code with GCC and Clang side by side. It's the best way to build your intuition about pattern differences.

**`-fverbose-asm`** — Both compilers support this flag, which adds comments to the generated assembly. Useful for understanding which C line corresponds to which instructions.

```bash
gcc -O2 -S -fverbose-asm -o output_gcc.s source.c  
clang -O2 -S -fverbose-asm -o output_clang.s source.c  
diff output_gcc.s output_clang.s  
```

**`-mllvm -print-after-all`** (Clang only) — Displays the LLVM IR after each optimization pass. It's the equivalent of `-fdump-tree-all` for GCC. Useful for understanding why Clang chose a particular pattern.
