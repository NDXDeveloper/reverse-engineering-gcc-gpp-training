## Chapter goals

In Chapter 5, we got acquainted with `objdump` as part of the quick triage of a binary: displaying headers, listing sections, taking a look at symbols. That was a first contact — we scratched the surface. In this chapter, we move to the next step: **using `objdump` as a real disassembly tool**, capable of turning the bytes of the `.text` section into readable assembly instructions.

By the end of this chapter, you will be able to:

- Disassemble a complete ELF binary with `objdump`, including when symbols have been removed with `strip`.  
- Switch between AT&T syntax (the GNU default) and Intel syntax, and understand why the latter is often preferred in RE.  
- Visually compare the assembly code produced by GCC at different optimization levels (`-O0`, `-O2`, `-O3`) and interpret the transformations applied by the compiler.  
- Read and annotate a function prologue and epilogue directly in `objdump`'s output.  
- Find the `main()` function in a stripped binary and identify C++ functions despite *name mangling*.  
- Use `c++filt` to make C++ symbols readable.  
- Recognize the limits of `objdump` as a linear disassembler, and understand when tools like Ghidra, IDA, or Radare2 become indispensable on complex binaries.

## Why `objdump` before Ghidra?

The question is worth asking. If Ghidra offers a decompiler, a flow graph, cross-references, and a rich graphical interface (all of which will be covered in Chapter 8), why dwell on a command-line tool that produces a raw listing?

Several reasons:

**`objdump` is universal and lightweight.** It is part of the `binutils` package, installed by default on practically every Linux distribution. No JVM to launch, no project to create, no automatic analysis to wait for. A simple `objdump -d ./binary` and you have your listing in a second. On a remote server accessible only via SSH, on an embedded machine, or in a minimal container, `objdump` is often the only tool available.

**It forces direct reading of assembly.** When starting out in RE, it is tempting to immediately lean on the C pseudo-code generated by a decompiler. It's comfortable, but it's also misleading: the decompiler makes hypotheses, simplifies constructions, and sometimes gets things wrong. Working first with `objdump` forces you to read instructions one by one, to trace jumps manually, to understand what the processor actually does. It's a fundamental exercise that then makes use of Ghidra much more effective, because you will know how to spot the decompiler's mistakes.

**It serves as a reference for comparison.** When Ghidra or IDA produce a surprising result, returning to `objdump`'s raw listing lets you verify what the binary actually contains, without an interpretation layer. It's your "ground truth".

In short, `objdump` is not a tool of the past that you skim before moving to serious things. It is a tool you will still use regularly, even after mastering Ghidra.

## Reminder: the Binutils family

`objdump` does not work alone. It is part of the **GNU Binutils** suite, a set of command-line tools designed to manipulate object files and binaries. You have already met several of them in Chapter 5. Here are the ones that appear in this chapter:

| Tool | Main role in this chapter |  
|---|---|  
| `objdump` | Disassembly, section and header display |  
| `readelf` | Fine-grained ELF structure inspection (complementary to `objdump`) |  
| `nm` | Symbol listing — locating functions before disassembling |  
| `c++filt` | Demangling C++ symbols (Itanium ABI) |  
| `strip` | Symbol removal — to create our own stripped test binaries |  
| `objcopy` | Section manipulation (extraction, addition) — mentioned in passing |

These tools share the same underlying library (`libbfd` — *Binary File Descriptor*) and natively understand the ELF format. They also work on other formats (PE, Mach-O to some extent), but they are most at home on ELF — which is convenient, since that is our target throughout this training.

## Linear vs recursive disassembly

Before diving into the commands, it is essential to understand **how** `objdump` disassembles, because it conditions what it can and cannot do.

`objdump` uses **linear disassembly** (*linear sweep*). The principle is simple: it starts at the beginning of the `.text` section (or at the address you specify) and decodes bytes sequentially, instruction after instruction, until the end of the section. It does not follow jumps, does not reconstruct the control flow graph, and does not distinguish code from data that might be interleaved in `.text`.

Conversely, disassemblers like Ghidra, IDA, and Radare2 use **recursive disassembly** (*recursive descent*). They start at the entry point, follow branches (`call`, `jmp`, `jz`…), and only disassemble paths that are actually reachable. This lets them correctly handle data inserted in the middle of code, `switch` *jump tables*, and functions indirectly called via pointers.

This difference has concrete consequences:

- **Data interpreted as code**: if the compiler places a constants table or a *jump table* in `.text`, `objdump` will attempt to decode them as instructions. You will then see absurd instruction sequences — a classic warning signal.  
- **Desynchronization**: x86-64 instructions have variable length (from 1 to 15 bytes). If `objdump` "misses" the start of an instruction (for example after going through data), it can desynchronize and produce fantasy instructions over several lines before getting back on track.  
- **No flow graph**: `objdump` will never show you a function or basic-block graph. It is linear text, period.

These limits do not make `objdump` useless — far from it. On a binary cleanly compiled with GCC without obfuscation, linear disassembly works remarkably well, because GCC clearly separates code and data into distinct sections. Problems appear mainly with obfuscated binaries, packed binaries, or binaries compiled with exotic options. We'll come back to this in detail in section 7.7.

## Chapter organization

The chapter follows a natural progression, from the simplest case to the most complex:

- **7.1** — We start by disassembling a binary **without symbols**, the most common case in real-world RE, and we learn to find our way around despite the absence of function names.  
- **7.2** — We tackle the question of **AT&T vs Intel syntax**, with a practical switch from one to the other.  
- **7.3** — We compare the disassembly of the same program compiled **at different GCC optimization levels** (`-O0`, `-O2`, `-O3`), to learn to recognize compiler transformations.  
- **7.4** — We focus on **reading function prologues and epilogues** in an `objdump` listing, a fundamental reflex for delimiting functions.  
- **7.5** — We learn to **locate `main()`** even without symbols, and to spot C++ functions despite *name mangling*.  
- **7.6** — We discover `c++filt`, the **demangling** tool of the Binutils suite.  
- **7.7** — We end with an honest overview of **`objdump`'s limitations**, to understand when it's time to switch to a recursive disassembler.
