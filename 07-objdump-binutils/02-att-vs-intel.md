# 7.2 — AT&T vs Intel syntax — switching from one to the other (`-M intel`)

> 🔧 **Tools used**: `objdump`, `gcc`, `gdb`  
> 📦 **Binaries**: `keygenme_O0` (`binaries/ch07-keygenme/` directory)

---

## Two syntaxes for the same machine code

When you disassemble an x86/x86-64 binary, the machine bytes are always the same — it is the processor that executes them, and it knows neither AT&T nor Intel. On the other hand, the **textual representation** of those instructions varies according to the convention chosen by the disassembly tool. Two conventions coexist in the ecosystem, inherited from different historical traditions, and they are encountered constantly in practice.

The **AT&T** syntax was born in the Bell Labs, cradle of Unix. It was subsequently adopted by the GNU suite: GCC, GAS (GNU Assembler), GDB (by default), and `objdump` (by default). If you work on Linux with GNU tools without any particular configuration, that is what you will see.

The **Intel** syntax is that of the official Intel documentation (*Intel Software Developer's Manual*). It was adopted by NASM, then by the majority of reverse-engineering tools: IDA, Ghidra (by default), Binary Ninja, Radare2/Cutter, and most tutorials, books, articles, and CTF write-ups.

These are not two different languages. The same instructions, the same registers, the same operations are represented — only the notation changes. It's comparable to the difference between the notations `3 + 4` and `(+ 3 4)`: the result is identical, but readability differs depending on what you are used to.

Mastering both syntaxes is not a luxury: it is a necessity. You will read AT&T syntax in GCC outputs, Linux-kernel *inline assembly*, GDB's default output, and `.s` files generated by the compiler. You will read Intel syntax in Ghidra, IDA, the Intel/AMD manuals, and nearly all RE literature. Switching from one to the other must become an automatic reflex.

## Differences, one by one

Let's take the same sequence of machine code and examine each difference between both representations. All instructions below correspond to exactly the same bytes — only the writing changes.

### 1. Operand order

This is the most important difference and the most frequent source of confusion.

```
AT&T:     mov    %rsp, %rbp          # source, destination  
Intel:    mov    rbp, rsp             # destination, source  
```

In AT&T, you read "move `rsp` to `rbp`" left to right. In Intel, you read "`rbp` receives the value of `rsp`", like an assignment in a programming language (`rbp = rsp`). The Intel order is often judged more intuitive by developers familiar with C, and that's one of the reasons for its prevalence in RE.

This reversal applies to **every** two-operand instruction: `mov`, `add`, `sub`, `cmp`, `xor`, `lea`, `test`, etc.

```
AT&T:     add    $0x10, %rax          # rax = rax + 0x10  
Intel:    add    rax, 0x10            # rax = rax + 0x10  
```

The result is the same, but the reading order is reversed.

> ⚠️ **Classic pitfall**: when you switch from an AT&T listing to an Intel listing (or vice versa), the operand that was on the left ends up on the right. If you mentally annotate "the first operand is the source", make sure you know which syntax you are in. An error on the operand order of `cmp` or `sub` can completely invert your understanding of the conditional logic.

### 2. Register and immediate-value prefixes

In AT&T, registers are prefixed with `%` and immediate values (constants) with `$`. In Intel, there is no prefix.

```
AT&T:     mov    $0x42, %eax          # eax = 0x42  
Intel:    mov    eax, 0x42            # eax = 0x42  
```

```
AT&T:     push   %rbp  
Intel:    push   rbp  
```

```
AT&T:     cmp    $0x0, %edi  
Intel:    cmp    edi, 0x0  
```

Intel syntax is visually cleaner. AT&T syntax has the advantage of being **unambiguous**: seeing `$0x42`, you immediately know it's a constant, not a memory address. In Intel, the distinction relies on context.

### 3. Size suffixes vs size keywords

In AT&T, the operation size is encoded in a **suffix** added to the instruction mnemonic:

| Suffix | Size | AT&T example |  
|---|---|---|  
| `b` | 1 byte | `movb $0x41, %al` |  
| `w` | 2 bytes (word) | `movw $0x0, %ax` |  
| `l` | 4 bytes (long/dword) | `movl $0x0, %eax` |  
| `q` | 8 bytes (quad) | `movq %rsp, %rbp` |

In Intel, the size is indicated by a **keyword** placed before the memory operand when necessary:

| Keyword | Size | Intel example |  
|---|---|---|  
| `BYTE PTR` | 1 byte | `mov BYTE PTR [rax], 0x41` |  
| `WORD PTR` | 2 bytes | `mov WORD PTR [rax], 0x0` |  
| `DWORD PTR` | 4 bytes | `mov DWORD PTR [rax], 0x0` |  
| `QWORD PTR` | 8 bytes | `mov QWORD PTR [rbp-0x8], rdi` |

When the size is obvious from the registers used (for example `mov rax, rbx` — both are 64-bit registers), Intel omits the keyword. AT&T still often adds the suffix, though GCC may omit it when there is no ambiguity.

```
AT&T:     movl   $0x0, -0x4(%rbp)     # 'l' = 4 bytes  
Intel:    mov    DWORD PTR [rbp-0x4], 0x0  
```

### 4. Memory access: parentheses vs brackets

This is the most visible difference on instructions that access memory.

**AT&T** uses **parentheses** and a syntax of the form `displacement(base, index, scale)`:

```
AT&T:     movl   -0x4(%rbp), %eax           # [rbp - 4]  
AT&T:     movl   (%rax,%rcx,4), %edx        # [rax + rcx*4]  
AT&T:     movl   0x10(%rax,%rcx,4), %edx    # [rax + rcx*4 + 0x10]  
```

**Intel** uses **brackets** and a natural algebraic syntax:

```
Intel:    mov    eax, DWORD PTR [rbp-0x4]  
Intel:    mov    edx, DWORD PTR [rax+rcx*4]  
Intel:    mov    edx, DWORD PTR [rax+rcx*4+0x10]  
```

The general formula for x86-64 memory addressing is:

```
Effective address = base + (index × scale) + displacement
```

Both syntaxes express this formula, but in radically different ways. The Intel version is a readable mathematical expression; the AT&T version is a compact positional format.

Here is a summary table with several addressing forms:

| Addressing mode | AT&T | Intel |  
|---|---|---|  
| Direct (register) | `%rax` | `rax` |  
| Immediate | `$42` | `42` |  
| Simple indirect | `(%rax)` | `[rax]` |  
| Base + displacement | `-0x8(%rbp)` | `[rbp-0x8]` |  
| Base + index | `(%rax,%rcx)` | `[rax+rcx]` |  
| Base + index × scale | `(%rax,%rcx,4)` | `[rax+rcx*4]` |  
| Full | `0x10(%rax,%rcx,8)` | `[rax+rcx*8+0x10]` |  
| RIP-relative | `0x2f3a(%rip)` | `[rip+0x2f3a]` |

### 5. Jump and call instructions (long)

For indirect absolute jumps and indirect calls, AT&T uses a `*` prefix:

```
AT&T:     jmpq   *%rax                # indirect jump via register  
Intel:    jmp    rax  

AT&T:     callq  *0x2fe2(%rip)        # indirect call via memory  
Intel:    call   QWORD PTR [rip+0x2fe2]  
```

Also, AT&T sometimes suffixes jump and call instructions with `q` (for *quad*, 64 bits) or `l` — you will often see `callq` and `jmpq` instead of `call` and `jmp`. In Intel, it's always simply `call` and `jmp`.

### 6. Slightly different mnemonics

A few instructions have a different mnemonic between the two syntaxes:

| AT&T | Intel | Operation |  
|---|---|---|  
| `movzbl` | `movzx` | Zero extension (byte → long) |  
| `movsbl` | `movsx` | Sign extension (byte → long) |  
| `movzbq` | `movzx` | Zero extension (byte → quad) |  
| `movslq` | `movsxd` | Sign extension (long → quad) |  
| `cbtw` | `cbw` | Conversion byte → word |  
| `cwtl` | `cwde` | Conversion word → dword |  
| `cltq` | `cdqe` | Conversion dword → qword |  
| `cltd` | `cdq` | Sign extension eax → edx:eax |  
| `cqto` | `cqo` | Sign extension rax → rdx:rax |

In AT&T, the `movzx`/`movsx` mnemonics encode source and destination sizes in the name (`movzbl` = *move zero-extend byte to long*). In Intel, the instruction is simply `movzx` or `movsx`, and the size is determined by the operands.

## Complete comparative example

Let's take a realistic excerpt from the start of a function, such as `objdump` would produce on `keygenme_O0`:

### AT&T version (default)

```
0000000000001139 <compute_hash>:
    1139:       55                      push   %rbp
    113a:       48 89 e5                mov    %rsp,%rbp
    113d:       48 89 7d e8             mov    %rdi,-0x18(%rbp)
    1141:       c7 45 fc 00 00 00 00    movl   $0x0,-0x4(%rbp)
    1148:       c7 45 f8 00 00 00 00    movl   $0x0,-0x8(%rbp)
    114f:       eb 1d                   jmp    116e <compute_hash+0x35>
    1151:       8b 45 f8                mov    -0x8(%rbp),%eax
    1154:       48 98                   cltq
    1156:       48 03 45 e8             add    -0x18(%rbp),%rax
    115a:       0f b6 00                movzbl (%rax),%eax
    115d:       0f be c0                movsbl %al,%eax
    1160:       01 45 fc                add    %eax,-0x4(%rbp)
    1163:       8b 45 fc                mov    -0x4(%rbp),%eax
    1166:       c1 e0 03                shl    $0x3,%eax
    1169:       89 45 fc                mov    %eax,-0x4(%rbp)
    116c:       83 45 f8 01             addl   $0x1,-0x8(%rbp)
```

### Intel version (`-M intel`)

```
0000000000001139 <compute_hash>:
    1139:       55                      push   rbp
    113a:       48 89 e5                mov    rbp,rsp
    113d:       48 89 7d e8             mov    QWORD PTR [rbp-0x18],rdi
    1141:       c7 45 fc 00 00 00 00    mov    DWORD PTR [rbp-0x4],0x0
    1148:       c7 45 f8 00 00 00 00    mov    DWORD PTR [rbp-0x8],0x0
    114f:       eb 1d                   jmp    116e <compute_hash+0x35>
    1151:       8b 45 f8                mov    eax,DWORD PTR [rbp-0x8]
    1154:       48 98                   cdqe
    1156:       48 03 45 e8             add    rax,QWORD PTR [rbp-0x18]
    115a:       0f b6 00                movzx  eax,BYTE PTR [rax]
    115d:       0f be c0                movsx  eax,al
    1160:       01 45 fc                add    DWORD PTR [rbp-0x4],eax
    1163:       8b 45 fc                mov    eax,DWORD PTR [rbp-0x4]
    1166:       c1 e0 03                shl    eax,0x3
    1169:       89 45 fc                mov    DWORD PTR [rbp-0x4],eax
    116c:       83 45 f8 01             add    DWORD PTR [rbp-0x8],0x1
```

Notice that the machine-bytes column (in the middle) is **strictly identical** in both cases. `55` is `push rbp`/`push %rbp` no matter what — it's the same byte decoded by the processor. Only the right column changes.

A few specific observations on this excerpt:

- `cltq` (AT&T) becomes `cdqe` (Intel) — sign extension from `eax` to `rax`.  
- `movzbl (%rax),%eax` (AT&T) becomes `movzx eax, BYTE PTR [rax]` (Intel) — the source-size encoding moves from suffix to keyword.  
- The memory operand `-0x4(%rbp)` becomes `[rbp-0x4]` — parentheses to brackets, same semantics.  
- The `$` and `%` disappear in Intel, making the listing more airy.

## Switching between syntaxes in common tools

### `objdump`

As seen in section 7.1, the `-M intel` option switches to Intel syntax:

```bash
objdump -d -M intel keygenme_O0
```

To return to AT&T (or check explicitly):

```bash
objdump -d -M att keygenme_O0
```

You can also combine sub-options of `-M`. For example, `-M intel,addr32` would force displaying addresses in 32 bits (rarely useful, mentioned for completeness).

### GCC: viewing the generated assembly

GCC can produce an assembly file instead of a binary, via the `-S` option:

```bash
gcc -S -O0 -o keygenme.s keygenme.c          # AT&T by default  
gcc -S -O0 -masm=intel -o keygenme.s keygenme.c  # Intel  
```

The resulting `.s` file contains the GNU Assembler (GAS) directives and your program's assembly code. Comparing both versions is an excellent exercise for anchoring the differences.

### GDB

By default, GDB uses AT&T syntax. To switch to Intel:

```
(gdb) set disassembly-flavor intel
```

To make this choice permanent, add this line to your `~/.gdbinit` file:

```
set disassembly-flavor intel
```

If you use GEF, pwndbg, or PEDA (Chapter 12), Intel syntax is often configured by default.

### Ghidra, IDA, Binary Ninja, Radare2

These tools use Intel syntax by default. Ghidra lets you switch to AT&T via *Edit → Tool Options → Listing Fields → Operands Field* (rarely desired). IDA is historically tied to Intel. Radare2 supports both via the command `e asm.syntax=att` or `e asm.syntax=intel`.

In practice, if you work with these tools, you will read Intel. If you then move to `objdump` or GDB without having configured the alias, you will land on AT&T. Hence the importance of mastering both.

## Which syntax to choose for this tutorial?

In the rest of this training, **we will use Intel syntax by default**, unless otherwise specified. The reasons:

- It's the dominant convention in the RE community, reference books (*Practical Reverse Engineering*, *Reversing: Secrets of Reverse Engineering*, *The IDA Pro Book*…), and CTF write-ups.  
- It's the default syntax of Ghidra and IDA, the two major tools we will use in Chapters 8 and 9.  
- The `destination, source` order matches C assignment (`a = b`), which makes mental translation to pseudo-code easier.  
- Memory accesses in brackets (`[rbp-0x8]`) read more naturally than in parentheses (`-0x8(%rbp)`).

That said, you **must** stay comfortable with AT&T. You will encounter it in the Linux kernel source code, in the raw outputs of GCC and GDB, in system-library `.S` files, and in some academic articles. Do not treat it as an exotic dialect to avoid — consider it a regional accent of the same language.

> 💡 **Practical tip**: configure your tools once and for all in Intel (alias `objdump`, `~/.gdbinit`), but regularly force yourself to read an AT&T listing without converting it. The day you debug a production crash on a server where only `objdump` is installed and where your alias does not exist, you will be happy you can read both.

## Quick conversion cheat sheet

For quick reference, here is a summary table of the transformations to apply mentally when switching from one syntax to the other:

| Element | AT&T → Intel | Intel → AT&T |  
|---|---|---|  
| Operands | Reverse order | Reverse order |  
| Registers | Remove `%` | Add `%` |  
| Immediates | Remove `$` | Add `$` |  
| Memory access | `(…)` → `[…]` | `[…]` → `(…)` |  
| Size on mnemonic | Remove `b`/`w`/`l`/`q` suffix | Add matching suffix |  
| Size on memory operand | Add `BYTE`/`WORD`/`DWORD`/`QWORD PTR` | Remove size keyword |  
| Indirect jump/call | Remove `*` | Add `*` |  
| `movzbl`/`movsbl` | → `movzx`/`movsx` | `movzx`/`movsx` → add source+dest suffixes |  
| `cltq`/`cltd`/`cwtl` | → `cdqe`/`cdq`/`cwde` | Conversely |

With a bit of practice, this conversion becomes instantaneous. You will no longer think in "AT&T" or "Intel" — you will think directly in terms of operations on registers and memory, whatever the notation shown on screen.

---

## Summary

AT&T and Intel are two different notations for the same instruction set. Their main differences are operand order (source/destination reversed), prefixes (`%`, `$` in AT&T, none in Intel), memory-access notation (parentheses vs brackets), and the encoding of operation size (mnemonic suffix vs `PTR` keyword). GNU tools (GCC, GAS, GDB, `objdump`) use AT&T by default but support Intel via an option. RE tools (Ghidra, IDA, Binary Ninja) use Intel by default. Being able to read both fluently is indispensable — Intel syntax will be our default choice for the rest of this training.
