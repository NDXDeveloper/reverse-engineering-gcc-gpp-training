# 7.7 — Limitations of `objdump`: why a real disassembler is necessary

> 🔧 **Tools mentioned**: `objdump`, Ghidra, IDA, Radare2, Binary Ninja  
> 📦 **Binaries**: `keygenme_O0`, `keygenme_O2`, `keygenme_strip` (`binaries/ch07-keygenme/` directory) and `oop` (`binaries/ch22-oop/` directory)  
> 📝 **Syntax**: Intel (via `-M intel`)

---

## An indispensable tool, but not sufficient

Throughout this chapter, we used `objdump` to disassemble, compare, annotate, and navigate ELF binaries. On the training programs we studied — a few functions, cleanly compiled with GCC, without obfuscation — `objdump` does a remarkable job. Combined with `nm`, `readelf`, `c++filt`, `grep`, and a bit of method, it makes it possible to understand the logic of a binary end-to-end.

But training binaries are designed to be pedagogical. The binaries you'll encounter in real conditions — commercial software, malware, firmware, advanced CTF challenges — pose problems that `objdump` is structurally unable to solve. This section makes an honest inventory of these limitations, so you know exactly when to switch to a more powerful tool, and especially **why**.

---

## Limitation 1: linear disassembly

This is the fundamental limitation, the one from which most of the others derive. We introduced it in the chapter README, but its consequences deserve to be detailed here, now that you have practical experience with `objdump`.

### The problem of data in `.text`

`objdump` sequentially decodes every byte of `.text` as instructions. If the compiler or linker inserted **data** in the middle of the code — jump tables, literal constants, non-NOP alignment padding — `objdump` interprets them as instructions, producing absurd sequences.

The classic example is the *jump table* generated by GCC for a `switch` with many `case`s. The compiler places an array of relative offsets directly in `.text`, right after the `switch` code:

```asm
; Switch code (simplified)
    cmp    eax, 7
    ja     default_case
    lea    rdx, [rip+0x...]          ; address of the jump table
    movsxd rax, DWORD PTR [rdx+rax*4]  ; load the relative offset
    add    rax, rdx                   ; compute the target address
    jmp    rax                        ; indirect jump

; Jump table (data, not code!)
    .long  case0 - table_base
    .long  case1 - table_base
    .long  case2 - table_base
    ...
```

`objdump` doesn't know the bytes after `jmp rax` are data. It decodes them as instructions, and you see meaningless mnemonics appear:

```asm
    1200:       ff e0                   jmp    rax
    1202:       00 00                   add    BYTE PTR [rax], al
    1204:       1c 00                   sbb    al, 0x0
    1206:       00 00                   add    BYTE PTR [rax], al
    1208:       38 00                   cmp    BYTE PTR [rax], al
    120a:       00 00                   add    BYTE PTR [rax], al
```

These `add BYTE PTR [rax], al` don't exist in the program — they are 32-bit offsets (`0x0000001c`, `0x00000038`…) misinterpreted as opcodes. An experienced reverse engineer recognizes the pattern (a sequence of `add BYTE PTR [rax], al` or short, repetitive instructions right after an indirect `jmp`), but a beginner can waste a lot of time trying to understand these "instructions".

A recursive disassembler like Ghidra or IDA identifies the `jmp rax` as an indirect jump, analyzes the constraints on `rax` (the preceding `cmp eax, 7`), recognizes the *jump table* pattern, and **labels the following bytes as data** instead of decoding them as code. The *jump table* then appears correctly in the listing, and each `case` is identified.

### Desynchronization

x86-64 instructions have variable length: from 1 byte (`ret` = `c3`) to 15 bytes. If `objdump` starts decoding at the wrong place — typically after traversing data inserted in `.text` — it can land "in the middle" of a multi-byte instruction. The decoder desynchronizes: it interprets a fragment of one instruction as the start of another, producing a cascade of fantasy instructions.

The decoder usually **resynchronizes** after a few instructions, because x86 opcodes have characteristic lengths and invalid sequences are rare in real code. But between the desynchronization point and resynchronization, the listing is unusable.

Recursive disassemblers don't encounter this problem, because they only decode paths reachable from known entry points. If they have no reason to decode a zone, they leave it as raw data.

## Limitation 2: no control-flow graph

`objdump` produces a **linear** listing: instructions scroll from top to bottom, in address order. Conditional and unconditional jumps appear in the text, but you must mentally follow the target addresses and reconstruct the flow graph in your head.

On a 20-instruction function with a single `if/else`, it's feasible. On a 200-instruction function with nested loops, `switch`es, and error-handling branches, mental jump-tracking quickly becomes impractical. You spend your time searching for target addresses in the listing, going back, losing the thread.

Modern disassemblers offer **graph views** that turn the linear listing into a diagram of *basic blocks* connected by arrows (green for the taken branch, red for not taken). This representation makes the code structure immediately visible: loops (arrows going back up), branches (`if/else` in a diamond shape), `switch`es (a node with multiple outgoing arrows).

Ghidra calls this view the "Function Graph". IDA calls it the "Graph View". Radare2/Cutter offers it via the `VV` mode. All transform the same information — an instruction listing with jumps — into a representation the human brain processes much more effectively.

`objdump` simply does not have this capability. It's raw text, period.

---

## Limitation 3: no decompilation

The feature that most radically separates `objdump` from modern RE tools is **decompilation**: the automatic translation of assembly into readable C pseudo-code.

When Ghidra shows you:

```c
int check_serial(char *input, char *expected) {
    int hash = compute_hash(input);
    char buffer[32];
    sprintf(buffer, "%08x", hash);
    return strcmp(buffer, expected) == 0;
}
```

… alongside the assembly listing, the time savings are considerable. The pseudo-code isn't perfect (variable names are invented, types are sometimes approximate, some constructions are poorly reconstructed), but it gives a near-instantaneous overview of what the function does.

With `objdump`, you have the same bytes at your disposal, but you must **mentally** reconstruct this pseudo-code by reading each instruction one by one. It's a formative exercise (and that's why we did it in this chapter), but on a real-sized binary, it's not viable as a daily method of work.

The main decompilers are:

- **Ghidra** (free, open source) — covered in Chapter 8  
- **IDA + Hex-Rays** (commercial) — covered in Chapter 9  
- **Binary Ninja** (commercial, free cloud version) — covered in Chapter 9  
- **RetDec** (free, open source, Avast) — covered in Chapter 20

## Limitation 4: no cross-references (XREF)

When you analyze a function in `objdump`, one question keeps coming back: **who calls this function?** And symmetrically: **this string, where is it used?** These questions concern *cross-references* (XREF) — the bidirectional links between a point in the code and the places that reference it.

With `objdump`, answering these questions requires manual `grep`ping:

```bash
# Who calls the function at 0x1139?
$ objdump -d -M intel keygenme_strip | grep "call.*1139"
    11f5:       e8 3f ff ff ff          call   1139
```

This `grep` works for direct `call`s, but misses the following cases:

- **Indirect calls** (`call rax` where `rax` contains `0x1139`) — `grep` can't resolve them without flow analysis.  
- **References to data** — a string in `.rodata` referenced by a `lea rdi, [rip+0x...]` is not findable by a `grep` on the string's address, because the listing shows the RIP-relative offset, not the final address.  
- **Function pointers stored in arrays** (vtables, callback arrays) — the reference is in `.data.rel.ro`, not in the code.  
- **Multiple references** — a function called from 15 different places requires a `grep` that can return noise if the address appears as a constant elsewhere.

Ghidra, IDA, and Radare2 automatically build a **complete index of all cross-references** during initial analysis. For each function, global variable, string, you can instantly obtain the list of every place that references it (and conversely). It's a relational database of the binary, navigable in one click.

This capability fundamentally changes the way you work. Instead of looking for a needle in a listing, you navigate a graph of relations.

## Limitation 5: no typing or structure reconstruction

When you read a memory access like `mov eax, DWORD PTR [rbx+0x14]` in `objdump`, you know a 4-byte integer is read at offset `0x14` relative to `rbx`. But you don't know what that offset **represents**. Is it the `age` field of a `Person` structure? The `length` field of a buffer? The fourth element of an integer array?

`objdump` works at the level of bytes and addresses. It has no notion of types, structures, classes, or memory layouts.

Advanced disassemblers let you **define types** and apply them to memory accesses. In Ghidra, you can create a structure:

```c
struct Animal {
    void **vtable;     // offset 0x00
    char name[32];     // offset 0x08
    int age;           // offset 0x28
};
```

Once this structure is applied to the `rbx` register, the access `[rbx+0x14]` transforms into `animal->name[12]` in the listing and in the decompiler. The code suddenly becomes readable, and memory accesses take on meaning.

This structure-reconstruction capability is covered in detail in Chapter 8 (section 8.6) for Ghidra and in Chapter 17 for specific C++ structures (vtables, RTTI, STL).

## Limitation 6: no inter-procedural analysis

`objdump` disassembles each instruction individually. It doesn't "understand" relationships between functions: what is the complete call graph of the program? Which functions are *leaf functions*? What is the maximum call-stack depth? Which functions access a given global variable?

Modern RE tools build a global **call graph** showing caller/callee relationships between every function of the program. This graph lets you:

- Identify "central" functions (called from many places) — often utility functions.  
- Spot "leaf" functions (that don't call any other function) — often pure-computation functions.  
- Trace the execution path from `main()` to a target function — essential to understand the context in which a function is invoked.  
- Identify dead functions (never called) — potentially debug code or disabled functions.

With `objdump`, you can manually build a call graph by collecting every `call` with `grep`, but it's tedious and incomplete (indirect calls escape this method).

---

## Limitation 7: no analysis persistence

When you spend an hour analyzing a binary with `objdump` — identifying functions, guessing their roles, understanding data structures — all this work exists only in your head (and perhaps in handwritten notes or comments in a text file). If you close the terminal and come back the next day, you start over from scratch.

Graphical disassemblers offer **persistent projects**. Ghidra saves your analysis in a project: the functions you renamed, the comments you added, the types you defined, the bookmarks you placed — everything is kept. You can interrupt your work and resume it days later without losing context.

IDA goes further with its `.idb`/`.i64` files that capture the complete state of the analysis. Radare2 offers "projects" with a similar mechanism. Binary Ninja saves its annotations in dedicated databases.

`objdump` saves nothing. Every execution starts from scratch. That's perfect for a quick glance, insufficient for long-running analysis.

---

## Limitation 8: no multi-architecture support in a single workflow

`objdump` supports many architectures (x86, ARM, MIPS, PowerPC, RISC-V…) via the `-m` option or automatically if the binary contains the information. But it can only disassemble **one architecture at a time**, and offers no mechanism to compare, correlate, or navigate between binaries of different architectures.

Ghidra, on the other hand, can open an x86-64 binary and an ARM binary simultaneously in the same project, apply the same data structures to both, and compare the code produced by different compilers for the same source. It's crucial for firmware analysis (where the same code is sometimes compiled for multiple targets) and for multi-platform RE.

---

## When to stay on `objdump`, when to switch

The limitations listed above don't mean `objdump` is obsolete. It remains the best choice in certain contexts:

| Situation | Recommended tool |  
|---|---|  
| Quick look at a binary (triage) | `objdump` |  
| Check the real content of a section | `objdump` |  
| Compare two compilations (quick diff) | `objdump` + `diff` |  
| Minimal environment (SSH server, container) | `objdump` (only tool available) |  
| Scripting and shell automation | `objdump` + `grep`/`awk`/`sed` |  
| Confirm a suspicious Ghidra/IDA output | `objdump` (ground truth) |  
| Complete analysis of a 50+ function stripped binary | Ghidra / IDA / Radare2 |  
| Reconstruction of C++ structures and classes | Ghidra / IDA |  
| Control-flow tracking in a complex function | Ghidra (graph view) / IDA |  
| Malware analysis with obfuscation | Ghidra + dynamic analysis |  
| Team work on a binary | Ghidra (shared projects) / IDA |

The practical rule is simple: **start with `objdump`** for triage and initial understanding (that's the workflow of Chapter 5, section 5.7). If the analysis exceeds a few minutes and the binary has more than a handful of functions, **import it into Ghidra**. You can always come back to `objdump` occasionally to verify a detail or to script an extraction.

## What you gained in this chapter

Despite its limitations, the work with `objdump` we did in this chapter gave you something no graphical tool can replace: the ability to **read raw assembly**. When Ghidra displays a listing in its CodeBrowser, it's the same assembly as `objdump`'s — with additional annotations, certainly, but the base is identical. When Ghidra's decompiler produces suspicious pseudo-code, it's by returning to the assembly that you'll verify whether it's right.

The skills built in this chapter — spotting function boundaries, reading prologues and epilogues, identifying the optimization level, finding `main()` on a stripped binary, demangling C++ symbols — are skills you'll use **inside** Ghidra, **inside** IDA, **inside** GDB, and in any other tool. `objdump` was the learning vehicle; the reflexes acquired are portable.

Chapter 8 will open Ghidra and show you how these same analyses become faster, deeper, and more comfortable with a tool designed for RE. You'll arrive there with the necessary foundations to get the most out of it.

---

## Summary

`objdump` is a linear disassembler: it decodes bytes sequentially without following branches. This approach makes it vulnerable to data inserted in code (jump tables, constants), to desynchronization, and incapable of reconstructing the control-flow graph. Beyond the disassembly algorithm, `objdump` offers no decompilation, no cross-references, no structure typing, no call graph, no analysis persistence. These features are precisely what modern disassemblers like Ghidra, IDA, Radare2, and Binary Ninja provide. That said, `objdump` remains irreplaceable for quick triage, scripting, verifying a suspicious output from another tool, and working in a minimal environment. The challenge isn't choosing one or the other, but knowing when to use each.
