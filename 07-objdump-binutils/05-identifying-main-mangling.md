# 7.5 — Identifying `main()` and C++ functions (name mangling)

> 🔧 **Tools used**: `objdump`, `readelf`, `nm`, `c++filt`, `strings`  
> 📦 **Binaries**: `keygenme_O0`, `keygenme_strip` (`binaries/ch07-keygenme/` directory) and `oop` (`binaries/ch22-oop/` directory)  
> 📝 **Syntax**: Intel (via `-M intel`)

---

## Finding `main()`: the starting point of any analysis

The first question a reverse engineer asks facing an unknown binary is almost always: **where is `main()`?** That's where the program's logic begins — the code before `main()` is C *runtime* initialization machinery, rarely interesting for analysis. Finding `main()` places you directly at the heart of the subject.

On a non-stripped binary, it's trivial:

```bash
$ objdump -d -M intel keygenme_O0 | grep '<main>'
00000000000011e2 <main>:
```

Just search for the label. But on a stripped binary, `main` has disappeared from the symbols. You have to find it otherwise.

---

## Method 1: trace back from `_start` via `__libc_start_main`

The real entry point of an ELF binary is not `main()` — it's `_start`. Even on a stripped binary, the entry point is recorded in the ELF header and `readelf` reveals it:

```bash
$ readelf -h keygenme_strip | grep "Entry point"
  Entry point address:               0x1060
```

Let's disassemble at that address:

```bash
$ objdump -d -M intel --start-address=0x1060 --stop-address=0x1090 keygenme_strip
```

You'll find the `_start` routine, generated by the C *runtime* (crt1.o). Its structure is standardized:

```asm
0000000000001060 <.text>:
    1060:       f3 0f 1e fa             endbr64
    1064:       31 ed                   xor    ebp, ebp
    1066:       49 89 d1                mov    r9, rdx
    1069:       5e                      pop    rsi
    106a:       48 89 e2                mov    rdx, rsp
    106d:       48 83 e4 f0             and    rsp, 0xfffffffffffffff0
    1071:       50                      push   rax
    1072:       54                      push   rsp
    1073:       45 31 c0                xor    r8d, r8d
    1076:       31 c9                   xor    ecx, ecx
    1078:       48 8d 3d 63 01 00 00    lea    rdi, [rip+0x163]
    107f:       ff 15 53 2f 00 00       call   QWORD PTR [rip+0x2f53]
    1085:       f4                      hlt
```

The key is the instruction `lea rdi, [rip+0x163]` at address `0x1078`. It loads the address of… `main()` into `rdi`. Indeed, the C *runtime* convention on Linux is to call `__libc_start_main` with `main` as the first argument (in `rdi`). The following `call` (to `__libc_start_main@plt` via the GOT) therefore receives the address of `main` in `rdi`.

Let's compute the address: the `lea` is at `0x1078`, it's 7 bytes (so the next instruction is at `0x107f`), and the RIP-relative displacement is `0x163`. The address of `main` is therefore:

```
0x107f + 0x163 = 0x11e2
```

Let's check on the non-stripped version: `main` is indeed at `0x11e2`. The method works.

> 💡 **This technique is universal** for ELF binaries dynamically linked with glibc. The pattern may vary slightly depending on the C *runtime* version and compilation options (PIE vs non-PIE), but the principle remains the same: `_start` loads the address of `main` into `rdi` before calling `__libc_start_main`. Look for the `lea rdi, [rip+...]` in the first instructions of `_start`.

### Automating with a one-liner

For those in a hurry, here is a quick extraction:

```bash
# Display the first instructions starting at the entry point
entry=$(readelf -h keygenme_strip | grep "Entry point" | awk '{print $4}')  
objdump -d -M intel --start-address=$entry --stop-address=$((entry + 0x40)) keygenme_strip  
```

Look for the `lea rdi` in the output — the target address is your `main`.

## Method 2: look for characteristic strings

This is a more pragmatic approach, which requires no knowledge of the C *runtime*. Most programs display something to the user: a welcome message, a prompt, a usage, or an error message. These strings live in `.rodata` and are referenced by the code via RIP-relative `lea`s.

```bash
$ strings keygenme_strip
...
Usage: %s <serial>  
Valid serial!  
Invalid serial.  
...
```

The string `"Usage: %s <serial>"` is probably used in `main()`, because it's the one that handles command-line arguments. Let's find where it's referenced:

```bash
$ objdump -d -M intel keygenme_strip | grep -B2 -A2 "Usage"
```

This won't work directly — `objdump -d` does not display strings in clear in the disassembly. But we can proceed otherwise: find the string's address in `.rodata`, then look for that address in the code.

```bash
# Find the string's offset
$ objdump -s -j .rodata keygenme_strip | grep "Usage"
 2004 55736167 653a2025 73203c73 65726961  Usage: %s <seria
```

The string is at address `0x2004`. In the code, it will be loaded via a RIP-relative `lea`. Let's look in the disassembly for accesses to `.rodata` followed by a `call` to `printf@plt` or `puts@plt`:

```bash
$ objdump -d -M intel keygenme_strip | grep -A3 "lea.*rdi"
```

Among the results, identify the `lea rdi, [rip+...]` whose address calculation points to `0x2004`. The instruction containing this `lea` is in `main()` — or in a function called directly by `main()` to display the usage.

This method takes a bit more work, but it works even when the binary has a non-standard entry point or when `_start` has been customized.

## Method 3: identify `main` by its behavioral signature

If the previous two methods fail, or if you want to confirm your find, `main()` has recognizable characteristics:

**It receives `argc` and `argv`.** In System V convention, `main(int argc, char **argv)` receives `argc` in `edi` and `argv` in `rsi`. You'll often see a comparison of `edi` with a constant in the first instructions (argument-count check):

```asm
; argc == 2 check
cmp    edi, 0x2  
jne    <usage_branch>  
```

**It calls I/O functions.** An interactive program's `main()` typically calls `printf`, `puts`, `scanf`, `fgets`, or similar in the first dozens of instructions. If an unidentified function does a `call printf@plt` and a `cmp edi, 2` at the start, it is very likely `main`.

**It ends with a return code.** Before its `ret`, `main()` generally loads a small integer into `eax`: `0` for success, `1` for error. Often both return paths are visible:

```asm
; Success path
xor    eax, eax                ; return 0  
jmp    .epilogue  

; Error path
mov    eax, 0x1                ; return 1

.epilogue:
leave  
ret  
```

## The C *runtime* functions: background noise

While searching for `main()`, you'll inevitably come across many functions that don't interest you. Knowing how to recognize them saves you time.

On a typical ELF binary dynamically linked with glibc, the `.text` section contains, in addition to your code, several functions injected by the C *runtime* and the linker. They appear systematically, even in a `hello world`:

| Function | Role | How to recognize it |  
|---|---|---|  
| `_start` | ELF entry point, calls `__libc_start_main` | At the *entry point* address, contains `xor ebp, ebp` |  
| `_init` | Global initialization (`.init` section) | Disassembled in `Disassembly of section .init` |  
| `_fini` | Finalization (`.fini` section) | Disassembled in `Disassembly of section .fini` |  
| `__do_global_dtors_aux` | Destruction of C++ global objects | Contains access to a `completed.0` flag in `.bss` |  
| `frame_dummy` | Prepares the frame for destructors | Very short, just before your functions |  
| `register_tm_clones` | Transactional memory management | Contains accesses to `__TMC_END__` |  
| `deregister_tm_clones` | Same | Mirror of the previous one |  
| PLT stubs | Trampolines to dynamic functions | `.plt` section, identified by `jmp QWORD PTR [rip+...]` |

These functions form a "background noise" surrounding your code. At `-O0`, your first user function appears after `frame_dummy`. In practice, if you see `_start`, `deregister_tm_clones`, `register_tm_clones`, `__do_global_dtors_aux`, `frame_dummy`, then an `endbr64`/`push rbp` — the function following `frame_dummy` is your first user function.

On a stripped binary, these *runtime* functions are still present but unlabeled. You'll recognize them by their position (very start of `.text`), by their brevity, and by the fact that they do nothing "logical" from the program's perspective. Skip them and focus on the functions that come after.

## C++ *name mangling*: when names become cryptic

Now let's move to C++ binaries. Let's disassemble the OOP binary from Chapter 22:

```bash
$ objdump -d -M intel binaries/ch22-oop/oop | less
```

You discover function names like:

```
0000000000001234 <_ZN6Animal5speakEv>:
00000000000012a0 <_ZN3Dog5speakEv>:
0000000000001310 <_ZN3Cat5speakEv>:
0000000000001380 <_ZN6AnimalC2ENSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEEi>:
0000000000001450 <_ZN3DogC1ENSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEEi>:
```

These names are not random noise — they are **name mangling**, a systematic encoding the C++ compiler uses to turn function names into unique identifiers.

### Why name mangling exists

In C, every function has a unique name in the binary: `compute_hash` in C becomes `compute_hash` in the symbol table (sometimes prefixed with `_` depending on the platform). There is no ambiguity.

In C++, function *overloading* allows multiple functions with the same name but different parameters. Methods belong to classes and namespaces. Templates generate multiple specializations. The linker, for its part, works with flat names — it needs a unique identifier for each function. *Name mangling* encodes all the necessary information (namespace, class, method name, parameter types) into a single textual identifier.

### The Itanium ABI: the standard under GCC and Clang

GCC and Clang follow the **Itanium C++ ABI** for mangling. Mangled names always start with `_Z`, which makes them immediately identifiable. The structure follows a hierarchical scheme:

```
_Z <length><namespace/class> <length><method_name> <parameter_types>
```

Let's decode `_ZN6Animal5speakEv` piece by piece:

| Fragment | Meaning |  
|---|---|  
| `_Z` | C++ mangled name prefix |  
| `N` | Start of a qualified name (*nested name*) |  
| `6Animal` | 6-character name: `Animal` |  
| `5speak` | 5-character name: `speak` |  
| `E` | End of qualified name |  
| `v` | Parameter type: `void` (no parameter) |

The decoded result is: `Animal::speak()`.

Let's decode a more complex case — `_ZN6AnimalC2ENSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEEi`:

| Fragment | Meaning |  
|---|---|  
| `_Z` | Mangled prefix |  
| `N` | Start of qualified name |  
| `6Animal` | Class `Animal` |  
| `C2` | Base constructor (*base object constructor*) |  
| `E` | End of qualified name |  
| `NSt7__cxx1112basic_stringIcSt11char_traitsIcESaIcEEE` | `std::__cxx11::basic_string<char, std::char_traits<char>, std::allocator<char>>` — that is `std::string` |  
| `i` | `int` |

The decoded result is: `Animal::Animal(std::string, int)` — the `Animal` constructor taking a `std::string` and an `int`.

### Common type encodings

Here are the type codes you'll most often encounter in mangled names:

| Code | C++ type |  
|---|---|  
| `v` | `void` |  
| `b` | `bool` |  
| `c` | `char` |  
| `i` | `int` |  
| `j` | `unsigned int` |  
| `l` | `long` |  
| `m` | `unsigned long` |  
| `f` | `float` |  
| `d` | `double` |  
| `PKc` | `const char*` (P = pointer, K = const, c = char) |  
| `Ri` | `int&` (R = reference) |

### Special codes for constructors and destructors

| Code | Meaning |  
|---|---|  
| `C1` | Complete constructor (*complete object constructor*) |  
| `C2` | Base constructor (*base object constructor*) |  
| `D0` | Deleting destructor (*deleting destructor*) |  
| `D1` | Complete destructor (*complete object destructor*) |  
| `D2` | Base destructor (*base object destructor*) |

The distinction between `C1`/`C2` and `D0`/`D1`/`D2` is related to virtual inheritance. In practice, for most simple classes, GCC generates a `C1` that calls `C2`, and both have nearly identical content. In RE, you can often treat them as the same constructor.

### Overloaded operators

C++ operators have specific encodings:

| Code | Operator |  
|---|---|  
| `ls` | `operator<<` (stream insertion) |  
| `rs` | `operator>>` (stream extraction) |  
| `pl` | `operator+` |  
| `eq` | `operator==` |  
| `cl` | `operator()` (function call) |  
| `ix` | `operator[]` (indexing) |

You'll encounter them frequently in C++ binaries that use streams (`std::cout << ...`) or STL containers.

## Quick demangling in `objdump`

Rather than manually decoding each symbol, `objdump` offers an integrated option:

```bash
objdump -d -M intel -C binaries/ch22-oop/oop | less
```

The **`-C`** option (or `--demangle`) enables automatic demangling. Labels become readable:

```
0000000000001234 <Animal::speak()>:
00000000000012a0 <Dog::speak()>:
0000000000001310 <Cat::speak()>:
0000000000001380 <Animal::Animal(std::__cxx11::basic_string<...>, int)>:
```

The readability difference is spectacular. It's an option to enable **systematically** on C++ binaries:

```bash
# Recommended alias for daily work on C++
alias objdump='objdump -M intel -C'
```

The `-C` option also works with `nm`:

```bash
$ nm -C binaries/ch22-oop/oop | grep ' T '
0000000000001234 T Animal::speak()
00000000000012a0 T Dog::speak()
0000000000001310 T Cat::speak()
...
```

And with `readelf` (via the `--demangle` or `-C` option depending on the version).

## Identifying C++ components in an `objdump` listing

Even without demangling, some patterns in mangled names give you immediate clues about the code's nature:

### Spotting classes

Look for common prefixes in mangled names:

```bash
$ nm binaries/ch22-oop/oop | grep '_ZN' | sed 's/_ZN\([0-9]*\)\(.*\)/\2/' | sort -u
```

Or more simply, with demangling:

```bash
$ nm -C binaries/ch22-oop/oop | grep '::' | awk -F'::' '{print $1}' | sort -u
```

This gives you the list of classes (and namespaces) present in the binary.

### Spotting vtables and RTTI

Vtables appear in symbols under the mangling `_ZTV`:

```bash
$ nm binaries/ch22-oop/oop | grep '_ZTV'
0000000000003d00 V _ZTV6Animal
0000000000003d28 V _ZTV3Cat
0000000000003d50 V _ZTV3Dog
```

RTTI (*Run-Time Type Information*) appears under `_ZTI` (typeinfo) and `_ZTS` (typeinfo name):

```bash
$ nm binaries/ch22-oop/oop | grep '_ZTI\|_ZTS'
0000000000002050 V _ZTI6Animal
0000000000002068 V _ZTI3Cat
0000000000002080 V _ZTI3Dog
0000000000002040 V _ZTS6Animal
0000000000002048 V _ZTS3Cat
0000000000002050 V _ZTS3Dog
```

This information is precious: even on a partially stripped binary (where `.symtab` has disappeared but dynamic symbols remain), vtables and RTTI may subsist if classes are exported or used with `dynamic_cast`. The RTTI strings (`_ZTS`) contain class names in clear in `.rodata`:

```bash
$ strings binaries/ch22-oop/oop | grep -E '^[0-9]+[A-Z]'
6Animal
3Cat
3Dog
```

These strings (format `<length><name>`) are the RTTI type names. Even on a fully stripped binary, `strings` can reveal them — it's an unexpected vein to recover class names.

### Spotting constructors and destructors

The `C1`, `C2` (constructors) and `D0`, `D1`, `D2` (destructors) patterns in mangled names are priority targets in RE:

```bash
# Constructors
$ nm binaries/ch22-oop/oop | grep '_ZN.*C[12]E'

# Destructors
$ nm binaries/ch22-oop/oop | grep '_ZN.*D[012]E'
```

Analyzing a constructor reveals member initialization of the class (and therefore their types and offsets in the structure). Analyzing a destructor shows you the released resources (and therefore what the class owns).

### Spotting STL functions

Mangled names containing `St` (for `std::`) or `NSt` (for a qualified name in `std::`) come from the C++ standard library. You'll see a lot of:

- `_ZNSt7__cxx1112basic_stringI...` → `std::string` methods  
- `_ZNSt6vectorI...` → `std::vector` methods  
- `_ZNSolsI...` → `std::ostream::operator<<`  
- `_ZStlsI...` → free-function `operator<<` for streams

In RE, these STL functions are generally "noise" you want to traverse quickly to reach the business logic. Recognizing them at first glance (even in mangled form) saves you a lot of time.

## The stripped C++ binary case

On a stripped C++ binary, mangled names from `.symtab` have disappeared. However:

**Dynamic symbols survive.** If the binary is dynamically linked to `libstdc++`, calls to STL functions remain visible in `.dynsym` and in the PLT:

```bash
$ nm -D binaries/ch22-oop/oop_strip | c++filt
                 U std::basic_ostream<char, std::char_traits<char> >& std::operator<< <...>(...)
                 U operator new(unsigned long)
                 U operator delete(void*)
                 U __cxa_atexit
                 U __cxa_begin_catch
                 U __cxa_end_catch
```

The `__cxa_*` symbols are characteristic of the C++ *runtime* and confirm the binary is C++ — even if no user class or method name is visible. These indicators let you adapt your approach before even starting disassembly: you know you'll deal with vtables, virtual dispatch, exceptions, and potentially STL.

**RTTI strings can survive stripping.** As mentioned above, the type names used by `dynamic_cast` and `typeid` are strings stored in `.rodata`. They are not part of `.symtab` and therefore survive `strip`. On a stripped C++ binary, run `strings` and look for patterns `<number><Name>`:

```bash
$ strings binaries/ch22-oop/oop_strip | grep -E '^[0-9]+[A-Z][a-z]'
6Animal
3Cat
3Dog
```

You just recovered class names without any symbol. It's not guaranteed (if the program is compiled with `-fno-rtti`, these strings don't exist), but it's common enough to deserve being systematically checked.

**Vtables remain in the `.data.rel.ro` or `.rodata` section.** Even stripped of their names, vtables are arrays of function pointers located in read-only data sections. You spot them by looking for sequences of pointers that point to the `.text` section. Chapter 17 (section 17.2) covers vtable reconstruction in detail — for now, remember that their presence betrays C++ polymorphism.

## Recognizing a C++ binary without symbols: converging clues

How to know a stripped binary is C++ rather than pure C? Here is a quick checklist of revealing signs, even in complete absence of symbols:

1. **Calls to `__cxa_*` in the PLT**: `__cxa_throw`, `__cxa_begin_catch`, `__cxa_end_catch`, `__cxa_atexit` — these functions are part of the C++ *runtime* and don't exist in a pure C program.

2. **Calls to `operator new` and `operator delete` in the PLT**: C++ allocation (instead of `malloc`/`free`).

3. **RTTI strings in `.rodata`**: format `<length><ClassName>`, like `6Animal`, `3Dog`.

4. **`.gcc_except_table` and `.eh_frame` sections**: C++ exception handling. The `.eh_frame` section also exists in C (for stack unwinding), but `.gcc_except_table` is C++-specific.

5. **Links to `libstdc++.so`**: verifiable with `ldd` or `readelf -d`.

6. **Indirect calls via arrays of pointers in `.data.rel.ro`**: pattern characteristic of virtual dispatch (call through vtable).

7. **Functions called right after loading a pointer from an object**: the pattern `mov rax, QWORD PTR [rdi]` followed by `call QWORD PTR [rax+N]` is a virtual method call. The first `mov` loads the object's vptr, the second calls the N-th entry of the vtable.

None of these clues is absolute proof in isolation, but their combination is near-certain. If you see `__cxa_*`, `operator new`, RTTI strings, and indirect calls via vtable, you're dealing with C++ — adapt your analysis strategy accordingly.

## Summary

Finding `main()` on a stripped binary relies on three complementary techniques: tracing back from the `_start` entry point by reading the `lea rdi` that loads the address of `main` before the call to `__libc_start_main`; searching for characteristic strings in `.rodata` (usage messages, prompts) and tracking their reference in the code; identifying `main`'s behavioral signature (argc check, I/O calls, return codes). For C++ binaries, the Itanium *name mangling* encodes class names, method names, and parameter types into identifiers starting with `_Z`, decodable automatically with `objdump`'s or `nm`'s `-C` option. The `_ZTV` (vtables), `_ZTI`/`_ZTS` (RTTI) prefixes, and the `C1`/`C2`/`D0`/`D1`/`D2` codes (constructors/destructors) are essential landmarks. Even on a stripped C++ binary, dynamic symbols (`__cxa_*`, `operator new`), RTTI strings, and virtual-dispatch patterns make it possible to identify the language and start reconstructing the class hierarchy.
