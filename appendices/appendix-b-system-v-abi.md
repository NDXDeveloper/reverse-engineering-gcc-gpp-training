# Appendix B — System V AMD64 ABI Calling Conventions (Summary Table)

> 📎 **Reference Sheet** — This appendix details the calling convention used by **all** ELF x86-64 binaries on Linux (and most Unix systems: macOS, FreeBSD, Solaris). This is the convention that GCC, G++, Clang, and Rust apply by default. Mastering it is essential for reading any disassembly: without it, you cannot know which register holds which argument, which function is responsible for saving what, or how to interpret the stack.

---

## 1 — Overview

The System V AMD64 calling convention (formally defined in the *System V Application Binary Interface — AMD64 Architecture Processor Supplement* document) governs three fundamental aspects of each function call:

- **How arguments are passed** from the caller to the callee  
- **How the return value** is sent back to the caller  
- **Which registers** must be preserved by the callee and which can be freely clobbered

Unlike the x86 32-bit convention (where almost everything was passed on the stack), System V AMD64 heavily favors **registers** for argument passing. This is what makes x86-64 code more compact and faster, but it also requires the reverse engineer to know precisely how registers are assigned.

---

## 2 — Passing Integer and Pointer Arguments

The first six arguments of integer, pointer, or reference type are passed in dedicated registers, in a fixed order. Beyond six, additional arguments are pushed onto the stack.

### 2.1 — Integer Argument Registers

| Position | Register | Example C Types |  
|----------|----------|-----------------|  
| 1st argument | `rdi` | `int`, `long`, `size_t`, `char *`, `void *`, `struct *` |  
| 2nd argument | `rsi` | same |  
| 3rd argument | `rdx` | same |  
| 4th argument | `rcx` | same |  
| 5th argument | `r8` | same |  
| 6th argument | `r9` | same |  
| 7th and beyond | stack (in left→right order, ascending addresses from `rsp`) | same |

> 💡 **Mnemonic device**: **D**iane **S**hould **D**ance **C**almly **8**→**9** — `rDi`, `rSi`, `rDx`, `rCx`, `r8`, `r9`. There are many variants of such mnemonic phrases; choose the one that works for you, because this order is the cornerstone of all x86-64 assembly reading.

### 2.2 — Arguments on the Stack

When a function has more than six integer arguments, the extra arguments are pushed onto the stack **right to left** (the 7th argument ends up at the lowest address, just above the return address). After the `call`, the stack layout as seen by the callee is:

```
              ┌───────────────────────┐  high addresses
              │     9th argument      │  [rsp + 0x20]
              ├───────────────────────┤
              │     8th argument      │  [rsp + 0x18]
              ├───────────────────────┤
              │     7th argument      │  [rsp + 0x10]
              ├───────────────────────┤
              │    return address     │  [rsp + 0x08]   ← pushed by call
              ├───────────────────────┤
              │     (local frame)     │  [rsp]           ← top of stack
              └───────────────────────┘  low addresses
```

> ⚠️ **Warning**: the offsets above are those **at function entry**, before the prologue adjusts `rsp`. After a `push rbp` / `sub rsp, N`, all offsets shift accordingly.

In practice, functions with more than 6 integer arguments are uncommon in idiomatic C/C++ code, but they do exist (some libc functions like `mmap` have exactly 6, and custom functions or those generated by frameworks may have more).

### 2.3 — Argument Counting Specifics

Argument counting follows the types **after promotion**:

- A `char` or `short` passed to a function is promoted to `int` (32 bits) but still occupies a full 64-bit register. GCC may use `edi` (32 bits) instead of `rdi` (64 bits) to load the value, since writing to a 32-bit register automatically zeroes the upper 32 bits.  
- A `bool` occupies a full integer register (typically via `edi`), with the value 0 or 1.  
- `enum` values are treated as integers of the underlying size (usually `int`).

## 3 — Passing Floating-Point Arguments

Arguments of type `float`, `double`, and `long double` (80 bits, x87) follow a separate path.

### 3.1 — Floating-Point Argument Registers

| Position | Register | Types |  
|----------|----------|-------|  
| 1st float argument | `xmm0` | `float`, `double` |  
| 2nd float argument | `xmm1` | same |  
| 3rd float argument | `xmm2` | same |  
| 4th float argument | `xmm3` | same |  
| 5th float argument | `xmm4` | same |  
| 6th float argument | `xmm5` | same |  
| 7th float argument | `xmm6` | same |  
| 8th float argument | `xmm7` | same |  
| 9th and beyond | stack | same |

### 3.2 — Separate Counters

A crucial point: the integer and floating-point argument counters are **independent**. Each type draws from its own register bank without affecting the other. Consider the following signature:

```c
void draw(int x, double opacity, int y, float scale, const char *label);
```

The register assignment will be:

| Argument | Type | Register |  
|----------|------|----------|  
| `x` | `int` | `edi` (1st integer) |  
| `opacity` | `double` | `xmm0` (1st float) |  
| `y` | `int` | `esi` (2nd integer) |  
| `scale` | `float` | `xmm1` (2nd float) |  
| `label` | `const char *` | `rdx` (3rd integer) |

The positions of arguments in the C signature determine the logical order, but each argument "draws" from the next available register in its category (integer or floating-point). There is no "hole" left in the integer registers for a floating-point argument.

### 3.3 — The `al` Register and Variadic Functions

For variadic functions (`printf`, `scanf`, and any function declared with `...`), the ABI requires that **`al` contains the number of SSE registers used** for arguments (from 0 to 8). This is why you will often see before a `call printf`:

```asm
xor     eax, eax          ; al = 0 (no floating-point arguments)  
call    printf@plt  
```

Or, if a `double` is passed:

```asm
mov     eax, 1            ; al = 1 (one floating-point argument in xmm0)  
call    printf@plt  
```

This `xor eax, eax` or `mov eax, N` just before a `call` is a **reliable marker of a variadic function** in RE. If you see it, you know the called function accepts a variable number of arguments.

> ⚠️ For **non-variadic** functions, the value of `al` is unspecified and does not need to be set by the caller. GCC may still emit `xor eax, eax` before certain non-variadic calls for performance reasons (to avoid dependencies on the upper bits of `rax`), which can be misleading. When in doubt, verify the function prototype.

## 4 — Return Values

### 4.1 — Integer Return

| Register | Usage |  
|----------|-------|  
| `rax` | Primary integer return value (up to 64 bits) |  
| `rdx` | Second half if the return is 128 bits (`__int128`, or certain structures of 2 × 64 bits) |

The vast majority of C functions return in `rax` only. Returning via the `rdx:rax` pair is rare outside of 128-bit arithmetic or compact structures with two 64-bit fields.

### 4.2 — Floating-Point Return

| Register | Usage |  
|----------|-------|  
| `xmm0` | `float` or `double` return value |  
| `xmm1` | Second part (structures containing 2 floats) |

### 4.3 — `void` Return

`void` functions do not place anything in `rax`. However, the contents of `rax` after a `call` to a `void` function are **undefined** — it contains a residual value. In RE, do not infer any meaning from a use of `rax` after a call to a function you have identified as `void`.

### 4.4 — Struct Return

Passing and returning structures is the most complex case in the ABI. The behavior depends on the size and composition of the structure:

| Size / Composition | Return Mechanism |  
|-------------------|------------------|  
| ≤ 8 bytes, integer types only | `rax` |  
| 9–16 bytes, integer types only | `rax` + `rdx` |  
| ≤ 8 bytes, floating-point types only | `xmm0` |  
| 9–16 bytes, floating-point types only | `xmm0` + `xmm1` |  
| 8 bytes integer + 8 bytes float (or vice versa) | `rax` + `xmm0` (or `xmm0` + `rax`) |  
| > 16 bytes, or contains non-trivial fields | Via hidden pointer (see below) |

**The hidden pointer**: when a structure is too large to fit in the return registers, the caller allocates space on its own stack and passes a **hidden pointer** as the first argument (in `rdi`). This pointer is invisible in C source code but very real in assembly. The callee writes the result to the address pointed to by `rdi` and returns that same address in `rax`.

This mechanism is a classic trap in RE: if you see a function taking a `rdi` argument that looks like a local pointer belonging to the caller, and that function writes extensively to `[rdi]`, `[rdi+8]`, etc., it is very likely a struct return via hidden pointer. The effective number of "real" arguments is then shifted by one (the 1st visible argument in C is actually in `rsi`, not `rdi`).

## 5 — Register Classification: Caller-Saved vs Callee-Saved

This is one of the most important tables in the entire course. During each function call, certain registers are **guaranteed to be preserved** by the callee (callee-saved, also called *non-volatile*) and others can be **freely clobbered** by the callee (caller-saved, also called *volatile*).

### 5.1 — Complete Table

| Register | Category | Preserved after a `call`? | Primary Usage |  
|----------|----------|--------------------------|---------------|  
| `rax` | Caller-saved | No | Return value, `mul`/`div` result |  
| `rbx` | **Callee-saved** | **Yes** | General purpose |  
| `rcx` | Caller-saved | No | 4th argument, counter for `rep`/shifts |  
| `rdx` | Caller-saved | No | 3rd argument, return extension, `mul`/`div` |  
| `rsi` | Caller-saved | No | 2nd argument, source for `rep movs` |  
| `rdi` | Caller-saved | No | 1st argument, destination for `rep movs/stos` |  
| `rbp` | **Callee-saved** | **Yes** | Frame pointer (if used) or general purpose |  
| `rsp` | **Callee-saved** | **Yes** | Stack pointer (must always be restored) |  
| `r8` | Caller-saved | No | 5th argument |  
| `r9` | Caller-saved | No | 6th argument |  
| `r10` | Caller-saved | No | Temporary use, 4th syscall argument |  
| `r11` | Caller-saved | No | Temporary use, clobbered by `syscall` |  
| `r12` | **Callee-saved** | **Yes** | General purpose |  
| `r13` | **Callee-saved** | **Yes** | General purpose |  
| `r14` | **Callee-saved** | **Yes** | General purpose |  
| `r15` | **Callee-saved** | **Yes** | General purpose |  
| `rip` | — | (modified by `call`/`ret`) | Instruction pointer |  
| `RFLAGS` | Caller-saved | No | Flags (condition codes) |  
| `xmm0–xmm7` | Caller-saved | No | Floating-point arguments and return value |  
| `xmm8–xmm15` | Caller-saved | No | Floating-point temporaries |  
| `x87 FPU stack` | Caller-saved | No | x87 FPU registers (rare in x86-64) |  
| `MMX` | Caller-saved | No | MMX registers (legacy, very rare) |

### 5.2 — Visual Summary

```
╔══════════════════════════════════════════════════════════════╗
║  CALLEE-SAVED (preserved by the called function)             ║
║  rbx, rbp, rsp, r12, r13, r14, r15                           ║
║                                                              ║
║  → If you see push rbx / push r12 in a prologue,             ║
║    it is saving these registers before using them.           ║
╠══════════════════════════════════════════════════════════════╣
║  CALLER-SAVED (may be clobbered by any call)                 ║
║  rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11                   ║
║  xmm0–xmm15, RFLAGS                                          ║
║                                                              ║
║  → After a call, these registers contain potentially         ║
║    different values. Do not assume they retain their         ║
║    pre-call values.                                          ║
╚══════════════════════════════════════════════════════════════╝
```

### 5.3 — Implications in RE

The caller-saved / callee-saved distinction has direct consequences on how you read disassembly:

**Prologues and epilogues** — Callee-saved registers used by a function are saved in the prologue (`push rbx`, `push r12`, etc.) and restored in the epilogue (`pop r12`, `pop rbx`). By counting the callee-saved `push` instructions in the prologue, you can determine which registers the function uses as "register local variables" (GCC allocates frequently used variables in these registers to avoid memory accesses).

**Identifying the number of register local variables** — If a function starts with `push rbx` / `push r12` / `push r13`, it uses three callee-saved registers in addition to its temporary registers. This often indicates a function of moderate complexity with 3 to 5 significant variables.

**Tracking values across `call` instructions** — If a value is in `rbx` and a `call` occurs, the value of `rbx` is still intact after the return (because `rbx` is callee-saved). Conversely, if a value was in `rdi` before the `call`, it is potentially lost. This is why GCC often moves an argument (`rdi`) into a callee-saved register (`rbx`) at the beginning of a function that makes multiple calls: the value will survive all `call` instructions.

## 6 — Standard Prologue and Epilogue

### 6.1 — Prologue with Frame Pointer (`-O0` or `-fno-omit-frame-pointer`)

```asm
push    rbp              ; save the old frame pointer (callee-saved)  
mov     rbp, rsp         ; rbp = new frame pointer  
sub     rsp, 0x30        ; allocate 48 bytes for local variables  
```

With a frame pointer, local variables are accessible via `[rbp - offset]` and stacked arguments (if any) via `[rbp + offset]` (above the return address).

```
              ┌───────────────────────┐  high addresses
              │   stacked arguments   │  [rbp + 0x18], [rbp + 0x10]
              ├───────────────────────┤
              │    return address     │  [rbp + 0x08]
              ├───────────────────────┤
              │  old rbp (saved)      │  [rbp]        ← rbp points here
              ├───────────────────────┤
              │  local variable 1     │  [rbp - 0x08]
              ├───────────────────────┤
              │  local variable 2     │  [rbp - 0x10]
              ├───────────────────────┤
              │        ...            │
              ├───────────────────────┤
              │  local variable N     │  [rbp - 0x30] ← rsp points here
              └───────────────────────┘  low addresses
```

Corresponding epilogue:

```asm
leave                    ; equivalent to: mov rsp, rbp ; pop rbp  
ret                      ; pop rip (return to caller)  
```

Or the expanded form:

```asm
mov     rsp, rbp         ; free local variables  
pop     rbp              ; restore old frame pointer  
ret  
```

### 6.2 — Prologue without Frame Pointer (`-O1` and above with `-fomit-frame-pointer`)

Starting from `-O1`, GCC omits the frame pointer by default. `rbp` is then used as an ordinary callee-saved register, and all local variables are referenced relative to `rsp`:

```asm
push    rbx              ; save a callee-saved register  
push    r12              ; save another one  
sub     rsp, 0x28        ; allocate space for locals + alignment  
```

Corresponding epilogue:

```asm
add     rsp, 0x28        ; free locals  
pop     r12              ; restore  
pop     rbx              ; restore  
ret  
```

Without a frame pointer, offsets relative to `rsp` change every time something is pushed onto or popped from the stack (e.g., to prepare a call). This is what makes optimized code harder to read manually than `-O0` code. Decompilers like Ghidra handle this complexity well by tracking `rsp` adjustments, but when reading manually, you must count each `push`, `pop`, and `sub rsp` to maintain the correct offset.

### 6.3 — The Red Zone

The System V AMD64 ABI defines a **red zone** of 128 bytes below `rsp` (addresses `[rsp-1]` to `[rsp-128]`) that leaf functions (functions that do not call any other function) may use **without adjusting `rsp`**:

```asm
; Simple leaf function — no sub rsp!
mov     dword ptr [rsp-0x4], edi    ; store the 1st argument in the red zone  
mov     eax, dword ptr [rsp-0x4]    ; reload it  
ret  
```

The red zone is guaranteed not to be clobbered by interrupts or signals (the kernel accounts for it). It allows simple leaf functions to avoid the cost of a prologue/epilogue, speeding up small function calls.

In RE, the red zone manifests as accesses to `[rsp - offset]` **without** a preceding `sub rsp`. If you see this in a function that contains no `call`, it is normal red zone usage. If you see it in a function that calls other functions, it is a compilation bug (extremely rare) or your stack analysis is incorrect.

> ⚠️ The red zone does **not** exist in kernel code (`-mno-red-zone`) or in low-level signal handlers, because interrupts could overwrite this area. This is an important distinction if you are analyzing kernel code.

## 7 — Stack Alignment

The ABI requires that the stack be aligned to **16 bytes at the time of the `call`**. This means that `rsp` must be a multiple of 16 just before the `call` instruction executes. After the `call` (which pushes 8 bytes of return address), `rsp` is therefore `16n + 8` at the entry of the called function.

This is why prologues often adjust `rsp` by multiples of 16 (accounting for `push` instructions already performed). If a function does a `push rbp` (8 bytes), `rsp` is once again aligned to 16 and the following `sub rsp` uses a multiple of 16. If it does three `push` instructions (24 bytes = 16 + 8), the `sub rsp` will need to compensate for the extra 8 bytes.

In RE, a standalone `sub rsp, 0x8` (with no apparent local variable) is often **pure alignment padding**. GCC inserts it to guarantee 16-byte alignment before the next `call`. Do not look for a corresponding local variable: there is none.

16-byte alignment is necessary for SSE instructions (`movaps`, etc.) that require aligned memory operands. A misalignment causes a `SIGSEGV` on `movaps` — this is, in fact, a common bug in exploitation.

## 8 — Comparison with the Linux Syscall Convention

Linux x86-64 system calls use a slightly different convention from the function calling convention. Confusing the two is a classic mistake.

| Aspect | Function Call (System V) | System Call (`syscall`) |  
|--------|--------------------------|-------------------------|  
| Number / target | Address in the `call` instruction | Syscall number in `rax` |  
| 1st argument | `rdi` | `rdi` |  
| 2nd argument | `rsi` | `rsi` |  
| 3rd argument | `rdx` | `rdx` |  
| 4th argument | **`rcx`** | **`r10`** |  
| 5th argument | `r8` | `r8` |  
| 6th argument | `r9` | `r9` |  
| Return value | `rax` | `rax` (negative = error = `-errno`) |  
| Clobbered registers | All caller-saved | **`rcx`** and **`r11`** only |  
| Floating-point arguments | `xmm0`–`xmm7` | Not supported (no floating-point arguments via syscall) |

The three differences to remember are as follows. First, the 4th argument: `rcx` for functions, `r10` for syscalls. This difference exists because the `syscall` instruction clobbers `rcx` (it saves `rip` there) and `r11` (it saves `RFLAGS` there), so `rcx` cannot carry an argument. Second, clobbered registers: a `syscall` only destroys `rcx` and `r11`, whereas a function `call` can potentially clobber all caller-saved registers. Third, errors: a negative return in `rax` after a `syscall` indicates an error, with the value being the negation of the `errno` code (for example, `rax = -2` means `ENOENT`).

## 9 — Passing Structures by Value

Passing structures as arguments follows fairly complex classification rules. Here is the practical summary:

### 9.1 — Small Structures (≤ 16 bytes)

Structures of 16 bytes or less are decomposed into "eightbytes" (8-byte blocks) and each block is classified independently:

| Block Content | Class | Register Used |  
|---------------|-------|---------------|  
| Integers, pointers, booleans | INTEGER | Next available integer register (`rdi`, `rsi`, …) |  
| `float`, `double` | SSE | Next available SSE register (`xmm0`, `xmm1`, …) |  
| Mix of integer + float in the same 8-byte block | Depends on merging rules | See the full specification |

Example with a simple structure:

```c
struct Point { int x; int y; };  // 8 bytes total → 1 eightbyte → rdi

void move(struct Point p, int dx);
// p is passed in rdi (x in the low 32 bits, y in the high 32 bits)
// dx is passed in esi
```

Example with a 16-byte structure:

```c
struct Rect { long x; long y; };  // 16 bytes → 2 eightbytes → rdi + rsi

void draw(struct Rect r, int color);
// r.x is passed in rdi, r.y is passed in rsi
// color is passed in edx (3rd integer register)
```

### 9.2 — Large Structures (> 16 bytes)

Structures larger than 16 bytes are passed **via an invisible pointer**: the caller copies the structure onto its stack and passes a pointer to that copy in the next available integer register. The callee works with an independent copy.

In RE, this manifests as a `lea rdi, [rsp+offset]` before the `call`: the caller loads the address of a local area into `rdi`, which looks like pass-by-reference even though the C code passes by value.

### 9.3 — Structures with Constructor/Destructor (C++)

In C++, structures and classes with a non-trivial copy constructor, a non-trivial destructor, or virtual members are **always** passed via an invisible pointer, regardless of their size. This is because the ABI cannot guarantee that the implicit bitwise copy of register passing respects the semantics of the copy constructor.

## 10 — C++ Convention: `this` and Member Functions

In C++ compiled with GCC/G++, non-static member functions receive an **implicit `this` pointer** as the first argument, in `rdi`:

```cpp
class Widget {  
public:  
    void resize(int w, int h);
};

// ABI equivalent:
// void Widget::resize(Widget* this, int w, int h);
// this → rdi, w → esi, h → edx
```

This means that all "visible" arguments are shifted by one position compared to what you would expect in C:

| Visible C++ Argument | Actual Register |  
|----------------------|-----------------|  
| `this` (implicit) | `rdi` |  
| 1st argument | `rsi` |  
| 2nd argument | `rdx` |  
| 3rd argument | `rcx` |  
| 4th argument | `r8` |  
| 5th argument | `r9` |

For **virtual functions**, the mechanism is the same but the call goes through the vtable:

```asm
mov     rax, qword ptr [rdi]        ; load the vptr (first qword of the object)  
call    qword ptr [rax + 0x10]       ; call the 3rd vtable entry (offset / 8 = index)  
```

The offset into the vtable directly gives the virtual method index (each entry is 8 bytes in x86-64): offset `0x00` = 1st virtual method, `0x08` = 2nd, `0x10` = 3rd, etc. Recognizing this pattern (dereferencing `[rdi]` followed by a `call [rax + constant]`) is one of the most reliable indicators for identifying a C++ virtual method call.
