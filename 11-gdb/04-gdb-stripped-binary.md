# 11.4 — GDB on a stripped binary — working without symbols

> **Chapter 11 — Debugging with GDB**  
> **Part III — Dynamic Analysis**

---

## The reality on the ground

Previous sections presented GDB in a comfortable context: a binary compiled with `-g`, resolved function names, local variables accessible by name, line numbers displayed at each step. This comfort almost never exists in real reverse engineering.

The majority of binaries encountered in real situations — commercial software, malware, firmware, CTF binaries — are **stripped**. The `strip` command (or GCC's `-s` flag) has removed the symbol table (`.symtab`) and DWARF sections (`.debug_*`). Sometimes even the `.dynsym` section is reduced to the strict minimum necessary for dynamic resolution.

When you open such a binary in GDB, the difference is immediate:

```
$ gdb -q ./keygenme_O2_strip
Reading symbols from ./keygenme_O2_strip...
(No debugging symbols found in ./keygenme_O2_strip)
(gdb) break main
Function "main" not found.  
Make breakpoint pending on future shared library load? (y or [n]) n  
```

GDB doesn't know `main`. It doesn't know any function in the binary. Commands like `info locals`, `info args`, `list`, `break <name>` are unusable. One might think GDB becomes useless — the opposite is true. It remains an extremely powerful observation tool, but you have to change your method.

## What remains available in a stripped binary

A stripped binary is not a total black box. Several information sources persist:

**The ELF header and entry point.** The `e_entry` field of the ELF header indicates the program's start address (the `_start` generated by libc). This entry point is always present — without it, the loader couldn't launch the program.

```
(gdb) info files
Entry point: 0x401060
  0x00401000 - 0x004011e0 is .text
  0x00402000 - 0x00402030 is .rodata
  ...
```

**The `.dynsym` section and the PLT.** Dynamic symbols — those necessary for resolving shared-library functions — survive stripping. You can still set breakpoints on `strcmp`, `printf`, `malloc`, `open`, etc.:

```
(gdb) break strcmp
Breakpoint 1 at 0x401030
(gdb) break printf
Breakpoint 2 at 0x401040
```

It's precious information: even if you don't know what the binary's internal functions are called, you know what external functions they use.

**The `.rodata` section.** Constant character strings are not affected by stripping. They remain intact and readable:

```
(gdb) x/s 0x402000
0x402000: "Enter your key: "
(gdb) x/s 0x402011
0x402011: "Correct!\n"
(gdb) x/s 0x40201b
0x40201b: "Wrong key!\n"
```

**The `.plt` and `.got` sections.** The dynamic linking mechanism (Chapter 2, section 2.9) remains intact and functional. The `call printf@plt` calls appear clearly in the disassembly.

**The machine code itself.** No byte of the `.text` section is modified by stripping. The code is identical, instruction for instruction, to that of the non-stripped binary. Only the metadata that described it has disappeared.

## Finding `main()` without symbols

The first task on a stripped binary is to locate the `main` function. It no longer appears by name, but its position is deterministic.

### Method 1: trace back from the entry point

The entry point (`_start`) is a minimalist trampoline generated by libc. On a classic GCC/glibc binary, it calls `__libc_start_main` passing `main`'s address as the first argument (in `rdi`). The assembly pattern is stable and recognizable:

```
(gdb) x/20i 0x401060
   0x401060:  endbr64
   0x401064:  xor    ebp,ebp
   0x401066:  mov    r9,rdx
   0x401069:  pop    rsi
   0x40106a:  mov    rdx,rsp
   0x40106d:  and    rsp,0xfffffffffffffff0
   0x401071:  push   rax
   0x401072:  push   rsp
   0x401073:  xor    r8d,r8d
   0x401076:  xor    ecx,ecx
   0x401078:  lea    rdi,[rip+0x111]        # 0x401190
   0x40107f:  call   0x401050 <__libc_start_main@plt>
```

The `lea rdi, [rip+0x111]` instruction loads address `0x401190` into `rdi` — this is the first argument to `__libc_start_main`, and per the glibc convention, this first argument is **`main`'s address**. We just located `main` at `0x401190`.

> 💡 **Variant with recent GCC and glibc ≥ 2.34:** the entry point may call `__libc_start_main_impl` or go through an address table. The principle remains the same — `main` is an argument passed in `rdi` — but the startup code may be slightly different. When in doubt, method 2 is more reliable.

### Method 2: breakpoint on `__libc_start_main`

Rather than reading `_start`'s code, you can let GDB do the work:

```
(gdb) break __libc_start_main
Breakpoint 1 at 0x7ffff7de0b40
(gdb) run
Breakpoint 1, __libc_start_main () at ../csu/libc-start.c:...
(gdb) print/x $rdi
$1 = 0x401190
```

The address in `rdi` when `__libc_start_main` is reached is `main`'s address. You can now set a breakpoint on it:

```
(gdb) break *0x401190
Breakpoint 2 at 0x401190
(gdb) continue
Breakpoint 2, 0x0000000000401190 in ?? ()
```

We're at `main`'s entry. The `?? ()` confirms that GDB doesn't know the function name, but we're there.

### Method 3: use prior static analysis

The most common method in practice is to spot the addresses of interest **in Ghidra or objdump before launching GDB**. Static analysis (Part II) already let us identify functions, their roles, and their addresses. We import this knowledge into GDB as address breakpoints:

```
# Addresses identified in Ghidra
(gdb) break *0x401190         # main
(gdb) break *0x401140         # verification function (check_key)
(gdb) break *0x40117a         # cmp instruction in check_key
```

This is the standard workflow: **Ghidra to understand, GDB to observe**. The two tools are complementary, not interchangeable.

## Navigating without symbols: fundamental techniques

### Setting breakpoints by address

All breakpoints use the address prefixed with `*`:

```
(gdb) break *0x401156
(gdb) break *0x401156 if $rdi != 0     # Conditional breakpoint on address
(gdb) tbreak *0x4011a0                  # Temporary breakpoint on address
```

To set a breakpoint relative to the current instruction pointer:

```
(gdb) break *($rip + 0x20)
```

### Disassembling around the current point

Without symbols, `disassemble` doesn't know function boundaries. The command without arguments tries to guess the bounds, but often fails. Use explicit address ranges instead:

```
(gdb) x/30i $rip
```

Displays the next 30 instructions. To see what precedes the current point:

```
(gdb) x/10i $rip-30
```

Or a precise range:

```
(gdb) disassemble 0x401140, 0x4011a0
```

> ⚠️ **Beware of retrospective disassembly.** x86-64 uses variable-length instructions (1 to 15 bytes). If you start disassembling in the middle of an instruction, the entire listing will be offset and absurd. `x/10i $rip-30` can land in the middle of an instruction and produce a misleading result. To avoid this, use function-start addresses identified in static analysis, or start from a known point (current `$rip`, confirmed breakpoint) and advance forward only.

### Identifying functions by observing patterns

Even without symbols, function boundaries are detectable thanks to GCC's standard prologues and epilogues:

**Classic prologue (non-optimized, `-O0`):**
```
push   rbp  
mov    rbp, rsp  
sub    rsp, 0x30          # Allocates space for local variables  
```

**Classic epilogue:**
```
leave                      # Equivalent to mov rsp, rbp ; pop rbp  
ret  
```

**Optimized prologue (`-O2` and above):**
```
sub    rsp, 0x28           # No frame pointer, direct allocation
```

Or for leaf functions (that don't call anything):
```
# No prologue at all — the function directly uses registers
```

By scanning the `.text` section with `x/100i 0x401000`, you can spot `ret` sequences followed by alignment (`nop`s or `nop DWORD PTR [rax]` as padding) and `push rbp` — each probably marks the start of a new function.

### Manually reconstructing the call stack

On a stripped binary, `backtrace` can be incomplete or incorrect — especially with optimizations that suppress the frame pointer (`-fomit-frame-pointer`, enabled by default at `-O2`). When `bt` shows dubious results, you reconstruct the stack manually.

**With frame pointer (`rbp` used):**

The frame chain is a linked list accessible via `rbp`:

```
(gdb) x/a $rbp           # Saved rbp of the calling frame
0x7fffffffe0f0: 0x7fffffffe130
(gdb) x/a $rbp+8         # Return address → identifies the caller
0x7fffffffe0f8: 0x4011a5

# Move up one level
(gdb) x/a 0x7fffffffe130        # Saved rbp of the next frame
0x7fffffffe130: 0x0
(gdb) x/a 0x7fffffffe138        # Next return address
0x7fffffffe138: 0x7ffff7de0c8a
```

You can thus walk up the stack frame by frame. The address `0x4011a5` correlates with the disassembly: it's the instruction just after the `call` that brought us into the current function.

**Without frame pointer (`rbp` not used):**

It's the most frequent case at `-O2`. The stack doesn't contain a chain of saved `rbp`s, and you have to scan values on the stack to identify those that look like return addresses:

```
(gdb) x/30ag $rsp
0x7fffffffe0c0: 0x0                          0x7fffffffe100
0x7fffffffe0d0: 0x7fffffffe208               0x100000000
0x7fffffffe0e0: 0x0                          0x0
0x7fffffffe0f0: 0x0                          0x4011a5
0x7fffffffe100: 0x59454b2d54534554           0xa
...
```

Look for values falling in the `.text` address range (`0x401000`–`0x4011e0` in our example). The value `0x4011a5` at address `0x7fffffffe0f8` is a candidate: it points into `.text` and is probably a return address. Verify:

```
(gdb) x/3i 0x4011a0
   0x4011a0:  call   0x401140          # The call that brought us here
   0x4011a5:  test   eax,eax           # ← return address (instruction after the call)
   0x4011a7:  je     0x4011b5
```

The instruction at `0x4011a5` follows a `call` — it's indeed a return address. We've identified the caller.

## Labeling discoveries: creating symbols on the fly

As the analysis progresses, you identify functions, global variables, constants. GDB allows giving them temporary names to make the session more readable.

### Defining "convenience variables"

Variables prefixed with `$` in GDB are internal variables (called *convenience variables*) you can define freely:

```
(gdb) set $main = 0x401190
(gdb) set $check_key = 0x401140
(gdb) set $cmp_instruction = 0x40117a

(gdb) break *$main
Breakpoint 1 at 0x401190
(gdb) break *$check_key
Breakpoint 2 at 0x401140
```

These variables only exist in the current GDB session. They don't modify the binary and are not automatically saved.

### Saving annotations in a command file

To avoid redoing this work at each session, create a GDB command file:

```bash
# keygenme_stripped.gdb — manual annotations
set $main         = 0x401190  
set $check_key    = 0x401140  
set $validate     = 0x401100  
set $transform    = 0x4010c0  

# Working breakpoints
break *$main  
break *$check_key  

# Automatic displays
display/x $rax  
display/x $rdi  
display/6i $rip  

# Configuration
set disassembly-flavor intel  
set pagination off  
```

Load this file at launch:

```bash
$ gdb -q -x keygenme_stripped.gdb ./keygenme_O2_strip
```

Or from a running session:

```
(gdb) source keygenme_stripped.gdb
```

This command file becomes the analysis **notebook**. Throughout the session, you add identified addresses, useful breakpoints, automatic commands. It's the dynamic equivalent of the renamings you do in Ghidra — and like in Ghidra, it's the accumulation of these small annotations that progressively makes the analysis readable.

### Loading symbols from another file

If you have a non-stripped version of the same binary (for example a development version, or a `-dbgsym` package), you can load its symbols:

```
(gdb) symbol-file keygenme_O2_debug
```

GDB will apply the symbols from `keygenme_O2_debug` to the stripped binary being debugged, provided the addresses match (same version, same compilation options, no ASLR rebase). It's the ideal situation — but rarely available in "hostile" RE.

You can also load a separate symbol file produced by `objcopy` (seen in section 11.1):

```
(gdb) add-symbol-file /path/to/program.debug 0x401000
```

The second argument is the `.text` section's load address, necessary if the binary is PIE and the address differs between the two files.

## Working with a stripped PIE binary

PIE (*Position-Independent Executable*) binaries compiled with `-pie` add an additional difficulty: addresses change at each execution due to ASLR. Addresses seen in Ghidra (which analyzes the unloaded binary) don't match those in memory during execution.

### The problem

In Ghidra, the function of interest is at address `0x1190` (address relative to the base). But at execution, the binary is loaded at a random address:

```
(gdb) info proc mappings
  0x0000555555554000 0x0000555555555000 0x1000  0x0  r--p  keygenme_pie_strip
  0x0000555555555000 0x0000555555556000 0x1000  0x1000 r-xp keygenme_pie_strip
```

The load base is `0x555555554000`. The function that was at `0x1190` in Ghidra is now at `0x555555554000 + 0x1190 = 0x555555555190`.

### Method: calculate the rebase

The workflow is as follows:

**1. Launch the program and stop it early.** You can't know the load base before the program is in memory. Use `starti` to stop on the very first instruction (before even `_start`):

```
(gdb) starti
Starting program: ./keygenme_pie_strip  
Program stopped.  
0x00007ffff7fe4c40 in _start () from /lib64/ld-linux-x86-64.so.2
```

Note: this `_start` is the **loader's**, not the program's. The program is not yet initialized, but it's already mapped in memory.

**2. Read the load base:**

```
(gdb) info proc mappings
  0x0000555555554000 ...  r--p  keygenme_pie_strip    ← base
```

Or more directly:

```
(gdb) info files
  Entry point: 0x555555555060
```

If the entry point in the binary is `0x1060`, and `info files` shows `0x555555555060`, then the base is `0x555555555060 - 0x1060 = 0x555555554000`.

**3. Calculate effective addresses:**

```
(gdb) set $base = 0x555555554000

# Ghidra addresses + base
(gdb) break *($base + 0x1190)       # main (0x1190 in Ghidra)
(gdb) break *($base + 0x1140)       # check_key (0x1140 in Ghidra)
```

**4. Automate in the command file:**

```bash
# keygenme_pie.gdb
starti

# Read the base from the entry point
# (assumes the binary's entry point offset is 0x1060)
python  
import gdb  
entry = int(gdb.execute("info files", to_string=True).split("Entry point: ")[1].split("\n")[0], 16)  
base = entry - 0x1060  
gdb.execute(f"set $base = {base}")  
print(f"Base address: {hex(base)}")  
end  

break *($base + 0x1190)  
break *($base + 0x1140)  
continue  
```

This Python script integrated into GDB automatically extracts the entry point, computes the base, and sets breakpoints with the correct offsets.

### Disabling ASLR to simplify

During analysis development, you can disable ASLR so that addresses remain stable between executions:

```
(gdb) set disable-randomization on
```

This is GDB's **default** behavior — it disables ASLR for the process it launches. If you notice addresses don't change between successive `run`s, that's why. To enable ASLR (reproduce real behavior):

```
(gdb) set disable-randomization off
```

You can also disable ASLR at the system level (for all processes):

```bash
$ echo 0 | sudo tee /proc/sys/kernel/randomize_va_space
# 0 = disabled, 1 = partial, 2 = full (default)
```

> ⚠️ **Only do this in a work VM.** Disabling ASLR on a production system is a bad security practice.

## Advanced strategies on stripped binaries

### Breakpoints on PLT functions

Even stripped, a dynamically linked binary exposes its library calls via the PLT. It's a rich observation surface. You can list PLT entries:

```
(gdb) info functions @plt
0x0000000000401030  strcmp@plt
0x0000000000401040  printf@plt
0x0000000000401050  fgets@plt
```

The `info functions` command works here because dynamic symbols (`.dynsym`) are not removed by `strip`. You get the binary's "functional profile": it uses `strcmp`, `printf`, `fgets` — already a strong indication of its behavior.

### Breakpoints on `.rodata` strings

If you've identified an interesting string in `.rodata` (for example `"Correct!"` at address `0x402011`), you can find the instructions referencing it by searching for cross-references in the disassembly. Without Ghidra, you can use an indirect approach in GDB: setting a hardware read **watchpoint** on the string's address isn't directly possible (x86 watchpoints monitor writes by default), but you can search in the disassembly with `find`:

```
(gdb) find /b 0x401000, 0x4011e0, 0x11, 0x20, 0x40
```

This searches for the byte sequence `11 20 40` (which would be the partial encoding of an offset to `0x402011` in a `lea` instruction) in the `.text` section. It's rudimentary — Ghidra is better for cross-references — but it's doable entirely from GDB.

### Identifying functions by their behavior

When static analysis isn't available or isn't sufficient, you can identify functions by **dynamic observation**:

```
(gdb) break *0x401140
(gdb) run
Breakpoint 1, 0x0000000000401140 in ?? ()

(gdb) x/s $rdi
0x7fffffffe100: "TEST-KEY\n"

(gdb) finish
Run till exit from #0  0x0000000000401140 in ?? ()
0x00000000004011a5 in ?? ()

(gdb) print $rax
$1 = 0
```

The function at `0x401140` receives the user-entered string in `rdi` and returns `0`. If you enter the right key, it will probably return `1`. It's the verification function. You can name it:

```
(gdb) set $check_key = 0x401140
```

This "black box" approach — observing a function's inputs and outputs to deduce its role — is sometimes faster than laboriously reading its disassembly.

### Combining GDB and `strace` / `ltrace`

On a stripped binary, running `ltrace` in parallel quickly gives the library-call profile:

```bash
$ ltrace ./keygenme_O2_strip
printf("Enter your key: ")                    = 16  
fgets("TEST-KEY\n", 64, 0x7f...)              = 0x7ffe...  
strcmp("TEST-KEY\n", "VALID-KEY-2025\n")       = -1  
printf("Wrong key!\n")                        = 11  
```

You immediately see that `strcmp` compares user input with `"VALID-KEY-2025\n"`. It's sometimes sufficient to solve a simple crackme without even opening GDB. For more complex cases, `ltrace` gives the first clues that you'll then deepen in GDB.

## Synthesis: adapting your method to the stripping level

| Situation | What's available | Strategy |  
|---|---|---|  
| **DWARF symbols** (`-g`) | Names, types, lines, local variables | Classic debugging: `break <name>`, `print <var>`, `next` |  
| **Symbol table only** (no `-g`) | Function names, no types or variables | `break <name>` works, but `info locals` is empty. Complement with `x` and `print $reg` |  
| **Stripped, non-PIE** | Fixed addresses, PLT, `.rodata` | Breakpoints by address, cross-reference with Ghidra, `.gdb` file |  
| **Stripped + PIE** | Variable addresses, PLT, `.rodata` | `starti` + base calculation, `$base + offset`, or disable ASLR |  
| **Stripped + PIE + statically linked** | No dynamic symbols, no PLT | The hardest case. Heavy static analysis mandatory (Ghidra), FLIRT signatures |

Whatever the stripping level, the machine code remains identical. It's always there, it executes the same way, and GDB can always observe it instruction by instruction. Symbols are a comfort, not a necessity. The essential is to have a **prior analysis strategy** — spotting functions and addresses of interest statically — and a **GDB command file** that capitalizes discoveries throughout the session.

---

> **Takeaway:** Working on a stripped binary in GDB means accepting to reason in addresses rather than names, and compensating for the absence of symbols through rigorous preparation in static analysis. The three pillars are: locating `main` via `__libc_start_main`, setting breakpoints by address (`*0x...`), and maintaining a `.gdb` command file that serves as persistent memory. The stripped binary hides nothing — it simply forces you to work more methodically.
