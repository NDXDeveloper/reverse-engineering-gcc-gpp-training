# Chapter 17 — Reverse Engineering C++ with GCC

> **Part IV — Advanced RE Techniques**

---

## Why C++ is a major challenge in Reverse Engineering

C, despite its proximity to the machine, produces relatively predictable assembly: functions have readable names, data structures translate into memory accesses at fixed offsets, and the control flow remains faithful to the source code. C++ radically changes the game. The compiler generates a considerable amount of code and metadata invisible to the developer — code that the reverse engineer must nevertheless understand to reconstruct the original logic.

When GCC compiles C++, it doesn't just translate instructions: it orchestrates a complete object model. Each polymorphic class is assigned a **vtable** (table of virtual function pointers) and each instance embeds a hidden **vptr** that points to this table. Multiple inheritance complicates the memory layout of objects with pointer adjustments (`thunks`). Function names are **mangled** according to the Itanium ABI rules to encode namespaces, parameter types, and qualifiers — transforming a simple `MyClass::process(int)` into a cryptic string like `_ZN7MyClass7processEi`. The exception mechanism relies on dedicated sections (`.eh_frame`, `.gcc_except_table`) and runtime functions (`__cxa_throw`, `__cxa_begin_catch`) that generate an invisible control flow in classic disassembly.

And that's just the tip of the iceberg. Templates cause a **combinatorial explosion of symbols**: each instantiation with a different type generates a complete code copy. STL containers (`std::vector`, `std::string`, `std::map`…) have specific internal memory layouts that must be known to correctly interpret memory accesses. Lambdas are compiled into anonymous classes with an `operator()`. Smart pointers add layers of indirection and reference counting. C++20 coroutines transform an apparently linear function into a state machine.

All of this means that an optimized, stripped C++ binary can be **several orders of magnitude harder to analyze** than an equivalent C binary in terms of functionality.

## What this chapter covers

This chapter is the densest in the training. It systematically breaks down each C++ mechanism as GCC compiles it, showing each time:

- **What the developer writes** (the C++ source code).  
- **What GCC produces** (the x86-64 assembly and generated data structures).  
- **How to recognize it** in a disassembler (Ghidra, objdump, radare2) without access to the source code.  
- **How to reconstruct it** to recover an exploitable high-level representation.

Here are the topics covered in order:

1. **Name mangling** — The Itanium ABI rules used by GCC to encode C++ identifiers, and tools to decode them (`c++filt`, automatic demangling in Ghidra).

2. **Object model: vtables and vptr** — How GCC implements polymorphism, the memory layout of an object with single then multiple inheritance, and how to identify virtual calls in disassembly.

3. **RTTI and `dynamic_cast`** — The runtime type structures generated by GCC, their location in the binary, and how they allow reconstructing the class hierarchy.

4. **Exception handling** — The zero-cost exception handling mechanism, the `.eh_frame` and `.gcc_except_table` sections, and the `__cxa_*` runtime functions.

5. **STL internals** — The memory layout of the most common containers (`vector`, `string`, `map`, `unordered_map`) as GCC/libstdc++ implements them.

6. **Templates** — How to recognize multiple instantiations, the impact on binary size and the symbol table.

7. **Lambdas and closures** — The transformation into anonymous classes, variable capture by value and by reference, and their assembly representation.

8. **Smart pointers** — The differences between `unique_ptr` (zero-cost abstraction) and `shared_ptr` (atomic reference counting) in generated code.

9. **C++20 coroutines** — The state machine pattern generated by GCC, the coroutine frame, and suspension points.

## Prerequisites for this chapter

This chapter relies heavily on knowledge from previous chapters. Before starting, make sure you're comfortable with:

- **Chapter 3** — x86-64 assembly: registers, System V AMD64 calling conventions, reading an assembly listing. You must be able to read a function prologue/epilogue and follow parameter passing via `rdi`, `rsi`, `rdx`… without hesitation.

- **Chapter 7** — Disassembly with objdump: Intel syntax, optimization level comparison, identifying `main()`. We'll extensively use `objdump -d -M intel` to examine generated code.

- **Chapter 8** — Ghidra: CodeBrowser navigation, decompiler, function renaming, structure reconstruction. Ghidra will be our primary tool for visualizing vtables and C++ data structures.

- **Chapter 16** — Compiler optimizations: inlining, loop unrolling, tail call optimization. Understanding GCC optimizations is essential because they profoundly transform C++ code (inlining removes virtual calls, templates are aggressively optimized, etc.).

Intermediate knowledge of C++ from the developer side is also necessary. You don't need to master the standard's subtleties, but you must understand the concepts of class, inheritance, polymorphism, templates, exceptions, and STL at the level of a developer who uses them daily.

## Training binary

The main binary for this chapter is `ch17-oop`, located in `binaries/ch17-oop/`. It's a C++ object-oriented application compiled with GCC, available in several variants:

| Variant | Compilation flags | Usage |  
|---|---|---|  
| `ch17-oop_O0` | `-O0 -g` | Learning — non-optimized code, full symbols |  
| `ch17-oop_O0_strip` | `-O0 -s` | Intermediate — non-optimized but without symbols |  
| `ch17-oop_O2` | `-O2 -g` | Realistic — optimized with symbols for verification |  
| `ch17-oop_O2_strip` | `-O2 -s` | Real conditions — optimized and stripped |  
| `ch17-oop_O3` | `-O3 -g` | Aggressive optimization — with symbols |  
| `ch17-oop_O3_strip` | `-O3 -s` | Maximum challenge — aggressively optimized and stripped |

The recommended progression is to start each section with the `ch17-oop_O0` variant to understand the mechanism, then move to `ch17-oop_O2_strip` to learn to recognize it under realistic conditions.

This binary intentionally contains:

- A class hierarchy with single and multiple inheritance.  
- Virtual and pure virtual methods (abstract classes).  
- Enabled RTTI (no `-fno-rtti`).  
- Exceptions with `try`/`catch` and custom exception types.  
- Various STL containers (`std::vector`, `std::string`, `std::map`).  
- Templates instantiated with multiple types.  
- Lambdas with different capture modes.  
- Smart pointers (`std::unique_ptr`, `std::shared_ptr`).

## Recommended methodology

When facing an unknown C++ binary, the recommended approach breaks down into phases:

**Phase 1 — Demangling and mapping.** Always start by recovering as much information as possible from symbols (if they exist) or from RTTI. Demangle names, list identifiable classes, and build a first draft of the hierarchy.

**Phase 2 — Vtable reconstruction.** Locate vtables in `.rodata`, identify virtual methods and their order. Each vtable gives you the complete list of a class's virtual methods, including inherited ones.

**Phase 3 — Object memory layout.** Determine the size and layout of each class by analyzing constructors (they initialize the vptr and members) and recurring memory accesses via offsets from `this` (first parameter, in `rdi`).

**Phase 4 — Control flow and business logic.** Once the class structure is understood, focus on the logic: what the methods do, how objects interact, what the overall execution flow is.

This chapter follows this progression by treating each mechanism individually before combining them in the final checkpoint.
