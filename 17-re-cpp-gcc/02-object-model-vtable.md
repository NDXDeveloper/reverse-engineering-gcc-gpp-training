# 17.2 — C++ object model: vtable, vptr, single and multiple inheritance

> **Chapter 17 — Reverse Engineering C++ with GCC**  
> **Part IV — Advanced RE Techniques**

---

## The problem virtual dispatch solves

Consider a pointer `Shape* s` that can designate a `Circle`, a `Rectangle`, or a `Triangle`. When the code calls `s->area()`, the compiler can't know at compile time which implementation of `area()` to execute — it depends on the object's actual type at runtime. C++ solves this problem through **virtual dispatch**: an indirection mechanism through a table of function pointers.

For the reverse engineer, understanding this mechanism is fundamental. A virtual call in disassembly doesn't look like a `call` to a fixed address — it's a `call` to an address read from a table, itself pointed to by a hidden field in the object. Without recognizing this pattern, you can't know which function is actually being called.

## The vtable: structure in memory

For each class containing at least one virtual method (including the destructor), GCC generates a **vtable** (*virtual method table*) in the binary's `.rodata` section. The vtable is an array of function pointers, one per virtual method, in the order of their declaration in the class hierarchy.

The complete vtable generated by GCC according to the Itanium ABI actually contains more than just function pointers. Its structure is as follows:

```
Decreasing address (negative offsets from the "visible" start)
┌─────────────────────────────────────────────┐
│  offset-to-top (ptrdiff_t)                  │  offset -16
├─────────────────────────────────────────────┤
│  pointer to typeinfo (RTTI)                 │  offset -8
├═════════════════════════════════════════════╡  ← address pointed to by the vptr
│  pointer to virtual method #0               │  offset 0
├─────────────────────────────────────────────┤
│  pointer to virtual method #1               │  offset +8
├─────────────────────────────────────────────┤
│  pointer to virtual method #2               │  offset +16
├─────────────────────────────────────────────┤
│  ...                                        │
└─────────────────────────────────────────────┘
```

The two fields at negative offsets (before the address the vptr points to) are:

- **`offset-to-top`**: the distance in bytes between the current sub-object and the beginning of the complete object. For single inheritance, this is always 0. For multiple inheritance, secondary sub-objects have a non-zero negative offset. We'll revisit this.

- **Pointer to typeinfo**: points to the class's `_ZTI...` structure (used by `dynamic_cast` and `typeid`). If the binary was compiled with `-fno-rtti`, this field is `0`.

The virtual function pointers start at offset 0 from the address the vptr stores. They're ordered according to the declaration order of virtual methods in the hierarchy, starting from the topmost base class.

## The vptr: hidden field in each object

Each instance of a polymorphic class contains a hidden pointer called **vptr** (*virtual table pointer*), which the compiler places **at the very beginning of the object** (offset 0). This vptr points to the vtable corresponding to the object's actual type.

The vptr appears nowhere in the C++ source code. It's the compiler that adds it, and the constructor that initializes it. Here's what this looks like for the `Shape` / `Circle` hierarchy:

```
Circle object in memory:
┌──────────────────────────────────────┐
│  vptr → Circle's vtable              │  offset 0    (8 bytes, added by GCC)
├──────────────────────────────────────┤
│  name_ (std::string)                 │  offset 8    (32 bytes with libstdc++)
├──────────────────────────────────────┤
│  x_ (double)                         │  offset 40
├──────────────────────────────────────┤
│  y_ (double)                         │  offset 48
├──────────────────────────────────────┤
│  radius_ (double)                    │  offset 56
└──────────────────────────────────────┘
                                         total: 64 bytes
```

Members inherited from `Shape` (`name_`, `x_`, `y_`) come first (after the vptr), followed by `Circle`'s own members (`radius_`). The vptr is always at offset 0.

> 💡 **In RE:** if you see a memory access at offset 0 of an object that loads a pointer, then an indirect access via that pointer with a constant offset, it's very likely a virtual call through the vptr.

## Single inheritance: vtable and virtual call

### The abstract base class vtable

Let's start with `Shape`, which declares three virtual methods (plus the destructor):

```cpp
class Shape {
    virtual ~Shape() = default;        // virtual #0 (D1) and #1 (D0)
    virtual double area() const = 0;   // virtual #2 (pure)
    virtual double perimeter() const = 0; // virtual #3 (pure)
    virtual std::string describe() const; // virtual #4
};
```

`Shape`'s vtable looks like this:

```
vtable for Shape (_ZTV5Shape):
  [offset-to-top]   = 0
  [typeinfo]         = &_ZTI5Shape
  [slot 0]           → Shape::~Shape() [D1]          (complete destructor)
  [slot 1]           → Shape::~Shape() [D0]          (deleting destructor)
  [slot 2]           → __cxa_pure_virtual             (area = 0 → pure)
  [slot 3]           → __cxa_pure_virtual             (perimeter = 0 → pure)
  [slot 4]           → Shape::describe() const
```

Pure virtual methods point to `__cxa_pure_virtual`, a C++ runtime function that terminates the program with an error if called. It should never be called in practice since `Shape` cannot be directly instantiated.

> 💡 **In RE:** if you see `__cxa_pure_virtual` in a vtable, you know the class is abstract and the corresponding slot is a pure virtual method. This is a very reliable marker.

### The derived class vtable

`Circle` inherits from `Shape` and implements the pure methods, plus an override of `describe()`:

```
vtable for Circle (_ZTV6Circle):
  [offset-to-top]   = 0
  [typeinfo]         = &_ZTI6Circle
  [slot 0]           → Circle::~Circle() [D1]
  [slot 1]           → Circle::~Circle() [D0]
  [slot 2]           → Circle::area() const           ← override
  [slot 3]           → Circle::perimeter() const      ← override
  [slot 4]           → Circle::describe() const       ← override
```

The slots are at the **same indices** as in `Shape`'s vtable. This is what makes virtual dispatch work: regardless of the actual type of the object behind a `Shape*`, slot 2 is always `area()`, slot 3 is always `perimeter()`, etc. Only the pointers change.

For `Triangle`, which doesn't override `describe()`:

```
vtable for Triangle (_ZTV8Triangle):
  [offset-to-top]   = 0
  [typeinfo]         = &_ZTI8Triangle
  [slot 0]           → Triangle::~Triangle() [D1]
  [slot 1]           → Triangle::~Triangle() [D0]
  [slot 2]           → Triangle::area() const
  [slot 3]           → Triangle::perimeter() const
  [slot 4]           → Shape::describe() const        ← inherited, not overridden
```

Slot 4 points to `Shape::describe()` because `Triangle` doesn't provide its own implementation. In RE, when a derived class vtable slot points to a parent class method, it indicates the absence of an override.

### The constructor initializes the vptr

The constructor is where the vptr is written. Here's a simplified version of what GCC generates for `Circle::Circle()` in `-O0`:

```nasm
; Circle::Circle(double x, double y, double r)
; rdi = this, xmm0 = x, xmm1 = y, xmm2 = r
Circle::Circle:
    push   rbp
    mov    rbp, rsp
    sub    rsp, 0x30
    mov    QWORD PTR [rbp-0x8], rdi      ; save this
    movsd  QWORD PTR [rbp-0x10], xmm0    ; save x
    movsd  QWORD PTR [rbp-0x18], xmm1    ; save y
    movsd  QWORD PTR [rbp-0x20], xmm2    ; save r

    ; ---- Call to parent constructor Shape::Shape() ----
    mov    rdi, QWORD PTR [rbp-0x8]      ; this (same address)
    lea    rsi, [rip+.LC_Circle]         ; "Circle"
    movsd  xmm0, QWORD PTR [rbp-0x10]   ; x
    movsd  xmm1, QWORD PTR [rbp-0x18]   ; y
    call   Shape::Shape(std::string const&, double, double)

    ; ---- Write Circle's vptr ----
    mov    rax, QWORD PTR [rbp-0x8]      ; this
    lea    rdx, [rip+_ZTV6Circle+16]     ; &vtable[0] (skip offset-to-top and typeinfo)
    mov    QWORD PTR [rax], rdx          ; this->vptr = &Circle::vtable

    ; ---- Initialize radius_ ----
    mov    rax, QWORD PTR [rbp-0x8]      ; this
    movsd  xmm0, QWORD PTR [rbp-0x20]   ; r
    movsd  QWORD PTR [rax+0x38], xmm0   ; this->radius_ = r (offset 56)

    ; ---- Check r > 0 (exception if not) ----
    ...
```

Key points to observe:

1. **The parent constructor is called first.** `Shape::Shape()` initializes the vptr to `&_ZTV5Shape+16` and `Shape`'s members. This vptr will be immediately overwritten.

2. **The vptr is overwritten by the derived class.** The instruction `mov QWORD PTR [rax], rdx` writes `&_ZTV6Circle+16` at offset 0 of `this`. The `+16` skips the two negative-offset fields (offset-to-top and typeinfo pointer, each 8 bytes).

3. **Each hierarchy level writes its own vptr.** During `Circle` construction, the vptr first points to `Shape`'s vtable (after calling `Shape::Shape`), then to `Circle`'s. This is why calling a virtual method in a base constructor calls the base class version, not the derived class version — a known C++ behavior that's very visible in RE.

> 💡 **RE pattern for identifying constructors:** look for functions that: (1) receive a pointer in `rdi` (this), (2) call another constructor with the same `rdi`, (3) write a vtable address at offset 0 of `this`, (4) initialize other fields at fixed offsets. This is the signature of a derived class constructor.

### The virtual call in assembly

The call `s->area()` where `s` is a `Shape*` compiles as:

```nasm
; rdi contains s (Shape*)
mov    rax, QWORD PTR [rdi]         ; rax = s->vptr (read vptr at offset 0)  
call   QWORD PTR [rax+0x10]         ; indirect call: vtable[slot 2] = area()  
                                     ; offset 0x10 = 16 = slot 2 × 8 bytes
```

This is the **fundamental virtual dispatch pattern**. Two instructions:

1. Load the vptr from the object's beginning: `mov rax, [rdi]`  
2. Call indirectly through the vtable at a fixed offset: `call [rax+offset]`

The offset in the vtable identifies the method. With `Shape`'s vtable:

| Offset | Slot | Method |  
|--------|------|--------|  
| `+0x00` | 0 | `~Shape()` (D1, complete destructor) |  
| `+0x08` | 1 | `~Shape()` (D0, deleting destructor) |  
| `+0x10` | 2 | `area() const` |  
| `+0x18` | 3 | `perimeter() const` |  
| `+0x20` | 4 | `describe() const` |

Seeing `call [rax+0x10]`, we know it's a call to the third virtual method (slot 2). By consulting the vtable, we identify `area()`.

> ⚠️ **Watch out for optimizations.** In `-O2`, GCC can devirtualize a call when it knows the actual type at compile time. For example, if the variable is a local `Circle` (not a polymorphic pointer), GCC will replace the indirect call with a direct `call` to `Circle::area()`. The virtual dispatch pattern is then lost in the disassembly, but you gain certainty about the called function.

### Vtable reconstruction method in Ghidra

To reconstruct a vtable in Ghidra:

1. **Locate the vtable.** Search for `_ZTV` symbols in the Symbol Tree, or navigate in `.rodata` and look for sequences of pointers into `.text`. If the binary is stripped, search for `_ZTS` strings (typeinfo names) and trace back to adjacent `_ZTI` structures, then to the vtables that reference them.

2. **Count the slots.** From the address pointed to by the vptr (after offset-to-top and typeinfo), each QWORD is a pointer to a virtual method. Count slots until the next vtable (recognizable by a new offset-to-top, usually 0 or a negative value).

3. **Identify each slot.** Double-click each pointer to navigate to the function. Analyze it to determine what it does. The first two slots are almost always the destructors (D1 and D0).

4. **Create a structure in Ghidra.** Menu Data Type Manager → New Structure. Define the vptr as the first field (type `pointer`), followed by members deduced from analyzing constructors and memory accesses.

## Multiple inheritance: the complex case

Multiple inheritance is where the C++ object model becomes significantly more complex. When a class inherits from multiple polymorphic bases, the object contains **multiple vptrs** and the vtable contains **multiple sub-tables**.

Let's take our `Canvas` class:

```cpp
class Canvas : public Drawable, public Serializable {
    std::string title_;
    std::vector<std::shared_ptr<Shape>> shapes_;
    int z_order_;
    ...
};
```

`Canvas` inherits from `Drawable` (which has virtual methods `draw()` and `zOrder()`) and `Serializable` (which has virtual methods `serialize()` and `deserialize()`).

### Memory layout with multiple inheritance

GCC places the base class sub-objects in the order of declaration in the inheritance list:

```
Canvas object in memory:
┌──────────────────────────────────────────┐
│                                          │
│  Drawable sub-object:                    │
│  ┌──────────────────────────────────┐    │
│  │  vptr_1 → vtable Canvas (part 1) │    │  offset 0
│  └──────────────────────────────────┘    │
│                                          │
│  Serializable sub-object:                │
│  ┌──────────────────────────────────┐    │
│  │  vptr_2 → vtable Canvas (part 2) │    │  offset 8
│  └──────────────────────────────────┘    │
│                                          │
│  Canvas's own members:                   │
│  ┌──────────────────────────────────┐    │
│  │  title_ (std::string)            │    │  offset 16
│  │  shapes_ (std::vector<...>)      │    │  offset 48
│  │  z_order_ (int)                  │    │  offset 72
│  └──────────────────────────────────┘    │
│                                          │
└──────────────────────────────────────────┘
```

The object contains **two vptrs**:

- **`vptr_1`** (offset 0): points to the part of `Canvas`'s vtable corresponding to the `Drawable` interface. This is also the vtable used when manipulating the object via a `Canvas*` or `Drawable*` pointer.

- **`vptr_2`** (offset 8): points to the part of `Canvas`'s vtable corresponding to the `Serializable` interface. This is the vtable used when manipulating the object via a `Serializable*` pointer.

### The composite vtable

GCC generates a single vtable for `Canvas` containing both sub-tables consecutively:

```
vtable for Canvas (_ZTV6Canvas):

  ═══ Part 1: Drawable interface (and Canvas itself) ═══
  [offset-to-top]   = 0
  [typeinfo]         = &_ZTI6Canvas
  [slot 0]           → Canvas::~Canvas() [D1]
  [slot 1]           → Canvas::~Canvas() [D0]
  [slot 2]           → Canvas::draw() const
  [slot 3]           → Canvas::zOrder() const
  [slot 4]           → Canvas::serialize() const      ← Canvas's own
  [slot 5]           → Canvas::deserialize(string const&) ← Canvas's own

  ═══ Part 2: Serializable interface ═══
  [offset-to-top]   = -8                               ← distance to object start
  [typeinfo]         = &_ZTI6Canvas                     ← same typeinfo
  [slot 0]           → thunk to Canvas::~Canvas() [D1]
  [slot 1]           → thunk to Canvas::~Canvas() [D0]
  [slot 2]           → thunk to Canvas::serialize() const
  [slot 3]           → thunk to Canvas::deserialize(string const&)
```

Several crucial points for RE:

1. **The first part** of the vtable serves both `Canvas*` and `Drawable*`. It includes methods inherited from `Drawable` at the same indices as in `Drawable`'s vtable, plus `Canvas`'s own methods and `Serializable`'s (for direct access from a `Canvas*`).

2. **The second part** contains `Serializable` interface methods, but they point to **thunks**, not directly to the methods.

3. **The offset-to-top** of part 2 is `-8`, meaning: "to find the beginning of the complete `Canvas` object from the `Serializable` sub-object, go back 8 bytes."

### Thunks: pointer adjustment

A **thunk** is a small code fragment generated by the compiler that adjusts the `this` pointer before calling the actual method. It's necessary because when calling a `Canvas` method via a `Serializable*` pointer, the `this` pointer points to the `Serializable` sub-object (offset 8 in the complete object), not to the beginning of the `Canvas` object.

Here's what a thunk looks like in assembly:

```nasm
; Non-virtual thunk for Canvas::serialize(), adjustment -8
_ZThn8_NK6Canvas9serializeEv:
    sub    rdi, 8          ; adjust this: Serializable* → Canvas*
    jmp    _ZNK6Canvas9serializeEv   ; jump to the real method
```

The thunk does exactly two things:
1. Subtracts the sub-object offset from `rdi` (this) to get the complete object's address.  
2. Jumps (not `call`, but `jmp`) to the actual implementation.

> 💡 **RE pattern for identifying thunks:** a 2-3 instruction function that adjusts `rdi` by adding or subtracting a constant then does a `jmp` to another function. The mangled symbol starts with `_ZThn` (non-virtual thunk with adjustment `n`) or `_ZTv` (virtual thunk). In Ghidra, thunks are sometimes automatically recognized and displayed as "thunk function."

### Pointer conversion at runtime

When the source code performs an implicit cast from `Canvas*` to `Serializable*`, the compiler generates a pointer adjustment:

```cpp
Canvas canvas("test", 1);  
Serializable* ser = &canvas;   // implicit conversion  
```

In assembly:

```nasm
; rax = address of canvas (Canvas*)
lea    rdx, [rax+8]      ; rdx = Serializable* (offset by 8 bytes)
```

The `Serializable*` pointer doesn't point to the same location as the `Canvas*` pointer. It points 8 bytes further, toward the `Serializable` sub-object and its vptr_2.

Conversely, when converting back from `Serializable*` to `Canvas*` (via `dynamic_cast` or `static_cast`), the compiler subtracts 8:

```nasm
; rdi = Serializable*
sub    rdi, 8             ; rdi = Canvas*
```

> ⚠️ **Common RE trap:** when you see an `add` or `sub` on an object pointer followed by a call or vptr access, it's a multiple inheritance adjustment. Don't confuse it with a member access — the clue is that the result is used as `this` (in `rdi`) for a call, or that a vptr (QWORD at offset 0) is loaded from the adjusted address.

### Distinguishing single and multiple inheritance in RE

Here are the clues that signal multiple inheritance in a binary:

| Clue | What it means |  
|------|---------------|  
| An object contains **two or more vptrs** (two vtable pointers at different offsets) | The object inherits from multiple polymorphic classes |  
| A vtable contains a **negative offset-to-top** (non-zero value) | It's the sub-table of a secondary sub-object |  
| **Thunks** (`sub rdi, N; jmp ...`) appear in `.text` | `this` adjustment for secondary base methods |  
| A **pointer adjustment** (constant `add`/`sub`) appears during a type cast | Conversion between primary and secondary base |  
| The constructor writes **multiple vptrs** at different offsets of `this` | Initialization of multiple polymorphic sub-objects |  
| The **typeinfo** references a `__vmi_class_type_info` instead of `__si_class_type_info` | RTTI encodes multiple inheritance (see Section 17.3) |

## Reconstructing a class hierarchy from the vtable

In RE, the procedure for reconstructing the hierarchy from vtables is:

**Step 1 — List all vtables.** Use `nm -C binary | grep 'vtable for'` (with symbols) or search for typeinfo name strings in `.rodata` (without symbols). Each vtable = one polymorphic class.

**Step 2 — Analyze each vtable's slots.** Note the function pointers in each slot. Two vtables sharing pointers in the same slots have an inheritance relationship.

**Step 3 — Compare vtables pairwise.** If `B`'s vtable has the same first slots as `A`'s vtable (same pointers or pointers to overrides), then `B` likely inherits from `A`. Additional slots at the end of `B`'s vtable are virtual methods added by `B`.

**Step 4 — Identify abstract classes.** Vtables containing pointers to `__cxa_pure_virtual` correspond to abstract classes.

**Step 5 — Look for thunks and composite vtables.** If a vtable contains a second section with a negative offset-to-top, it's multiple inheritance. Associated thunks indicate which secondary base methods are implemented.

**Concrete example with our binary:**

```bash
$ nm -C oop_O0 | grep 'vtable for' | awk '{print $NF}'
vtable for Shape  
vtable for Circle  
vtable for Rectangle  
vtable for Triangle  
vtable for Drawable  
vtable for Serializable  
vtable for Canvas  
```

Examining the vtables:
- `Circle`, `Rectangle`, `Triangle` have the same slot structure as `Shape` → they inherit from `Shape`.  
- `Circle` and `Rectangle`'s vtables have different pointers at slots 2-4 but no additional slots → they override `Shape`'s methods without adding new ones.  
- `Canvas`'s vtable has a composite structure with two parts → multiple inheritance from `Drawable` and `Serializable`.  
- `Shape`'s vtable contains `__cxa_pure_virtual` at slots 2 and 3 → `Shape` is abstract.

## Visualizing a vtable in Ghidra

In Ghidra, a vtable in `.rodata` appears as a sequence of 8-byte values. By default, Ghidra doesn't necessarily recognize them as pointers. Here's how to make them readable:

1. **Navigate to the vtable address.** If symbols are present, search for `_ZTV` in the Symbol Tree. Otherwise, search in `.rodata` for sequences of addresses pointing into `.text`.

2. **Type each entry as pointer.** Select the first 8 bytes after offset-to-top and typeinfo, right-click → Data → pointer. Repeat for each slot. Ghidra will automatically resolve addresses to function names.

3. **Create a structure type.** In the Data Type Manager, create a structure named `vtable_ClassName` with pointer-to-function fields. Apply this type at the vtable address.

4. **Verify with constructors.** Each constructor writes the address `&vtable + 16` into the vptr. Double-click the vtable's cross-references to find constructors and confirm which class uses which vtable.

## Impact of optimizations on vtables

The optimization level significantly affects how vtables and virtual dispatch appear in the binary:

**In `-O0`:** each virtual call faithfully produces the `mov rax, [rdi]; call [rax+offset]` pattern. Constructors explicitly write the vptr. Everything is readable and predictable.

**In `-O2` / `-O3`:** GCC applies several transformations:

- **Devirtualization.** When the compiler can prove the object's actual type (local variable, `new` result, type propagation), it replaces the indirect call with a direct call. The virtual call disappears from the disassembly.

- **Speculative devirtualization.** GCC can emit a comparison of the vptr against the expected vtable, and if it succeeds, call the method directly (fast path) instead of going through the indirection (slow path). This produces a pattern like:

  ```nasm
  mov    rax, [rdi]                    ; load vptr
  lea    rdx, [rip+_ZTV6Circle+16]     ; expected vtable
  cmp    rax, rdx                      ; is it a Circle?
  jne    .slow_path                    ; if not, normal dispatch
  call   Circle::area                  ; fast path: direct call
  jmp    .done
  .slow_path:
  call   [rax+0x10]                    ; slow path: virtual dispatch
  .done:
  ```

- **Inlining of small virtual methods.** If a virtual method is trivial (accessor, constant return), GCC can inline it after devirtualization. The method then completely disappears as a separate call.

- **Elimination of unused vtables.** With `-ffunction-sections` + `--gc-sections` (or LTO), vtables of never-instantiated classes can be eliminated by the linker.

> 💡 **Practical tip:** always start analysis with the `-O0` variant to understand the class hierarchy and the role of each virtual method. Then switch to `-O2` to see which optimizations GCC applied and understand why certain virtual calls disappeared.

## Summary of patterns to recognize

| Assembly pattern | Meaning |  
|------------------|---------|  
| `mov rax, [rdi]` then `call [rax+N]` | Virtual call, slot N/8 in the vtable |  
| Writing a `.rodata` address to `[rdi+0]` in a constructor | vptr initialization |  
| Writing **two** `.rodata` addresses to `[rdi+0]` and `[rdi+K]` | Constructor with multiple inheritance (two vptrs) |  
| `sub rdi, N; jmp func` (2-3 instructions) | Adjustment thunk for multiple inheritance |  
| `lea rdx, [rdi+N]` followed by a store or call with `rdx` as `this` | Pointer conversion between bases (multiple inheritance) |  
| Sequence of pointers to `.text` in `.rodata`, preceded by a 0 and a `_ZTI` pointer | Vtable (offset-to-top = 0, typeinfo, then slots) |  
| `__cxa_pure_virtual` in a vtable slot | Pure virtual method → abstract class |  
| Direct call where indirect dispatch was expected | Devirtualization by the compiler (`-O2`+) |
