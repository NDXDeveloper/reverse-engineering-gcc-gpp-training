# 17.7 — Lambda, closures, and captures in assembly

> **Chapter 17 — Reverse Engineering C++ with GCC**  
> **Part IV — Advanced RE Techniques**

## What the compiler does with a lambda

A C++ lambda is not a function in the classic sense. The standard defines a lambda as an object of a **unique, anonymous closure type**, generated by the compiler. This closure type is a class (in the `struct` sense) that:

1. Has a **`const operator()`** (or non-const if the lambda is `mutable`) whose body is the lambda's code.  
2. Contains as **members** all captured variables (by value or by reference).  
3. Has implicit constructors and destructors to initialize and clean up captures.

In other words, GCC transforms each lambda expression into an anonymous class. The lambda's code becomes a method of this class. Captured variables become fields. The closure object is instantiated (often on the stack) at the point where the lambda is defined.

For the reverse engineer, this means each lambda manifests in the binary as:

- A **function** in `.text` (the `operator()`) with a mangled name containing an anonymous identifier.  
- An **object** on the stack or in a register, whose size depends on captures.  
- Initialization code at the definition point, copying captured values (or pointers/references) into the closure object.

## Anatomy of a capture-less lambda

The simplest lambda is one without captures:

```cpp
auto printSeparator = []() {
    std::cout << "------------------------" << std::endl;
};
printSeparator();
```

### What GCC generates

GCC creates an empty anonymous class (no members) with an `operator()`:

```cpp
// Conceptual equivalent generated by GCC:
struct __lambda_printSeparator {
    void operator()() const {
        std::cout << "------------------------" << std::endl;
    }
};
```

Since the class has no members, its `sizeof` is 1 (minimum C++ object size). The closure object takes up virtually no space and the compiler often optimizes it away entirely.

### The mangled symbol

The `operator()` name of a capture-less lambda in GCC follows a pattern like:

```
_ZZ20demonstrateLambdasRKSt6vectorISt10shared_ptrI5ShapeESaIS2_EEENKUlvE_clEv
```

Demangled:

```
demonstrateLambdas(std::vector<std::shared_ptr<Shape>> const&)::{lambda()#1}::operator()() const
```

The name structure is:

- `_ZZ`: symbol local to a function (the enclosing function).  
- `demonstrateLambdas(...)`: the enclosing function.  
- `{lambda()#1}`: the first no-argument lambda defined in this function.  
- `::operator()() const`: the call operator.

The `#1`, `#2`, `#3` identifies lambdas in order of appearance in the function. This number is the only way to distinguish lambdas in the symbols.

> 💡 **In RE:** when you see `{lambda` in a demangled symbol, it's a lambda body. The enclosing function name and lambda number locate you in the source code. On a stripped binary, these symbols disappear, but the code structure (small function called indirectly or inlined) remains recognizable.

### Conversion to function pointer

A capture-less lambda can be implicitly converted to a function pointer. GCC generates a conversion operator and a static thunk function for this:

```cpp
void (*fp)() = printSeparator;  // implicit conversion  
fp();                            // call via function pointer  
```

GCC generates:

```
{lambda()#1}::operator void (*)()() const    — conversion operator
{lambda()#1}::_FUN()                         — static function (the thunk)
```

The `_FUN` function is an ordinary static function (no `this`) containing the same code as the `operator()`. It's what the function pointer designates.

> 💡 **In RE:** if you see a symbol containing `_FUN` associated with a lambda, it's the function pointer conversion thunk. Its presence indicates the lambda was used as a function pointer somewhere in the code.

## Capture by value

### The mechanism

When a lambda captures a variable by value, the variable is **copied** into the closure object at definition time:

```cpp
double minArea = 10.0;  
auto isLargeShape = [minArea](const std::shared_ptr<Shape>& s) {  
    return s->area() > minArea;
};
```

### The generated closure class

```cpp
// Conceptual equivalent:
struct __lambda_isLargeShape {
    double minArea;    // capture by value → copied member

    bool operator()(const std::shared_ptr<Shape>& s) const {
        return s->area() > this->minArea;
    }
};
```

The class contains a `double minArea` field (8 bytes). The closure's `sizeof` is 8.

### In assembly

**Closure initialization (at the definition point):**

```nasm
; minArea is in xmm0 or on the stack
; The closure is built on the stack
movsd  QWORD PTR [rbp-0x30], xmm0    ; copy minArea into the closure object
; [rbp-0x30] is now the closure object (8 bytes)
```

**`operator()` call:**

```nasm
; rdi = pointer to closure (this)
; rsi = pointer to shared_ptr<Shape> (first argument)
isLargeShape::operator()() const:
    push   rbp
    mov    rbp, rsp
    mov    QWORD PTR [rbp-8], rdi     ; save this (closure)
    mov    QWORD PTR [rbp-16], rsi    ; save argument

    ; Virtual call: s->area()
    mov    rax, QWORD PTR [rsi]       ; load Shape* from shared_ptr
    mov    rcx, QWORD PTR [rax]       ; load vptr
    mov    rdi, rax                   ; this = Shape*
    call   QWORD PTR [rcx+0x10]      ; vtable[2] = area()

    ; Compare with captured minArea
    mov    rdi, QWORD PTR [rbp-8]     ; reload this (closure)
    ucomisd xmm0, QWORD PTR [rdi]    ; compare area() with this->minArea (offset 0)
    seta   al                          ; al = (area > minArea)
    ret
```

The key point is the `QWORD PTR [rdi]` access in the `operator()`: it's reading the captured `minArea` field at offset 0 of the closure. The initial `rdi` is the `this` pointer to the closure object, exactly like for any class method.

> 💡 **RE pattern:** in a lambda's `operator()`, accesses to `[rdi + offset]` are reads of captured variables. The offset identifies which capture is being read. If the `operator()` never accesses `[rdi]`, the lambda probably has no captures (or they were optimized away).

## Capture by reference

### The mechanism

When a lambda captures by reference, the closure stores a **pointer** to the original variable, not a copy:

```cpp
double totalArea = 0.0;  
int count = 0;  
auto accumulate = [&totalArea, &count](const std::shared_ptr<Shape>& s) {  
    totalArea += s->area();
    count++;
};
```

### The generated closure class

```cpp
// Conceptual equivalent:
struct __lambda_accumulate {
    double* totalArea;   // capture by reference → pointer
    int*    count;       // capture by reference → pointer

    void operator()(const std::shared_ptr<Shape>& s) const {
        *this->totalArea += s->area();
        (*this->count)++;
    }
};
```

Each reference capture adds a pointer (8 bytes) to the closure. The `sizeof` here is 16 (two pointers).

### In assembly

**Closure initialization:**

```nasm
; totalArea is at [rbp-0x40], count at [rbp-0x44]
lea    rax, [rbp-0x40]               ; address of totalArea  
mov    QWORD PTR [rbp-0x60], rax     ; closure.totalArea = &totalArea  
lea    rax, [rbp-0x44]               ; address of count  
mov    QWORD PTR [rbp-0x58], rax     ; closure.count = &count  
; The closure at [rbp-0x60] contains two pointers
```

**In the `operator()`:**

```nasm
accumulate::operator()() const:
    ; rdi = this (closure), rsi = argument (shared_ptr)
    ; ... call to s->area(), result in xmm0 ...

    ; *totalArea += area
    mov    rax, QWORD PTR [rdi]       ; rax = this->totalArea (the pointer, offset 0)
    addsd  xmm0, QWORD PTR [rax]     ; xmm0 = *totalArea + area
    movsd  QWORD PTR [rax], xmm0     ; *totalArea = result

    ; (*count)++
    mov    rax, QWORD PTR [rdi+8]    ; rax = this->count (the pointer, offset 8)
    add    DWORD PTR [rax], 1         ; (*count)++
    ret
```

The difference from capture by value is visible: the code first loads the **pointer** from the closure (`[rdi]`), then dereferences that pointer to access the variable (`[rax]`). It's a double indirection.

> 💡 **RE pattern:** in a lambda's `operator()`, a value capture = direct access `[rdi+offset]`. A reference capture = two levels of indirection: `mov rax, [rdi+offset]; op [rax]`. This distinction allows reconstructing the capture mode of each variable.

## Mixed capture (value + reference)

```cpp
std::string prefix = ">> ";  
std::vector<std::string> descriptions;  
auto describeAndCollect = [prefix, &descriptions](const std::shared_ptr<Shape>& s) {  
    std::string desc = prefix + s->describe();
    descriptions.push_back(desc);
    std::cout << desc << std::endl;
};
```

### The generated closure class

```cpp
struct __lambda_describeAndCollect {
    std::string                prefix;        // by value → copy (32 bytes)
    std::vector<std::string>*  descriptions;  // by reference → pointer (8 bytes)

    void operator()(const std::shared_ptr<Shape>& s) const {
        // ...
    }
};
// sizeof = 40 (32 + 8, potentially aligned)
```

### Closure memory layout

```
Closure object on the stack:
┌──────────────────────────────────────┐
│  prefix (std::string, 32 bytes)      │  offset 0    — capture by value
│    _M_p                              │    +0
│    _M_string_length                  │    +8
│    _M_local_buf[16]                  │    +16
├──────────────────────────────────────┤
│  descriptions (std::vector<...>*)    │  offset 32   — capture by reference
└──────────────────────────────────────┘
                                          sizeof = 40
```

> 💡 **In RE:** the closure's total size gives a clue about the number and type of captures. An 8-byte closure = one simple capture (a `double`, an `int`, or a pointer). A 40-byte closure with a `std::string` (32 bytes) + a pointer (8 bytes) = a string by value and something by reference. Cross-reference with accesses in the `operator()` to confirm.

The initialization of this closure involves **copying** the `std::string` (call to copy constructor) and storing the pointer:

```nasm
; Build the closure on the stack
; 1. Copy prefix (std::string by value) → call to copy constructor
lea    rdi, [rbp-0x60]               ; destination = offset 0 of closure  
lea    rsi, [rbp-0x30]               ; source = local prefix variable  
call   std::string::basic_string(std::string const&)  

; 2. Store pointer to descriptions
lea    rax, [rbp-0x80]               ; address of descriptions  
mov    QWORD PTR [rbp-0x28], rax     ; closure.descriptions = &descriptions (offset 32)  
```

The presence of a copy constructor call during closure initialization confirms a by-value capture of a non-trivial type. Conversely, a simple `lea` + `mov` (address storage) confirms a by-reference capture.

## Full copy capture (`[=]`)

```cpp
auto formatShape = [=](const auto& s) -> std::string {
    return prefix + s->name() + " (area >= " +
           std::to_string(minArea) + "? " +
           (s->area() >= minArea ? "yes" : "no") + ")";
};
```

The `[=]` captures **all used variables** by value. GCC only includes in the closure the variables actually referenced in the lambda body — not all variables from the enclosing scope.

In this example, `prefix` (`std::string`, 32 bytes) and `minArea` (`double`, 8 bytes) are captured. The closure is 40 bytes (32 + 8).

> ⚠️ **In RE:** with `[=]` or `[&]`, you don't know in advance which variables are captured. Only analyzing the `operator()` code (which closure offsets are read) and the initialization (which variables are copied/referenced) reveals the effective captures.

## Full reference capture (`[&]`)

```cpp
auto accumulate = [&](const std::shared_ptr<Shape>& s) {
    totalArea += s->area();
    count++;
};
```

The closure contains a pointer for each captured variable. With `[&]`, GCC can optimize by storing only a single pointer to the enclosing function's stack frame, and accessing variables via offsets from that pointer. But in practice (especially in `-O0`), GCC stores a separate pointer per variable.

## Generic lambda (C++14 `auto`)

```cpp
auto formatShape = [=](const auto& s) -> std::string { ... };
```

The `auto` parameter makes the lambda an **implicit template**. GCC generates a template `operator()`, and each call with a different argument type produces a distinct instantiation:

```
{lambda(auto:1 const&)#5}::operator()<std::shared_ptr<Shape>>(std::shared_ptr<Shape> const&) const
{lambda(auto:1 const&)#5}::operator()<std::shared_ptr<Circle>>(std::shared_ptr<Circle> const&) const
```

In RE, if you see multiple `operator()` for the same lambda (same `#N`) with different parameter types, it's a generic lambda instantiated with multiple types.

> 💡 **In RE:** generic lambdas combine the complications of lambdas (anonymous class, captures) and templates (multiple instantiations, symbol explosion). In a stripped binary, look for structurally identical functions that differ only in access sizes or calls to parameter methods — this is the pattern of a template lambda instantiated with different types.

## `mutable` lambda

By default, a lambda's `operator()` is `const` — the lambda can't modify its value captures. The `mutable` keyword removes this restriction:

```cpp
int counter = 0;  
auto increment = [counter]() mutable {  
    return ++counter;  // modifies the local copy
};
```

In assembly, the only difference is that the `this` passed to the `operator()` is non-const, which manifests in the mangling:

```
Without mutable: {lambda()#1}::operator()() const    — K in the mangling (NKUl...)  
With mutable:    {lambda()#1}::operator()()           — no K (NUl...)  
```

In the code, the `operator()` writes to the closure (`mov [rdi+offset], value`) instead of only reading it. In `-O0`, GCC may even pass `this` in a non-const register. In practice, the difference is subtle in the disassembly, but the mangling clearly reveals it.

## Lambdas and `std::function`

When a lambda is stored in a `std::function`, a **type erasure** mechanism comes into play. The lambda (of anonymous type) must be encapsulated in a polymorphic object that `std::function` can uniformly manipulate.

```cpp
std::function<void(const std::string&, const std::shared_ptr<Shape>&)> callback;  
callback = [](const std::string& key, const std::shared_ptr<Shape>& s) {  
    std::cout << key << ": " << s->describe() << std::endl;
};
```

GCC generates a **manager function** and an **invoker function** that encapsulate the lambda:

```
std::_Function_handler<void(string const&, shared_ptr<Shape> const&),
                       {lambda(...)}>::_M_invoke(...)
std::_Function_handler<void(string const&, shared_ptr<Shape> const&),
                       {lambda(...)}>::_M_manager(...)
```

The `std::function` layout is:

```
std::function<R(Args...)> (sizeof = 32 bytes):
┌──────────────────────────────────────┐
│  _M_functor (union, 16 bytes)        │  offset 0    — inline storage or heap pointer
├──────────────────────────────────────┤
│  _M_invoker (function pointer)       │  offset 16   — pointer to invoker
├──────────────────────────────────────┤
│  _M_manager (function pointer)       │  offset 24   — pointer to manager
└──────────────────────────────────────┘
```

The `_M_functor` field stores the closure directly if it's 16 bytes or less (Small Buffer Optimization, analogous to `std::string`'s SSO). Otherwise, the closure is heap-allocated and `_M_functor` contains a pointer to it.

**`std::function` call in assembly:**

```nasm
; Call callback(key, shape)
; rdi = pointer to std::function, rsi = key, rdx = shape
mov    rax, QWORD PTR [rdi+16]       ; _M_invoker (function pointer)  
call   rax                            ; indirect call → _M_invoke  
```

> 💡 **RE pattern:** a `call [rdi+16]` where `rdi` points to a 32-byte object is very likely a `std::function` call. The indirection via `_M_invoker` masks the actual lambda. To find the lambda code, follow the `_M_invoker` pointer — it points to the `_M_invoke` function, which itself calls the closure's `operator()`.

The `std::function` type erasure adds an indirection layer that makes control flow tracing more difficult. In RE, if you see many `_Function_handler::_M_invoke`, the code heavily uses `std::function` (common with callbacks, event handlers, and Observer patterns).

## Lambdas in STL algorithms

Lambdas are very often passed to STL algorithms. In `-O2`, GCC will typically inline both the algorithm and the lambda, making both disappear:

```cpp
auto largeCount = std::count_if(shapes.begin(), shapes.end(), isLargeShape);
```

**In `-O0`:** GCC generates an explicit call to `std::count_if`, which receives a pointer to the closure. The `operator()` is called once per element.

**In `-O2`:** GCC inlines `std::count_if` and inlines the lambda's `operator()`. The result is a simple loop comparing each `area()` with `minArea` directly, with no visible trace of the lambda or the algorithm:

```nasm
; std::count_if inlined with inlined lambda
    mov    rbx, QWORD PTR [rdi]          ; _M_start of vector
    mov    r12, QWORD PTR [rdi+8]        ; _M_finish
    xor    r13d, r13d                     ; counter = 0
.L_loop:
    cmp    rbx, r12
    je     .L_done
    ; Load element (shared_ptr<Shape>) and call area()
    mov    rax, QWORD PTR [rbx]          ; Shape*
    mov    rcx, QWORD PTR [rax]          ; vptr
    mov    rdi, rax
    call   QWORD PTR [rcx+0x10]         ; area() — non-devirtualizable virtual call

    ; Compare with minArea (captured value, now immediate constant)
    ucomisd xmm0, QWORD PTR [rip+.LC_10_0]  ; area > 10.0?
    seta   al
    movzx  eax, al
    add    r13, rax                       ; counter += (area > 10.0)

    add    rbx, 16                        ; advance by sizeof(shared_ptr)
    jmp    .L_loop
.L_done:
    ; r13 = number of elements with area > 10.0
```

Notice that the `minArea` capture (which was 10.0) has become an **immediate constant** in the code (`[rip+.LC_10_0]`). The closure has entirely disappeared — the captured value was propagated as a constant by the optimizer. This is the advantage of lambda inlining: GCC can propagate captures as constants when their values are known at compile time.

> 💡 **In RE (`-O2`):** when a lambda is inlined, it leaves no structural trace in the binary. The lambda code is fused with the algorithm's loop. The only way to recognize the former lambda is to understand the logic: "this loop iterates over a vector and compares something with a constant, it's probably a `count_if` with a predicate."

## Capturing `this`

In a class method, a lambda can capture `this` to access the object's members:

```cpp
class Canvas {
    void draw() const {
        auto drawShape = [this](const std::shared_ptr<Shape>& s) {
            std::cout << title_ << ": " << s->describe() << std::endl;
        };
        std::for_each(shapes_.begin(), shapes_.end(), drawShape);
    }
};
```

The closure captures `this` as a pointer:

```
drawShape closure:
┌──────────────────────────────────────┐
│  this (Canvas*)                      │  offset 0   — 8 bytes
└──────────────────────────────────────┘
```

In the `operator()`, accessing `title_` goes through two levels of indirection:

```nasm
; rdi = closure (lambda's this)
mov    rax, QWORD PTR [rdi]          ; rax = closure->this (Canvas*)
; Access Canvas::title_ (std::string at some offset in Canvas)
lea    rsi, [rax+0x10]               ; rsi = &(canvas->title_)
```

> 💡 **In RE:** a closure that stores a single pointer and uses it to access multiple fields via varied offsets is a lambda capturing `this`. The offsets correspond to the enclosing class's members.

In C++17, `[*this]` captures the object by **copy** instead of capturing the pointer. The closure then contains a complete copy of the object (potentially large). In RE, this manifests as a call to the enclosing class's copy constructor during closure initialization.

## Immediately Invoked Lambda Expression (IILE)

A common pattern in modern C++ is using an immediately invoked lambda to initialize a complex variable:

```cpp
const auto config = [&]() {
    Config c("default", 100, true);
    if (argc > 2) c.maxShapes = std::stoi(argv[2]);
    return c;
}();
```

In `-O0`, GCC generates the closure, calls it immediately, then destroys it. In `-O2`, the lambda is entirely inlined: the lambda code ends up directly at the call site, and the closure never exists as an object. The IILE is then indistinguishable from an ordinary code block in the disassembly.

## Summary of patterns to recognize

| Pattern | Meaning |  
|---------|---------|  
| Symbol containing `{lambda` or `UlvE` / `UlRK...E` in mangling | Lambda's `operator()` |  
| Symbol containing `_FUN` associated with a lambda | Function pointer conversion thunk (capture-less lambda) |  
| Small function receiving `this` (rdi) and accessing `[rdi+offset]` | Lambda `operator()` with captures |  
| Direct access `[rdi+offset]` in the operator() | Capture by value (direct member read) |  
| Double indirection `mov rax, [rdi+offset]; op [rax]` | Capture by reference (pointer then dereference) |  
| `lea` + `mov` (address storage in closure) | Reference capture initialization |  
| Copy constructor call during closure initialization | By-value capture of non-trivial type (string, vector...) |  
| `call [rdi+16]` on a 32-byte object | `std::function` call (invoker via _M_invoker) |  
| `_Function_handler::_M_invoke` / `_M_manager` | Type erasure for lambda in `std::function` |  
| Lambda disappeared in `-O2`, logic fused into a loop | Lambda inlined by optimizer |  
| Immediate constant replacing a value capture in `-O2` | Constant propagation after lambda inlining |  
| Closure with a single pointer accessing varied offsets | Lambda capturing `this` |  
| Multiple `operator()` for the same `#N` with different types | Generic lambda (`auto` parameter) instantiated multiple times |
