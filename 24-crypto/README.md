## Why a chapter dedicated to encryption?

Cryptography is ubiquitous in modern binaries: user data protection, license verification, encrypted network communications, ransomware, DRM. When encountering a binary that manipulates encrypted data, RE takes on an additional dimension: it is no longer enough to understand *what the program does*, you also need to understand *how it protects* (or hides) its data, and above all find the secret elements — keys, initialization vectors, parameters — that allow reproducing or reversing the process.

This chapter does not aim to teach cryptography as such. The objective is to develop the reflexes and techniques that, when facing an unknown binary, allow answering three fundamental questions:

1. **What algorithm is used?** — Does the binary embed a known implementation (AES, ChaCha20, RSA...) or a custom scheme?  
2. **Where are the secrets?** — Is the key hardcoded, derived from user input, read from a file, received over the network?  
3. **How to reproduce the scheme?** — Once the algorithm and parameters are identified, can we write an independent tool (in Python, for example) that encrypts or decrypts the same data?

## What this chapter covers

The `ch24-crypto` binary is a C application compiled with GCC that encrypts a file using a combination of standard algorithms. The encryption key is derived internally according to a logic that must be reconstructed. The `secret.enc` file produced by the binary will serve as the target: by the end of the chapter, you will be able to decrypt it without having the source code.

The path follows a natural progression, from identification to reproduction:

- **Section 24.1** — Identifying cryptographic routines by spotting characteristic magic constants (AES S-box, SHA-256 initialization vectors, permutation tables...) in the binary, whether via `strings`, Ghidra, or YARA rules.  
- **Section 24.2** — Determining whether the binary uses a known crypto library (OpenSSL, libsodium, mbedTLS...) or a homegrown implementation, and adapting the analysis strategy accordingly.  
- **Section 24.3** — Extracting keys and IVs directly from the running process memory, using GDB and Frida — because even a complex key derivation scheme always ends up producing a cleartext key somewhere in RAM.  
- **Section 24.4** — Visualizing the encrypted file structure with ImHex: spotting headers, blocks, cleartext metadata, padding, and understanding the binary's output format.  
- **Section 24.5** — Assembling all the pieces to reproduce the complete encryption scheme in Python and decrypt `secret.enc`.

## Prerequisites

This chapter mobilizes nearly all the tools seen previously. Before starting, make sure you are comfortable with:

- **Static analysis** (Part II) — You will use Ghidra to navigate decompiled code, spot constants, and reconstruct structures. ImHex will serve to inspect the encrypted file format. The `strings` and `readelf` commands will be useful during triage.  
- **Dynamic analysis** (Part III) — GDB will be essential for setting breakpoints at key moments (key derivation, encryption calls) and inspecting memory buffers. Frida will allow hooking crypto functions to capture parameters on the fly.  
- **Cryptography notions** — No need to be a cryptographer, but you should know the difference between symmetric and asymmetric encryption, know what a block cipher mode is (ECB, CBC, CTR...), and understand the role of an IV and a key derivation function (KDF). If these terms are unclear, a quick refresher is provided below.

## Express refresher: minimal crypto vocabulary

To approach this chapter without ambiguity, here are the concepts that will come up constantly.

**Symmetric encryption** — A single secret (the *key*) is used to both encrypt and decrypt. The most common algorithms in binaries are AES (128/256 bits), ChaCha20, and — in legacy or custom code — RC4, DES, Blowfish. This type of encryption is what our `ch24-crypto` binary uses.

**Mode of operation** — A block cipher like AES only encrypts 16 bytes at a time. The *mode* defines how to chain blocks to encrypt an arbitrarily long message. ECB (each block independent, no diffusion — to be avoided); CBC (each block XORed with the previous one, requires an IV); CTR (transforms the block cipher into a stream cipher). Recognizing the mode is crucial because it determines the parameters needed for decryption.

**IV (Initialization Vector)** — A random or unique value used so that two encryptions of the same message with the same key produce different results. The IV is not secret: it is often stored in cleartext at the beginning of the encrypted file.

**KDF (Key Derivation Function)** — A function that transforms a password (or raw secret) into a fixed-size key suitable for the algorithm. PBKDF2, scrypt, Argon2, or a simple hash (SHA-256 of the password) are common approaches. In binaries, the KDF is often the most interesting link to reverse, because it is what connects an observable input (user password, hardcoded value...) to the key actually used.

**Magic constants** — Each crypto algorithm relies on specific mathematical constants: the AES S-box (256 bytes starting with `0x63, 0x7c, 0x77, 0x7b...`), SHA-256 initialization vectors (`0x6a09e667, 0xbb67ae85...`), SHA-256 round constants, the DES substitution table... These constants are infallible signatures for identifying an algorithm in a binary, even stripped and optimized.

## General strategy for a "crypto" binary

Before diving into the sections, here is the mental approach to adopt. It applies well beyond our training binary.

**Step 1 — Classic triage.** As with any binary, start with `file`, `strings`, `checksec`, `readelf`. Look for direct clues: crypto function names in the symbol table (`AES_encrypt`, `EVP_CipherInit`, `SHA256_Update`...), revealing strings (`"AES-256-CBC"`, `"PBKDF2"`...), or dynamically linked libraries (`libcrypto.so`, `libsodium.so`).

**Step 2 — Algorithm identification.** If triage is not enough (stripped binary, statically embedded crypto, custom implementation), move to magic constant search. This is often the most reliable lever: a developer can rename functions, strip symbols, obfuscate control flow — but cannot modify AES mathematical constants without breaking the algorithm.

**Step 3 — Secret localization.** Identify where the key is generated or stored. Static analysis (XREF to crypto constants, tracing back to calling functions) complemented by dynamic analysis (breakpoint on the encryption routine, argument inspection).

**Step 4 — Understanding the output format.** Examine the encrypted file with ImHex to understand its structure: is there a header? Is the IV stored as a prefix? Is there a MAC (authentication code)? What padding is used?

**Step 5 — Reproduction and validation.** Write a Python script that implements the same scheme and verify that you can decrypt `secret.enc`.
