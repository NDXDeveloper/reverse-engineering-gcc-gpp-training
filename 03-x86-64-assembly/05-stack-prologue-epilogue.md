# 3.5 ‚ÄĒ The stack: prologue, epilogue and System V AMD64 calling conventions

> ūüéĮ **Goal of this section**: understand how the stack works in x86-64, know how to decode the function prologues and epilogues generated by GCC, and master the rules of the System V AMD64 calling convention that governs how functions communicate with each other under Linux.

---

## The x86-64 stack: fundamental principles

The stack is a memory zone used by the processor to temporarily store data during program execution. It serves three fundamental purposes:

1. **Saving the return address** on a `call` (section 3.2).  
2. **Storing the local variables** of functions.  
3. **Saving the callee-saved registers** that the function wishes to use.

### Growth toward lower addresses

On x86-64, the stack grows toward **lower addresses**. This means each `push` **decrements** `rsp`, and each `pop` **increments** `rsp`. This is counterintuitive at first, but it is a hardware convention inherited from the early x86 architectures.

```
High addresses (e.g.: 0x7fffffffe000)
    ‚ĒĆ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚Ēź
    ‚Ēā                             ‚Ēā  ‚Üź bottom of stack (start of program)
    ‚Ēā    frames of caller         ‚Ēā
    ‚Ēā    functions                ‚Ēā
    ‚Ēā         ...                 ‚Ēā
    ‚Ēā‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚Ēā
    ‚Ēā    frame of the current     ‚Ēā

    "Pop"   = go upward   (rsp increases)
```

### The `rsp` register as an anchor

The `rsp` register **always** points to the last element pushed (the top of the stack). Any operation that touches the stack goes through `rsp`:

| Operation | Effect on `rsp` | Effect in memory |  
|---|---|---|  
| `push rax` | `rsp -= 8` | writes `rax` at `[rsp]` |  
| `pop rax` | `rsp += 8` | reads `[rsp]` into `rax` |  
| `sub rsp, 0x20` | `rsp -= 32` | reserves 32 bytes |  
| `add rsp, 0x20` | `rsp += 32` | frees 32 bytes |  
| `call func` | `rsp -= 8` | pushes `rip` (return address) |  
| `ret` | `rsp += 8` | pops into `rip` |

Each function call creates a **stack frame** ‚ÄĒ a contiguous zone on the stack that contains everything the function needs. Here is the complete layout of a stack frame under the System V AMD64 convention, in the most general case (with frame pointer, saved registers, and arguments on the stack):

## The function prologue

The prologue is the sequence of instructions at the very start of a function that **builds** the stack frame. It is the first code you see when navigating to a function in a disassembler.

### Classic prologue with frame pointer (`-O0`)

```asm
push    rbp             ; 1. saves the old base pointer  
mov     rbp, rsp        ; 2. establishes the new base pointer  
sub     rsp, 0x30       ; 3. reserves 48 bytes for local variables  
```

**Step 1 ‚ÄĒ `push rbp`**: saves the `rbp` value of the calling function. This makes it possible to restore the old frame pointer at the end of the function, which "walks back" the chain of stack frames.

**Step 2 ‚ÄĒ `mov rbp, rsp`**: sets `rbp` to the current top of the stack. From this moment on, `rbp` serves as a stable reference: local variables are at negative offsets (`[rbp-0x4]`, `[rbp-0x8]`‚Ä¶), and the return address is at `[rbp+0x8]`.

**Step 3 ‚ÄĒ `sub rsp, N`**: reserves space on the stack for local variables and alignment. The value of `N` tells you the amount of local space used.

### Prologue with callee-saved register saving

When the function uses callee-saved registers (`rbx`, `r12`‚Äď`r15`), they are saved after (or sometimes before) establishing the frame pointer:

```asm
push    rbp  
mov     rbp, rsp  
push    rbx             ; saves rbx (will be used in the body)  
push    r12             ; saves r12 (will be used in the body)  
sub     rsp, 0x20       ; reserves local space  
```

Each extra `push` decrements `rsp` by 8 bytes. The space reserved by `sub rsp, N` accounts for these `push`es so that the total is properly aligned.

> ūüí° **For RE**: the number of callee-saved register `push`es is a hint about the function's complexity. A function with `push rbx` / `push r12` / `push r13` / `push r14` / `push r15` uses 5 callee-saved registers ‚ÄĒ it probably has many variables that must survive across internal function calls.

### Prologue with stack canary

On modern distributions, GCC compiles by default with `-fstack-protector-strong`, which inserts a *stack canary* ‚ÄĒ a sentinel value placed between local variables and the return address to detect buffer overflows:

```asm
push    rbp  
mov     rbp, rsp  
sub     rsp, 0x40  
mov     rax, qword [fs:0x28]       ; reads the canary from the TLS  
mov     qword [rbp-0x8], rax       ; places the canary on the stack  
xor     eax, eax                    ; clears rax (security good practice)  
```

The `[fs:0x28]` access is the immediate visual signature of the stack canary. When you see it in a prologue, you know the function is protected and the epilogue will contain the corresponding check.

### Prologue without frame pointer (`-O1` and above)

With optimizations enabled, GCC uses `-fomit-frame-pointer` by default. The prologue is shorter ‚ÄĒ no `push rbp` or `mov rbp, rsp`:

```asm
sub     rsp, 0x28           ; reserves space directly
; or simply:
push    rbx                  ; saves a callee-saved, adjusts rsp by 8  
sub     rsp, 0x20            ; reserves the rest  
```

Without a frame pointer, **all accesses to local variables use offsets relative to `rsp`** instead of `rbp`:

```asm
; With frame pointer (-O0)
mov     eax, dword [rbp-0x4]       ; local variable via rbp (stable)

; Without frame pointer (-O1+)
mov     eax, dword [rsp+0x1c]      ; same variable, but via rsp
```

The difficulty in RE is that offsets relative to `rsp` **change** with each `push`, `pop`, or `sub rsp` within the function, whereas offsets relative to `rbp` remain fixed. Ghidra and IDA handle this complexity automatically by tracking the value of `rsp` throughout the function (*stack pointer tracking*), but in manual analysis with `objdump`, it is more demanding.

## The function epilogue

The epilogue **undoes** what the prologue built, in reverse order, then returns to the caller.

### Classic epilogue with frame pointer

```asm
; === canary check (if -fstack-protector) ===
mov     rax, qword [rbp-0x8]       ; re-reads the canary from the stack  
xor     rax, qword [fs:0x28]       ; compares with the original value  
jne     .stack_chk_fail             ; if different ‚Üí corruption detected!  

; === restoration ===
add     rsp, 0x20                   ; frees local space (or implicit via leave)  
pop     r12                         ; restores r12 (reverse order of prologue)  
pop     rbx                         ; restores rbx  
pop     rbp                         ; restores old base pointer  
ret                                  ; pops return address, jumps  
```

Or, more frequently, with the `leave` instruction:

```asm
leave                               ; equivalent to: mov rsp, rbp / pop rbp  
ret  
```

`leave` is a shortcut that combines the restoration of `rsp` and the `pop rbp` into a single instruction. It is the form GCC uses most often at `-O0`.

### Epilogue without frame pointer

```asm
add     rsp, 0x20           ; frees local space  
pop     rbx                  ; restores rbx  
ret  
```

Without a frame pointer, there is no `leave` or `pop rbp` ‚ÄĒ you just raise `rsp` and restore the callee-saved registers.

### The return value

Before the epilogue, the function places its return value in `rax` (or `eax` for an `int`):

```asm
mov     eax, dword [rbp-0x4]    ; loads the result
; ... epilogue ...
ret                               ; the caller reads the value from eax/rax
```

For floating-point values (`float`, `double`), the return happens via `xmm0`. For small structures (‚Č§ 16 bytes), GCC may return them in `rax` + `rdx`. Larger structures are returned via a hidden pointer passed as the first argument (cf. below in this section).

## 16-byte stack alignment

The System V AMD64 convention imposes a strict alignment rule:

> **At the time of a `call`, `rsp` must be aligned on 16 bytes.**

More precisely, just *before* executing the `call`, `rsp` must be a multiple of 16. Since `call` pushes 8 bytes (the return address), this means that at the called function's entry point, `rsp` is a **multiple of 16 + 8** (i.e., `rsp mod 16 == 8`).

The prologue must therefore ensure that `rsp` becomes a multiple of 16 before any outgoing `call`. GCC handles this automatically by adjusting the size of `sub rsp, N` or inserting an additional `push`.

### Why 16 bytes?

SSE/AVX instructions require certain memory accesses to be aligned on 16 bytes (`movaps`, `movdqa` instructions‚Ä¶). Since libc and other libraries use these instructions, violating alignment causes an immediate crash (`SIGSEGV` or `SIGBUS`).

### What it looks like in practice

```asm
; Function that calls printf ‚ÄĒ prologue adjusted for alignment
push    rbp                 ; rsp -= 8 ‚Üí rsp mod 16 == 0  
mov     rbp, rsp  
sub     rsp, 0x10           ; 16 bytes ‚Üí rsp remains aligned on 16  
; ... argument preparation ...
call    printf@plt          ; rsp is properly aligned before the call
```

If the number of `push`es + the size of `sub rsp` does not yield a multiple of 16, GCC inserts padding:

```asm
push    rbp                 ; rsp -= 8  
push    rbx                 ; rsp -= 8 ‚Üí rsp mod 16 == 0 (2 √ó 8 = 16)  
sub     rsp, 0x18           ; 24 bytes ‚Üí rsp -= 24 ‚Üí total = 8+8+24 = 40  
                            ; 40 mod 16 = 8... not aligned!
                            ; GCC would rather do sub rsp, 0x20 (32 bytes)
                            ; to reach a total of 48 (multiple of 16)
```

> ūüí° **For RE**: if you see a `sub rsp` whose value seems larger than necessary for local variables, the surplus is probably **alignment padding**. Do not look for a hidden local variable in those extra bytes.

## The System V AMD64 convention in detail

The calling convention is the set of rules that defines how functions pass arguments, return results, and share registers. Under Linux x86-64, the **System V AMD64 ABI** applies to any code compiled by GCC (and Clang, Rust, Go via the GNU linker‚Ä¶).

### Passing integer and pointer arguments

The **first six** integer or pointer arguments are passed via registers, in this order:

| Argument | Register |  
|---|---|  
| 1st | `rdi` |  
| 2nd | `rsi` |  
| 3rd | `rdx` |  
| 4th | `rcx` |  
| 5th | `r8` |  
| 6th | `r9` |

From the **7th argument** onward, values are pushed onto the stack, right-to-left (the 7th argument is at the top).

```c
// C code
long result = func(10, 20, 30, 40, 50, 60, 70, 80);
```

```asm
; Argument preparation
push    80                  ; 8th argument ‚Üí stack  
push    70                  ; 7th argument ‚Üí stack  
mov     r9d, 60             ; 6th argument ‚Üí r9  
mov     r8d, 50             ; 5th argument ‚Üí r8  
mov     ecx, 40             ; 4th argument ‚Üí rcx  
mov     edx, 30             ; 3rd argument ‚Üí rdx  
mov     esi, 20             ; 2nd argument ‚Üí rsi  
mov     edi, 10             ; 1st argument ‚Üí rdi  
call    func  
add     rsp, 16             ; cleans up the 2 pushed arguments (2 √ó 8 bytes)  
```

> ūüí° **For RE**: spotting the `mov edi, ...` / `mov esi, ...` / `mov edx, ...` just before a `call` gives you the function's arguments directly. It is a fundamental reflex for understanding what a call does.

### Passing floating-point arguments

`float` and `double` arguments use the SSE registers:

| Float argument | Register |  
|---|---|  
| 1st | `xmm0` |  
| 2nd | `xmm1` |  
| 3rd | `xmm2` |  
| 4th | `xmm3` |  
| 5th | `xmm4` |  
| 6th | `xmm5` |  
| 7th | `xmm6` |  
| 8th | `xmm7` |

Integer and floating-point arguments are counted **independently**. A call `func(int a, double b, int c, double d)` uses `rdi` for `a`, `xmm0` for `b`, `rsi` for `c`, and `xmm1` for `d`.

### Variadic functions (`printf`, `scanf`‚Ä¶)

For functions with a variable number of arguments (declared with `...`), the convention requires that **`al`** (the low byte of `rax`) contain the **number of SSE registers used** to pass floating-point arguments. This is a very recognizable pattern:

```asm
; printf("x = %d, y = %f\n", x, y)
lea     rdi, [rip+0x1234]      ; 1st arg: format string  
mov     esi, ebx                ; 2nd arg: x (integer ‚Üí rsi)  
movsd   xmm0, qword [rbp-0x10] ; 3rd arg: y (double ‚Üí xmm0)  
mov     eax, 1                  ; 1 SSE register used (xmm0)  
call    printf@plt  
```

If no floating-point argument is passed, you will see:

```asm
xor     eax, eax           ; 0 SSE registers used  
call    printf@plt  
```

> ūüí° **For RE**: a `xor eax, eax` or `mov eax, N` just before a `call` to a known function like `printf`, `scanf`, `sprintf`‚Ä¶ is the SSE argument counter, not a real argument. It is specific to variadic functions ‚ÄĒ fixed-prototype functions do not use it.

### Return value

| Return type | Register(s) |  
|---|---|  
| Integer, pointer (‚Č§ 64 bits) | `rax` |  
| 128-bit integer | `rax` (low) + `rdx` (high) |  
| `float`, `double` | `xmm0` |  
| Small struct (‚Č§ 16 bytes) | `rax` and/or `rdx` and/or `xmm0`/`xmm1` |  
| Large struct (> 16 bytes) | Hidden pointer as 1st argument (`rdi`) |

The last case is the most surprising: when a function returns a structure larger than 16 bytes, the caller reserves space on its own stack and passes a pointer to that space as a **hidden first argument** in `rdi`. All other arguments shift one position. The function writes the result at the address pointed to by `rdi` and returns that same address in `rax`.

```c
// C code
typedef struct { char data[64]; } BigStruct;  
BigStruct make_big(int x);  

BigStruct s = make_big(42);
```

```asm
; The caller
lea     rdi, [rbp-0x50]     ; rdi = &s (space reserved on the caller's stack)  
mov     esi, 42              ; esi = x (shifted to 2nd argument!)  
call    make_big  
; rax points to [rbp-0x50], which now contains the result
```

> ‚ö†ÔłŹ **For RE**: if you analyze a function and the first argument (`rdi`) seems to be a pointer the function writes to heavily without ever reading from, it is probably the **hidden return pointer** for a large structure. This shifts all visible arguments by one position ‚ÄĒ the argument you thought was first is actually second.

### Callee-saved vs caller-saved registers (summary)

| Category | Registers | Who saves? |  
|---|---|---|  
| **Caller-saved** (volatile) | `rax`, `rcx`, `rdx`, `rsi`, `rdi`, `r8`‚Äď`r11` | The **caller** must save these registers before a `call` if it wants to keep their value |  
| **Callee-saved** (non-volatile) | `rbx`, `rbp`, `r12`‚Äď`r15` | The **called function** must restore them before returning |  
| **Stack pointer** | `rsp` | Restored implicitly by the epilogue |

In concrete terms for RE:

- If a function uses `rbx` in its body, it **must** do `push rbx` in the prologue and `pop rbx` in the epilogue ‚ÄĒ otherwise it violates the ABI and corrupts the caller's state.  
- If a function needs a value to survive an internal `call`, it stores it in a callee-saved register (which costs a `push`/`pop`) or on the stack (which costs a memory `mov`).

## The Red Zone ‚ÄĒ 128 free bytes below `rsp`

The System V AMD64 ABI defines a **red zone**: a 128-byte zone **below** `rsp` (at addresses `rsp-1` through `rsp-128`) that the function can use freely **without decrementing `rsp`**.

```asm
; Leaf function using the red zone (no sub rsp)
mov     dword [rsp-0x4], edi      ; stores 1st argument below rsp  
mov     dword [rsp-0x8], esi      ; stores 2nd argument  
; ... computations using [rsp-0x4] and [rsp-0x8] ...
ret
```

The red zone is only usable by **leaf functions** ‚ÄĒ those that do not call any other function. A `call` would overwrite the red zone by pushing the return address. Similarly, a signal or interrupt can overwrite the red zone (except that the Linux kernel respects this convention for signals).

Advantage: no need for `sub rsp` / `add rsp`, which saves instructions and makes the prologue/epilogue non-existent. GCC exploits the red zone at `-O1` and above for small leaf functions.

> ūüí° **For RE**: if you see a function that accesses `[rsp-0x...]` without having done a `sub rsp` beforehand, it is not a bug ‚ÄĒ it is red zone usage. These functions often have **no visible prologue or epilogue**, just useful code followed by a `ret`.

> ‚ö†ÔłŹ **Warning**: the red zone does **not** exist in the Windows x64 convention (Microsoft ABI), nor in Linux kernel mode. If you analyze a driver or kernel module, accesses below `rsp` without `sub rsp` are indeed suspicious.

## Prologue variations depending on context

In practice, you will not always see the "textbook" prologue `push rbp` / `mov rbp, rsp` / `sub rsp, N`. Here are the common variations and how to recognize them:

### Trivial leaf function (red zone, no prologue)

```asm
; int add(int a, int b) { return a + b; }
add:
    lea     eax, [rdi+rsi]
    ret
```

No prologue, no epilogue. The function is so simple it needs neither stack nor saving.

### Leaf function with local variables (red zone)

```asm
; Uses the red zone ‚ÄĒ no sub rsp
process:
    mov     dword [rsp-0x4], edi
    mov     dword [rsp-0x8], esi
    ; ... computations ...
    mov     eax, dword [rsp-0x4]
    ret
```

### Function with frame pointer (`-O0` or `-fno-omit-frame-pointer`)

```asm
func:
    push    rbp
    mov     rbp, rsp
    sub     rsp, 0x20
    ; ... body with [rbp-X] accesses ...
    leave
    ret
```

It is the "classic" form taught in architecture courses. Local variables are at fixed negative offsets relative to `rbp`.

### Function without frame pointer (`-O1` and above)

```asm
func:
    sub     rsp, 0x28
    ; ... body with [rsp+X] accesses ...
    add     rsp, 0x28
    ret
```

Or with callee-saved saving:

```asm
func:
    push    rbx
    push    r12
    sub     rsp, 0x18
    ; ... body ...
    add     rsp, 0x18
    pop     r12
    pop     rbx
    ret
```

### Prologue of `main()` ‚ÄĒ special alignment

The prologue of `main()` often contains an explicit alignment of `rsp` on 16 bytes, because `main` is called by the *C runtime* (`__libc_start_main`) and GCC wants to guarantee alignment regardless of context:

```asm
main:
    push    rbp
    mov     rbp, rsp
    and     rsp, 0xfffffffffffffff0   ; aligns rsp on 16 bytes
    sub     rsp, 0x20
    ; ...
```

The `and rsp, -16` instruction (displayed as `and rsp, 0xfffffffffffffff0`) is the signature of `main()`'s prologue.

## Reconstructing the stack frame in RE

One of the most common exercises in reverse engineering is **reconstructing the layout of the stack frame** ‚ÄĒ i.e., determining which local variables are stored at which offsets, and assigning them a name and a type.

### Method with frame pointer (`rbp`)

When the frame pointer is present, the task is relatively simple. Each negative offset relative to `rbp` is a local variable:

```asm
mov     dword [rbp-0x4], 0          ; variable at rbp-4, 4 bytes ‚Üí int  
mov     qword [rbp-0x10], rdi       ; variable at rbp-16, 8 bytes ‚Üí pointer  
mov     byte [rbp-0x11], 0x41       ; variable at rbp-17, 1 byte ‚Üí char  
lea     rax, [rbp-0x50]             ; address of rbp-80 ‚Üí array/buffer  
```

From which you can deduce:

```
Offset      Size      Probable type    Suggested name  
rbp-0x04    4         int              var_4 (or counter, index‚Ä¶)  
rbp-0x10    8         pointer          var_10 (or ptr, buf‚Ä¶)  
rbp-0x11    1         char             var_11  
rbp-0x50    64        char[64]         buffer (because lea takes the address)  
```

### Method without frame pointer (`rsp`)

Without a frame pointer, offsets are relative to `rsp` and vary throughout the function. Ghidra and IDA normalize these offsets automatically. In manual analysis, you must trace `rsp`'s value instruction by instruction:

```asm
sub     rsp, 0x28                    ; rsp -= 40  
mov     dword [rsp+0x1c], edi        ; ‚Üí real offset from base = 0x1c  
mov     dword [rsp+0x18], esi        ; ‚Üí real offset = 0x18  
; ...
call    some_func                     ; rsp does not change (caller clean-up)
; ...
mov     eax, dword [rsp+0x1c]        ; re-reads the same variable  
add     rsp, 0x28                    ; rsp += 40  
ret  
```

> ūüí° **For RE**: in Ghidra, the "Decompiler" window performs this work automatically and presents you with named local variables and inferred types. It is one of the major advantages of a decompilation tool over raw assembly listing.

## Visualizing the stack frame chain

This is the chain GDB walks when you type `backtrace` ‚ÄĒ it follows the `rbp` pointers from frame to frame to reconstruct the call stack. ## What to remember going forward

1. **The stack grows downward** ‚ÄĒ `push` decrements `rsp`, `pop` increments it.  
2. **The prologue** builds the stack frame (`push rbp` / `mov rbp, rsp` / `sub rsp, N`), **the epilogue** tears it down (`leave` / `ret` or `add rsp, N` / `pop` / `ret`).  
3. **With frame pointer** (`-O0`): local variables at `[rbp-X]`, stack arguments at `[rbp+0x10]` and beyond. **Without frame pointer** (`-O1+`): everything is relative to `rsp`, and offsets vary.  
4. **16-byte alignment** is mandatory before every `call` ‚ÄĒ that is the cause of apparently oversized `sub rsp`s and "useless" `push`es.  
5. **The red zone** (128 bytes below `rsp`) allows leaf functions to avoid any prologue.  
6. **System V AMD64 ABI**: 6 integer arguments by registers (`rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9`), return in `rax`, floats via `xmm0`‚Äď`xmm7`. Arguments beyond the 6th go on the stack.  
7. **For variadic functions**, `al` contains the number of SSE registers used ‚ÄĒ that is the mysterious `xor eax, eax` or `mov eax, N` before `call printf`.  
8. **Large structures** are returned via a hidden pointer in `rdi`, shifting all arguments by one position.
