# Chapter 3 — x86-64 Assembly Basics for RE

> 🎯 **Chapter goal**: acquire the essential x86-64 assembly basics needed to read and understand the disassembled code produced by GCC/G++. The point is not to become an assembly developer, but to know how to *read* what the compiler has generated.

---

## Why this chapter is essential

When you open a binary in a disassembler — whether it's `objdump`, Ghidra, IDA, or Radare2 — you do not see C, nor C++. You see assembly. It is the processor's mother tongue, and it is the lowest representation level at which the reverse engineer works daily.

No need to master everything at once: in practice, a handful of instructions covers the vast majority of the code generated by GCC. This chapter focuses on that essential core, always keeping the same angle: **"what do I see in a real binary, and what does it mean?"**

---

## What you will learn

- **The x86-64 registers** and their role in compiled code: general-purpose registers, stack pointer, instruction pointer, flags register.  
- **The fundamental instructions**: data movements (`mov`, `lea`), stack manipulation (`push`, `pop`), function calls (`call`, `ret`).  
- **Arithmetic and logic**: the operations the compiler uses to translate your C expressions (`add`, `sub`, `imul`, `xor`, `shl`, `shr`, `test`, `cmp`).  
- **Conditional jumps**: how a simple `if` in C becomes a `cmp` / `jz` pair in assembly, and how to recognize `for` and `while` loops.  
- **The stack and calling conventions**: function prologue and epilogue, the System V AMD64 convention used under Linux, and how parameters are passed via registers then the stack.  
- **A reading method**: a practical 5-step approach to tackling an assembly listing without feeling overwhelmed.  
- **The difference between `call printf@plt` and `syscall`**: two very different ways of interacting with the system, which you encounter constantly in RE.  
- **A first look at SIMD instructions** (SSE/AVX): not to master them, but to recognize them when they appear in optimized code and not panic.

## Conventions used in this chapter

Throughout this chapter, assembly is displayed in **Intel syntax** (destination on the left, source on the right), which is the default syntax of most modern RE tools (Ghidra, IDA, Binary Ninja). Section 3.2 briefly covers the differences with AT&T syntax (used by default by `objdump` and GAS), and Chapter 7 revisits it in detail.

Assembly code examples are systematically paired with the corresponding C code, to ease the mental mapping between the two representation levels:

```c
// C code
int result = a + b;
```

```asm
; x86-64 assembly (Intel syntax)
mov     eax, dword [rbp-0x8]    ; load a  
add     eax, dword [rbp-0xc]    ; add b  
mov     dword [rbp-0x4], eax    ; store into result  
```

The sample compilations use GCC with `-O0` (no optimization) unless otherwise stated, so that the assembly code stays close to the structure of the C source. The effects of higher optimization levels are covered in depth in Chapter 16.
