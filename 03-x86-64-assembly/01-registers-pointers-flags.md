# 3.1 — General registers, pointers and flags (`rax`, `rsp`, `rbp`, `rip`, `RFLAGS`…)

> 🎯 **Goal of this section**: know the x86-64 registers you encounter constantly in disassembled code, understand their respective roles, and know why the same register can appear under several names (`rax`, `eax`, `ax`, `al`).

---

## What is a register?

A register is a tiny and ultra-fast storage area, directly embedded in the processor. Where a RAM access takes dozens of nanoseconds, a register access takes a single clock cycle — it is the CPU's immediate workspace.

When the compiler translates your C code into assembly, it strives to keep the most frequently used variables in registers rather than in memory (on the stack). This is one of the reasons the disassembled code is full of register names: they are, in a sense, the processor's "local variables".

In x86-64, the general-purpose registers are **64 bits** (8 bytes), which makes it possible to manipulate memory addresses and 64-bit integers directly.

---

## General-purpose registers

The x86-64 architecture has **16 general-purpose registers** (against 8 on x86 32-bit). Here is the complete list, with the role each one plays *by convention* in code generated by GCC under Linux (System V AMD64 ABI convention — detailed in sections 3.5 and 3.6):

| 64-bit register | Conventional role | Callee-saved? |  
|---|---|---|  
| `rax` | Function return value | No |  
| `rbx` | General purpose (historically "base") | **Yes** |  
| `rcx` | 4th integer argument | No |  
| `rdx` | 3rd integer argument | No |  
| `rsi` | 2nd integer argument ("source index") | No |  
| `rdi` | 1st integer argument ("destination index") | No |  
| `rbp` | Stack frame base pointer | **Yes** |  
| `rsp` | Stack top pointer | **Yes** (implicitly) |  
| `r8` | 5th integer argument | No |  
| `r9` | 6th integer argument | No |  
| `r10` | General purpose (temporary) | No |  
| `r11` | General purpose (temporary) | No |  
| `r12` | General purpose | **Yes** |  
| `r13` | General purpose | **Yes** |  
| `r14` | General purpose | **Yes** |  
| `r15` | General purpose | **Yes** |

The "callee-saved" column is crucial for RE: if a function uses `rbx`, `r12`–`r15`, or `rbp`, it **must** save them at the start of the function (generally with a `push`) and restore them before returning (with a `pop`). You will systematically see these `push`/`pop` operations in prologues and epilogues — it is not noise, it is the ABI convention in action.

Conversely, caller-saved registers (`rax`, `rcx`, `rdx`, `rsi`, `rdi`, `r8`–`r11`) can be freely overwritten by any `call`. If a function needs to preserve their value across a call, it must save them itself beforehand.

> 💡 **For RE**: when you see a series of `push rbx` / `push r12` / `push r13` at the start of a function, it immediately tells you that the function uses these registers to store values that must survive sub-function calls. It is a hint about the function's complexity.

---

## Sub-registers: why `rax`, `eax`, `ax`, `al`?

The x86-64 architecture is backward-compatible with all previous generations (x86 32-bit, 16-bit, and even 8-bit). Each 64-bit register is therefore accessible by slices, through different names:

```
Bits:   63                           31              15      7      0
        ┌────────────────────────────┬───────────────┬───────┬──────┐
  rax   │         (bits 63-32)       │     eax       │  ax   │      │
        │                            │               ├───────┤      │
        │                            │               │  ah   │  al  │
        └────────────────────────────┴───────────────┴───────┴──────┘

  rax  = full 64 bits
  eax  = low 32 bits (bits 31-0)
  ax   = low 16 bits (bits 15-0)
  ah   = bits 15-8 (high byte of ax)
  al   = bits 7-0  (low byte)
```

The same principle applies to all the classic registers:

| 64-bit | 32-bit | 16-bit | 8-bit high | 8-bit low |  
|---|---|---|---|---|  
| `rax` | `eax` | `ax` | `ah` | `al` |  
| `rbx` | `ebx` | `bx` | `bh` | `bl` |  
| `rcx` | `ecx` | `cx` | `ch` | `cl` |  
| `rdx` | `edx` | `dx` | `dh` | `dl` |  
| `rsi` | `esi` | `si` | — | `sil` |  
| `rdi` | `edi` | `di` | — | `dil` |  
| `rbp` | `ebp` | `bp` | — | `bpl` |  
| `rsp` | `esp` | `sp` | — | `spl` |

For the `r8`–`r15` registers, the naming is more regular:

| 64-bit | 32-bit | 16-bit | 8-bit low |  
|---|---|---|---|  
| `r8` | `r8d` | `r8w` | `r8b` |  
| `r9` | `r9d` | `r9w` | `r9b` |  
| … | … | … | … |  
| `r15` | `r15d` | `r15w` | `r15b` |

### Why this matters for RE

The choice of register name in the disassembly **reveals the size of the type being manipulated**:

```asm
mov     eax, 0x1          ; 32-bit operation → probably an int  
mov     al, byte [rdi]    ; 8-bit operation  → probably a char / byte  
mov     rax, [rbp-0x10]   ; 64-bit operation → probably a pointer (or a long)  
```

This is an essential reflex: when you read disassembled code and see `eax` rather than `rax`, you immediately know the underlying variable is a 32-bit integer (like an `int` in C under Linux x86-64), not a pointer.

> ⚠️ **Classic pitfall**: writing to a 32-bit sub-register (`eax`) automatically zeros out the upper 32 bits of the corresponding 64-bit register (`rax`). This behavior is specific to 64-bit mode and is part of the AMD64 specification. In contrast, writing to an 8- or 16-bit sub-register (`al`, `ax`) does **not** touch the upper bits. GCC exploits this rule: a `mov eax, 0` is an efficient way to zero `rax` (and in practice, GCC often prefers `xor eax, eax`, which is even shorter — we will revisit this in section 3.3).

## Special-purpose registers

Three registers are not interchangeable with the others: the processor assigns them a specific hardware role.

### `rip` — the instruction pointer (*Instruction Pointer*)

`rip` contains the address of the **next instruction** to execute. You never manipulate it directly with a `mov` — it advances automatically after each instruction, and it is modified implicitly by jump instructions (`jmp`, `jz`, `call`, `ret`…).

In RE, `rip` is everywhere:

- In **GDB**, when you do `break *0x401234`, you set a breakpoint at a future value of `rip`.  
- In **Ghidra**, the address column on the left of the listing corresponds to the successive values of `rip`.  
- **RIP-relative addressing** is the norm in x86-64 for accessing global data and strings:

```asm
lea     rdi, [rip+0x2e5a]    ; load the address of a .rodata string  
call    puts@plt  
```

This `lea` instruction does not load the value *at* the address `rip+0x2e5a` — it computes the address itself and places it in `rdi`. We will see this in detail in section 3.2.

### `rsp` — the stack pointer (*Stack Pointer*)

`rsp` always points to the **top of the stack** (the lowest address of the current stack frame, since the stack grows toward lower addresses on x86-64). It is modified implicitly by `push`, `pop`, `call`, and `ret`, and explicitly by instructions like `sub rsp, 0x20` (reserving local space in the function prologue).

```
High addresses
    ┌──────────────────┐
    │   ...            │
    │  stack arguments │  ← beyond the 6th argument
    │  return address  │  ← pushed by call
    │  saved old rbp   │  ← saved by the prologue
    │  local           │
    │  variables       │
    │                  │  ← rsp points here (top)
    └──────────────────┘
Low addresses
```

> 💡 **For RE**: if you see `mov eax, [rsp+0x8]` or `mov [rsp+0x10], rdi`, the function is accessing its local variables or arguments passed through the stack. The offset value relative to `rsp` helps you reconstruct the stack frame layout.

### `rbp` — the frame base pointer (*Base Pointer*)

In non-optimized mode (`-O0`), GCC uses `rbp` as a **fixed anchor** of the current stack frame. The classic prologue of a function looks like this:

```asm
push    rbp             ; saves the old rbp  
mov     rbp, rsp        ; rbp = new base pointer  
sub     rsp, 0x20       ; reserves 32 bytes for local variables  
```

With this scheme, local variables are accessed via negative offsets from `rbp` (`[rbp-0x4]`, `[rbp-0x8]`, etc.) and arguments passed through the stack (if any) are accessed via positive offsets (`[rbp+0x10]`, `[rbp+0x18]`, etc.).

> ⚠️ **Frame pointer omission**: with optimizations enabled (`-O1` and above), GCC activates `-fomit-frame-pointer` by default. The `rbp` register is then freed to serve as a general-purpose register, and all stack accesses happen relative to `rsp`. This is more efficient (one more register available), but makes reading disassembly a bit more demanding, because offsets relative to `rsp` change with every `push` or `sub rsp`. This is one of the effects of optimizations we will detail in Chapter 16.

## The `RFLAGS` register — the processor flags

`RFLAGS` is a special register that never appears by its name in assembly code, but whose **individual bits** (called *flags*) control the behavior of conditional jumps. It is the fundamental mechanism that translates C's `if`, `else`, `while`, and `for` into assembly.

The principle is simple and ubiquitous:

1. A **comparison** or **arithmetic** instruction (`cmp`, `test`, `sub`, `add`…) modifies the flags.  
2. A **conditional jump** instruction (`jz`, `jnz`, `jl`, `jge`…) reads the flags and decides whether to jump.

### The essential flags for RE

Of the 64 bits of `RFLAGS`, only a few flags appear regularly in code produced by GCC:

| Flag | Name | Bit | Meaning | Set to 1 when… |  
|---|---|---|---|---|  
| **ZF** | Zero Flag | 6 | The result is zero | `cmp a, b` with `a == b`, or `sub` / `test` whose result is 0 |  
| **SF** | Sign Flag | 7 | The result is negative (high-order bit = 1) | The result interpreted as signed is negative |  
| **CF** | Carry Flag | 0 | Carry in unsigned arithmetic | Overflow on an unsigned operation |  
| **OF** | Overflow Flag | 11 | Overflow in signed arithmetic | The signed result does not fit in the register |  
| **PF** | Parity Flag | 2 | Parity of the low byte of the result | Even number of set bits in the low byte |

In practice, **ZF** and **SF** cover the majority of cases you will encounter in RE of applicative code compiled by GCC. **CF** and **OF** mainly come into play in unsigned comparisons and overflow checks.

### The `cmp` + conditional-jump pair

The `cmp` / conditional-jump pair is the most frequent pattern in any compiled binary. Here is how it works:

```c
// C code
if (x == 42) {
    do_something();
}
```

```asm
; Assembly generated by GCC (-O0)
cmp     dword [rbp-0x4], 0x2a   ; compare x with 42  
jne     .skip                    ; if x != 42, jump past  
call    do_something             ; otherwise, execute the call  
.skip:
```

The `cmp` instruction performs a subtraction (`x - 42`) without storing the result, but updating the flags. If `x == 42`, the subtraction gives 0, so **ZF = 1**. The `jne` (*Jump if Not Equal*) instruction jumps if **ZF = 0** — so it does not jump, and the call to `do_something` is executed.

> 💡 **For RE**: you do not need to memorize the correspondence table between each jump instruction and the flags it tests. What matters is reading the `cmp` that precedes it and understanding the condition in natural language: `jne` = "jump if not equal", `jl` = "jump if less (signed)", `ja` = "jump if above (unsigned)". Section 3.4 lays out the full table.

### The `test` + conditional-jump pair

The other very common pattern is `test reg, reg`, which performs a bitwise AND without storing the result:

```c
// C code
if (ptr != NULL) {
    use(ptr);
}
```

```asm
; Assembly generated by GCC
test    rax, rax          ; bitwise AND of rax with itself  
jz      .skip             ; jump if rax == 0 (ZF = 1)  
mov     rdi, rax  
call    use  
.skip:
```

`test rax, rax` is GCC's standard idiom for checking whether a register is zero. The result of `rax AND rax` is `rax` itself — if that value is zero, ZF is set to 1. It is shorter and faster than a `cmp rax, 0`.

## Segment registers — a vestige to know about

You will occasionally come across references to segment registers (`cs`, `ds`, `ss`, `es`, `fs`, `gs`) in disassembly. In 64-bit mode, `cs`, `ds`, `ss`, and `es` no longer play a memory segmentation role — the memory model is flat.

However, **`fs` and `gs`** are still used:

- Under **Linux**, `fs` points to the Thread Local Storage (TLS). You will frequently see accesses like `mov rax, qword [fs:0x28]` — that is the read of the **stack canary** (an anti-buffer-overflow sentinel value), a pattern found in practically every binary compiled with `-fstack-protector`:

```asm
; Typical prologue with stack canary
mov     rax, qword [fs:0x28]     ; reads the canary from the TLS  
mov     qword [rbp-0x8], rax     ; places it on the stack  

; ... function body ...

; Epilogue — canary check
mov     rax, qword [rbp-0x8]     ; re-reads the canary from the stack  
xor     rax, qword [fs:0x28]     ; compares with the original value  
jne     .stack_chk_fail           ; if different → corruption detected  
```

This pattern is detailed in Chapter 19 (anti-reversing and protections), but it is useful to recognize it now so as not to mistake it for application logic.

## Floating-point and vector registers — quick overview

The x86-64 architecture also has registers dedicated to floating-point and vector (SIMD) operations. We do not cover them in detail here (section 3.9 is devoted to them), but it is useful to know they exist so you are not surprised when you come across them.

**SSE/AVX registers**: `xmm0`–`xmm15` (128 bits), extended into `ymm0`–`ymm15` (256 bits) with AVX. In the System V AMD64 convention, floating-point arguments (`float`, `double`) are passed via `xmm0`–`xmm7`, and the floating-point return value is in `xmm0`.

```c
// C code
double add_doubles(double a, double b) {
    return a + b;
}
```

```asm
; Assembly — a in xmm0, b in xmm1
addsd   xmm0, xmm1       ; double-precision addition  
ret                        ; result in xmm0  
```

**x87 FPU registers** (`st0`–`st7`): the old stack-based floating-point unit inherited from the 8087. You rarely encounter them in modern 64-bit GCC code (GCC prefers SSE), but they can appear with certain options or in 32-bit code.

## What to remember going forward

Five reflexes to develop when reading disassembled code:

1. **The size of the register name indicates the data size**: `rax` → 64-bit (pointer or long), `eax` → 32-bit (int), `ax` → 16-bit (short), `al` → 8-bit (char/byte).  
2. **`rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9`** are the six integer-argument registers — when you see them being written just before a `call`, those are the parameters.  
3. **`rax`** after a `call` contains the return value.  
4. **`rsp`** and **`rbp`** delimit the stack frame — accesses like `[rbp-X]` or `[rsp+X]` touch local variables.  
5. **`cmp`/`test`** modify the flags, and the **conditional jump** that immediately follows exploits them — that is the direct translation of your `if` / `while` / `for`.
