# Preface

---

## Tutorial objectives and target audience

This tutorial was born from a simple observation: resources on the reverse engineering of binaries compiled with the GNU toolchain are scarce, scattered, and often either too theoretical or too focused on a single tool without providing a big-picture view. The goal of this training is to fill that gap by offering a structured, progressive, and resolutely hands-on learning path.

By the end of this tutorial, you will be able to:

- **Understand** what a compiler like GCC or G++ actually produces, from C/C++ source code to the ELF binary loaded in memory.  
- **Inspect** an unknown binary methodically, combining command-line tools, advanced hex editors, and professional disassemblers.  
- **Statically analyze** disassembled and decompiled code, including optimized or stripped binaries.  
- **Dynamically analyze** a running program using debuggers, instrumentation, and fuzzing.  
- **Recognize** the patterns generated by GCC for C++ constructs (vtables, exceptions, STL, templates, smart pointers), as well as the effects of the different optimization levels.  
- **Bypass** common protections (stripping, packing, control-flow obfuscation, debugger detection).  
- **Apply** these skills to concrete cases: crackmes, network applications, encrypted binaries, custom file formats, and malicious samples in an isolated environment.  
- **Automate** your analyses thanks to Python scripting, tool APIs (Ghidra headless, r2pipe, GDB Python, Frida), and YARA rules.

### Who is this training for?

This training is designed for developers, computer science students, or security professionals who want to acquire or deepen reverse engineering skills on native Linux binaries. More specifically:

- **C/C++ developers** curious to understand what the compiler does with their code, or facing bugs that can only be explained at the assembly level.  
- **Computer security students** who want to go beyond theoretical courses and get their hands dirty with disassembly.  
- **CTF participants** (Capture The Flag) looking to structure their approach to "reversing" challenges.  
- **Junior SOC or malware analysts** wishing to acquire the basics of ELF binary analysis before specializing.  
- **Software engineers** working on interoperability, advanced debugging, or auditing third-party libraries without source code.

This training is **not** intended for complete programming beginners. It assumes you already know how to write and compile code, and that you are comfortable in a Linux terminal. The sections that follow detail the exact prerequisites.

## Recommended prerequisites

To get the most out of this training, a foundation of prior knowledge is necessary. You do not need to be an expert in each of these areas — an intermediate level is enough — but a complete absence in one of them is likely to make certain chapters difficult to follow.

### C programming (intermediate)

You should be comfortable with the following concepts:

- Basic types, pointers, and pointer arithmetic.  
- Structures (`struct`), unions, enumerations.  
- Dynamic allocation (`malloc`, `free`) and the concept of stack vs. heap.  
- C strings (arrays of `char` terminated by `\0`).  
- Compilation with `gcc`: common flags (`-o`, `-Wall`, `-g`, `-O2`).  
- Reading a simple `Makefile`.

### C++ programming (basics)

Several chapters (notably Part IV and the practical cases) focus on C++ binaries. A basic level is sufficient:

- Classes, inheritance, polymorphism (virtual methods).  
- Notions of templates and the STL (`vector`, `string`, `map`).  
- General understanding of C++ compilation with `g++`.

> 💡 If you do not know C++, Parts I through III and most of the C practical cases remain perfectly accessible. You can come back to the C++ chapters later.

### Linux command line (intermediate)

Reverse engineering under Linux relies heavily on command-line tools. You should know how to:

- Navigate a filesystem, manipulate files and permissions.  
- Use a terminal-based text editor (`vim`, `nano`, or equivalent).  
- Launch programs, redirect streams (`stdin`, `stdout`, `stderr`), use pipes.  
- Install packages with `apt` (or your distribution's package manager).  
- Work in a multi-terminal environment or with `tmux`/`screen`.

### Memory and architecture concepts

You do not need a full computer architecture course, but the following concepts should be familiar to you:

- The distinction between RAM and storage.  
- The notion of a memory address and hexadecimal representation.  
- The concepts of stack and heap at the operating system level.  
- The idea that a processor executes machine instructions sequentially.

> 💡 Chapter 3 (x86-64 assembly) covers these notions from the ground up in the context of RE. Even if your knowledge is rusty, you will be able to follow along provided you spend some time on it.

### What is **not** a prerequisite

- **x86-64 assembly**: Chapter 3 teaches it from scratch, with an RE focus.  
- **RE tools** (Ghidra, GDB, Frida, etc.): each tool is introduced progressively with installation and getting-started instructions.  
- **Malware analysis**: Part VI starts from zero, with the setup of a secure lab before any handling.  
- **Rust, Go, or C# languages**: the bonus parts (VII and VIII) are self-contained and require no prior experience with these languages.

## How to use this tutorial

This training is designed to work according to two reading modes, depending on your profile and objectives.

### Linear path (recommended for RE beginners)

If you are discovering reverse engineering or wish to consolidate your foundations, follow the parts in order, from Chapter 1 to Chapter 36. The progression is designed so that each chapter builds on the previous ones:

1. **Part I** lays the foundations: concepts, compilation chain, assembly, environment.  
2. **Part II** introduces static analysis with tools of increasing complexity.  
3. **Part III** adds dynamic analysis (debugging, instrumentation, fuzzing).  
4. **Part IV** deepens advanced techniques (optimizations, C++, symbolic execution, anti-reversing).  
5. **Part V** puts everything into practice on concrete applications.  
6. **Parts VI through IX** broaden toward malware analysis, modern languages, and automation.

Each chapter ends with a **checkpoint** — a mini-exercise that validates acquired knowledge before moving on. Do not skip the checkpoints: they are calibrated to quickly reveal points that need revisiting.

### On-demand access (for experienced profiles)

If you already have RE experience and are looking to deepen a specific topic, each chapter is written to be as self-contained as possible. A few examples of targeted paths:

- **"I know the basics but I struggle with optimized C++ binaries"** → Chapters 16 (compiler optimizations) and 17 (RE of C++ with GCC), then the practical case in Chapter 22.  
- **"I want to learn Ghidra"** → Chapter 8, then the practical cases in Chapters 21 through 25 which use it extensively.  
- **"I want to get into Linux malware analysis"** → Chapter 26 (secure lab), then Chapters 27 through 29.  
- **"I have to analyze Go/Rust binaries at work"** → Directly to Chapters 33 or 34, with occasional returns to Part I if necessary.  
- **"I want to automate my analyses"** → Chapter 35, drawing on the tool chapters (8, 9, 11, 13) for scripting APIs.

> 💡 Dependencies between chapters are flagged at the beginning of each file when they exist. If a chapter assumes mastery of a tool or concept seen earlier, an explicit cross-reference will indicate it.

### The training binaries

All binaries handled in this training are provided in the `binaries/` directory, along with their source code and a `Makefile`. You can recompile them at any time with `make all` from the repository root, or individually in each sub-directory. Each binary exists in several variants (optimization levels, with or without symbols) so that you can observe the impact of these parameters on the analysis.

No real malicious binary is distributed. The samples in Part VI are entirely designed by us, for educational purposes, and only work in the controlled environment described in Chapter 26.

## Typographic conventions

Throughout this training, the following conventions are used to make reading and information lookup easier.

### Text formatting

- `inline code` — names of commands, functions, files, registers, opcodes, compilation flags, and short code excerpts. Example: the command `readelf -S` displays the sections of an ELF binary.  
- **Bold** — terms defined for the first time, key concepts introduced in a section, and tool names at their first mention. Example: **Ghidra** is an open source RE framework developed by the NSA.  
- *Italics* — technical terms kept in their original form when there is no standard equivalent, or when highlighting them aids readability. Example: *name mangling* is the renaming mechanism applied to C++ symbols.

### Code blocks

Code blocks use syntax highlighting adapted to the language concerned:

```bash
readelf -h ./keygenme_O0
```

In Chapter 4 (work environment), commands are prefixed with `[vm]` (to be executed inside the VM) or `[host]` (to be executed on the host machine) to avoid any confusion:

```bash
[vm] sudo apt install -y gdb
[host] virsh snapshot-create-as RE-Lab tools-ready
```

When an assembly excerpt is presented, Intel syntax is used by default (unless otherwise stated), in line with the most common convention across RE tools:

```asm
mov    rdi, rax  
call   strcmp@plt  
test   eax, eax  
jnz    0x401234  
```

### Icons

The following callouts are used to draw attention to particular pieces of information:

> 💡 **Tip** — A practical piece of advice, a shortcut, or a technique that can save you time.

> 📖 **Reminder** — A callback to a concept seen in an earlier chapter, with a cross-reference to the relevant section.

> ⚠️ **Warning** — A common trap, a frequent mistake, or a point of vigilance not to be overlooked.

> 🔬 **Deep dive** — An advanced technical detail, optional on first reading, intended for readers who wish to go further.

> 🎯 **Checkpoint** — The end-of-chapter validation mini-exercise. Do it before moving on.

### Responsibility and ethics

This tutorial is published under the **MIT license** and has a strictly educational purpose. All techniques presented here must be used within a legal framework: authorized security auditing, vulnerability research in your own software, CTF competitions, interoperability, or learning on binaries you have compiled yourself.
